## 第一章 登入與帳號設定

### 1.1 首次登入

1. 開啟瀏覽器，前往 `http://<主機 IP>`（預設 `http://localhost`）
2. 輸入預設帳號：
   - **使用者名稱**：`admin`
   - **密碼**：`admin123`
3. 點擊「登入」

**首次登入強制設定**：系統會彈出設定視窗，必須填寫：
- 顯示名稱（Display Name）
- Email
- 新密碼（至少 6 個字元）
- 確認新密碼

填寫完畢後點「儲存並繼續」才能進入主介面。

### 1.2 Zoho SSO 登入

若管理員已設定 Zoho OIDC，登入頁下方會出現橘色「使用 Zoho 登入」按鈕。點擊後跳轉至 Zoho 授權頁，完成授權後自動回到主介面。初次 SSO 登入會自動建立帳號（`role_id=NULL`），需管理員後續分配專案與角色才能使用功能。

### 1.3 忘記密碼

1. 登入頁點「忘記密碼」
2. 輸入使用者名稱與 Email
3. 收到重設連結（需管理員已設定 SMTP）
4. 點連結進入重設頁面，輸入新密碼（至少 6 個字元）

### 1.4 個人設定

點右上角使用者名稱 → **個人設定**（或從頂部導覽列選「設定」）：

- **帳戶資訊**：修改顯示名稱、Email；上傳 / 移除大頭貼（最大 5MB，支援 PNG / JPEG / WebP / GIF）
- **修改密碼**：需輸入目前密碼
- **個人偏好**：
  - 預設執行環境（Docker / 本機 Agent）
  - 自動重整報告（5 秒輪詢開關）
  - 切換測試案例前是否確認放棄修改
  - 執行時啟用 Trace + 影片錄製

## 第三章 專案與測試樹

### 3.1 測試樹結構

AutoTest 採用五層樹狀結構管理測試案例：

```
Project（專案）
  └── Feature（功能模組）
        └── Page（頁面 / API 群組）
              └── Scenario（情境）
                    └── TestCase（測試案例，葉節點）
```

左側側邊欄顯示目前選定專案的樹狀結構。

### 3.2 建立新專案

1. 點左側側邊欄頂部的「**+ 新增專案**」按鈕
2. 輸入專案名稱
3. （可選）勾選「**複製自現有專案**」，從下拉選單選擇來源專案
   → 複製包含：完整樹狀結構、TestCase 內容、前置案例連結
4. 點「建立」

### 3.3 建立樹節點

在左側樹狀結構上**右鍵點擊**任一節點，彈出選單：

| 操作 | 說明 |
|---|---|
| **新增子節點** | 在此節點下新增 Feature / Page / Scenario / TestCase |
| **重新命名** | 修改節點名稱 |
| **刪除** | 刪除此節點（含所有子節點，DB CASCADE，不可恢復）|
| **移動** | 拖曳排序（Drag-and-drop）|

新建立的葉節點（TestCase）會自動建立 `entity_version` 快照，預設 `content_status = approved`。

### 3.4 切換專案

點左側側邊欄頂部的專案名稱下拉選單，選擇其他專案。

## 第四章 撰寫測試案例

### 4.1 測試案例編輯器

點擊樹狀結構中的 **TestCase** 節點，右側開啟編輯器，分為以下區塊：

#### 基本資訊
- 驗收標準（Acceptance Criteria）：Markdown 文字欄位
- 前置動作（Pre-Setup Actions）：執行前的準備工作描述

#### 前置案例（Setup Chain）
測試案例可串接其他案例作為前置條件：

1. 點「**+ 新增前置案例**」
2. 從下拉選單選擇前置案例
3. 設定 sort_order（執行順序）
4. 可選「失敗即中止」選項
5. 勾選/取消勾選可暫時停用

> 前置案例依 sort_order 順序執行，任一失敗（且標記「失敗即中止」）則後續全部跳過。

#### 步驟表格（BDD / KDT）

| 欄位 | 說明 |
|---|---|
| **No.** | 步驟編號（自動遞增）|
| **BDD 關鍵字** | Given / When / Then / And / But（下拉選單可選）|
| **說明** | 步驟的自然語言描述 |
| **動作 / Action** | Robot Framework keyword；下拉選單依當前案例平台動態顯示（WEB / API / APP / E2E / UI / DB），也可手動輸入自訂 keyword |
| **定位器 / Locator** | CSS selector、XPath 或元素描述 |
| **輸入值 / Input** | 填入的資料，支援 `${ENV_KEY}`、`{{= expr }}` |
| **斷言條件** | 比對方式（等於 / 包含 / 大於…）|
| **預期結果** | 斷言的預期值 |

**切換模式**：點步驟表格右上角的 `ALL / BDD / KDT` 按鈕切換顯示模式。

### 4.2 動態運算式（Mini DSL）

在「輸入值」或「預期結果」欄位中使用 `{{= expr }}` 語法：

```
{{= env.BASE_URL }}          → 取環境變數
{{= row.username }}          → 取 DDT 資料列的欄位
{{= uuid() }}                → 產生隨機 UUID
{{= now() }}                 → 目前時間戳記
{{= fakerEmail() }}          → 隨機 Email
{{= fakerName() }}           → 隨機姓名
{{= fakerPhone() }}          → 隨機電話
{{= 1 + 2 }}                 → 算術運算
{{= "prefix_" + row.id }}    → 字串拼接
```

### 4.3 Capture 步驟

Capture 步驟可從畫面元素或 API 回傳值取值，存入變數供後續步驟使用：

1. 在步驟表格新增一行，動作選「**Capture**」
2. 定位器填入要擷取的元素或 JSON 路徑
3. 輸入值欄填入變數名稱（例如 `user_token`）
4. 後續步驟即可用 `${user_token}` 引用

### 4.4 條件分支

支援 If / ElseIf / Else / EndIf 控制流程（由 Robot Framework 7 真實執行）：

```
If    條件：${status} == "200"
  When    驗證登入成功    …
ElseIf  條件：${status} == "401"
  When    驗證錯誤訊息    …
Else
  When    驗證未知狀態    …
EndIf
```

### 4.5 資料驅動測試（DDT）

**方法一：在 TestCase 內建資料表**
1. 點編輯器下方「**DDT 資料表**」區塊
2. 新增欄位（Column），再新增資料列
3. 步驟中以 `{{= row.欄位名稱 }}` 引用

**方法二：引用獨立測試資料集**
1. 先到頂部導覽「**測試資料集**」建立資料集（見 8.3 節之前的測試資料集說明）
2. 回 TestCase 編輯器，點「**套用資料集**」下拉選單選取

執行時系統會自動對每筆資料列各跑一次，報告中可見每列的獨立結果。

### 4.6 儲存與版本歷史

- 點「**儲存**」按鈕（或 `Ctrl+S`）儲存目前案例
- 每次儲存皆自動建立版本快照（`entity_versions` 表）
- 點編輯器右上角「**版本歷史**」可瀏覽所有快照，並可一鍵還原至任意版本
- 已 Approved 的案例被編輯儲存後，狀態自動退回 Pending，需重新送審

### 4.7 Markdown 匯入 / 匯出

- **匯出**：點「**匯出 .md**」，將目前 TestCase 的步驟下載為 Markdown 格式
- **匯入**：點「**匯入 .md**」，貼上或上傳 Markdown 文字，自動解析為步驟列表

### 4.8 步驟 CSV / Excel 匯入匯出

步驟表格上方提供 **匯出 / 匯入 / 下載範本** 三組按鈕，方便以試算表批次編輯步驟。

**匯出**：

- **CSV**：純文字格式，相容任何工具
- **Excel（.xlsx）**：除主工作表外另附 `選項清單` 工作表，且 **「BDD 關鍵字」與「動作（Action）」兩欄會自動套用下拉選單**（依當前案例平台動態給出 Action 選項，避免拼字錯誤造成匯入失敗）

**下載範本**：附 4 行範例步驟（Given / When / And / Then）與選項清單，可下載後填入再匯入。

**匯入**：

1. 點「**匯入**」按鈕，選擇 `.csv` / `.xlsx` / `.xls` 檔
2. 跳出預覽 modal 顯示偵測到的步驟數與前 5 行
3. 若目前案例已有步驟，可選擇「**取代全部**」或「**附加到後方**」
4. 確認後送入

**欄位對應**：`BDD 關鍵字` / `步驟描述` / `動作(Action)` / `定位器(Locator)` / `輸入值(Input)` / `比對條件` / `預期結果`，欄位標題必須與範本一致。

## 第五章 錄製測試操作

點頂部導覽「**錄製**」進入錄製工作台。三種模式共用一份 image（`autotest-recorder`），由後端依 `RECORDER_MODE` env 切換 entrypoint，首次建置完之後三種模式都能立刻使用。

### 5.1 Web 錄製（Playwright Docker + noVNC 模式）

適合：不想在本機安裝 Playwright，或需要遠端協作錄製。

**步驟：**

1. **輸入目標網址**（例如 `https://your-app.example.com/login`）
2. 點「**建立 / 重啟 Session**」
3. 首次使用需建置錄製用 Docker image（約 5–10 分鐘），建置進度顯示在畫面上
4. 建置完成後點「**啟動容器開始錄製**」
5. 畫面出現 noVNC 遠端瀏覽器畫面，在裡面操作你的應用程式
6. 操作完成後點「**停止錄製**」
7. 系統自動解析操作步驟，顯示 BDD 步驟預覽表格
8. 確認步驟無誤後，點「**建立為新案例**」（需先選好 Scenario 節點）或「**套用至現有案例**」

**自動建立模式**：勾選「自動建立測試案例」並選擇 Scenario，錄製停止後全自動建立。

### 5.2 API 錄製（mitmproxy / mitmweb 模式）

適合：錄製前後端 API 流量，自動產生 API 測試步驟。

1. 點「**API 錄製**」分頁
2. 點「**啟動 API 錄製容器**」（首次建置與 Web 共用 image）
3. 設定瀏覽器的 proxy 指向顯示的埠號
4. 在瀏覽器操作應用程式，所有 API 請求自動擷取
5. 點「**停止**」，系統將 HTTP 請求轉換為 BDD 請求 / 驗證步驟

### 5.3 App 錄製（Appium 腳本轉換）

1. 使用 Appium Inspector 對 App 進行錄製，匯出 Python 腳本
2. 點「**App 錄製**」分頁
3. 貼上 Appium Python 腳本
4. 點「**轉換**」，自動轉為 AppiumLibrary keyword 步驟

### 5.4 錄製歷史

畫面下方「**錄製記錄**」顯示所有過去的錄製 Session：
- 建立時間、Session ID、目標網址
- 可重新查看、下載 trace.zip，或刪除

## 第六章 執行測試

### 6.1 執行單一 TestCase

1. 在測試樹點選要執行的 TestCase
2. 點編輯器右上角「**執行**」按鈕（▶）
3. 確認執行環境（Docker / 本機 Agent）
4. 底部終端機面板即時顯示執行日誌

執行選項（可在個人設定或執行前調整）：
- **DDT 展開**：若有資料表，每筆資料各跑一次
- **啟用 Trace + 影片**：錄製 Playwright trace.zip 與 .webm 影片

### 6.2 執行環境說明

| 模式 | 說明 | 適用情境 |
|---|---|---|
| **Docker（預設）** | Celery worker 在後端動態 `docker run --rm autotest-robot-runner` 執行；結束即移除 | 正式 CI 環境、headless 執行 |
| **本機 Agent** | 下載 `local_agent.py`，在本機以有頭瀏覽器執行 | Debug、需要看到瀏覽器畫面 |

**本機 Agent 使用方式**：
```bash
# 下載 agent（從「本機 Agent」下載連結取得）
python local_agent.py --server http://localhost
```
Agent 啟動後持續等待任務，點「執行」後它會自動接收並執行。

### 6.3 即時日誌面板

底部終端機面板顯示：
- 綠色：步驟通過
- 紅色：步驟失敗
- 黃色：警告訊息
- 灰色：一般資訊

執行中可點「**停止**」中斷。

### 6.4 TestRun（測試回合批次執行）

將多個 TestCase 組合成一次批次執行：

1. 點頂部「**TestRun**」
2. 點「**+ 新增回合**」
3. 輸入回合名稱（例如「Sprint 23 Smoke Test」）
4. 從列表勾選要加入的 TestCase
5. 選擇執行環境
6. 點「**建立**」

建立後點回合右側「**執行**」按鈕，系統依序執行所有選定案例，統一顯示結果儀表板。

### 6.5 排程自動執行

1. 點頂部「**排程**」
2. 點「**+ 新增排程**」
3. 填寫：
   - 排程名稱
   - **目標節點**（多選，支援搜尋過濾）：可選單一 TestCase 或整個 Scenario / Feature
   - **重複類型**：
     - `ONCE`：指定時間執行一次
     - `DAILY`：每天指定時間
     - `WEEKLY`：指定星期幾（可多選）+ 時間
     - `MONTHLY`：每月指定日期 + 時間
   - 執行時間（datetime-local）
   - 執行環境
   - 啟用 / 停用開關
4. 點「**儲存**」

排程會在設定的時間由後端（每 30 秒掃描）自動觸發，無需手動介入。
點排程右側「**立即執行**」可不等排程時間直接觸發。

## 第七章 測試報告

### 7.1 報告列表

點頂部「**報告**」進入報告工作台：

- **統計卡片**：總計 / 通過 / 失敗 / 通過率 / 平均時長
- **狀態分佈圖** 與 **趨勢圖**
- **報告列表**：每次執行一筆記錄，顯示狀態、觸發類型（Manual / Schedule / CI）、執行模式、時間

點報告列表任一行 → 進入**詳細報告**。

### 7.2 詳細報告

- 報告標題、執行時間、觸發方式、總耗時
- **步驟列表**：每個步驟的結果（通過 / 失敗）、截圖縮圖
- **截圖查看**：點縮圖放大檢視（lightbox 模式，z-index 提到最高，不會被其他 modal 擋住）
- **影片回放**：點影片連結開啟 .webm 播放器
- **Trace Viewer**：點「查看 Trace」開啟 Playwright 內建 trace 播放器（需 HTTPS 或 localhost）
- **PDF 匯出**：點「匯出 PDF」下載報告
- **每個 failed step 旁邊有「⚠ 開缺陷」按鈕**：點下去自動帶入該 step 的截圖到 attachments、`error_message` 到 actual_result、testcase 到 linked_testcase、本報告到 linked_report，跳到「缺陷管理」detail drawer

### 7.3 刪除報告

在報告列表勾選多筆，點「**批次刪除**」，或點單筆右側刪除鈕。

## 第八章 缺陷管理 / 測試看版 / 測試時程

v1.1.9 新增的三個工作流模組，共同串接「testcase → todo → defect → schedule」的追蹤鏈。

### 8.1 缺陷管理（#defects）

點頂部「**缺陷管理**」進入缺陷工作台。nav 右側顯示「未指派 + 未關閉」缺陷數量的紅色 badge。

**Toolbar**：

```
[+ 新增缺陷]  [View: List ◉ | Kanban ○]  [狀態 ▼] [Severity ▼]
[Priority ▼] [Assignee ▼] [搜尋 code/title…]  [重新整理]
```

#### List 模式（預設）

table 顯示，欄位：缺陷編號（`DEF-XXXXX`）/ 標題 / 嚴重度 / 優先 / 狀態 / 指派 / 連結 testcase / 建立時間。

- 嚴重度配色：Critical=紅 / Major=橘 / Minor=黃 / Trivial=灰
- 狀態 7 色 pill：NEW=灰 / ASSIGNED=藍 / IN_PROGRESS=琥珀 / IN_REVIEW=紫 / REWORK_REQUIRED=玫紅 / VERIFIED=綠 / CLOSED=石灰
- 點 row 開 detail drawer
- 多選 + 「批次關閉」「批次指派」
- 分頁 25/50/100

#### Kanban 模式

7 個 column 對應 7-state status。
- 卡片可**拖拽換 column** → 後端發 `PATCH /api/defects/{id}` 帶 new status
- **非法 transition** 後端回 422，前端 toast 提示 + 卡片動畫退回原欄位
- 合法 transition 規則：
  ```
  NEW → ASSIGNED / IN_PROGRESS / CLOSED
  ASSIGNED → IN_PROGRESS / CLOSED
  IN_PROGRESS → IN_REVIEW / CLOSED
  IN_REVIEW → REWORK_REQUIRED / VERIFIED / CLOSED
  REWORK_REQUIRED → IN_PROGRESS / CLOSED
  VERIFIED → CLOSED
  CLOSED → ASSIGNED  (re-open)
  ```

#### 新增缺陷

點「**+ 新增缺陷**」，填寫：
- title（必填）/ description / steps_to_reproduce / expected_result / actual_result
- severity / priority / assignee 下拉
- linked_testcase（可搜尋下拉）/ linked_report
- 附件區：drag-drop / 多檔上傳（PNG / JPG，存 SeaweedFS `pic` bucket）

送出後跳 detail drawer。

#### Detail drawer

- 上方：`DEF-XXXXX` code + title + 狀態 pill + severity + priority
- 中間 tab：詳情 / 附件 / 歷史紀錄
  - **附件**：thumbnail grid，**點圖開 lightbox**（最高 z-index 不會被其他 modal 擋住，右上角叉叉關閉）
  - **歷史紀錄**：狀態轉移 + assignee changes + 附件變動，類似 reviews_history 模式

#### 從失敗報表一鍵建立

在 **報告詳情頁** 的 failed step 點「**⚠ 開缺陷**」按鈕：
- 後端讀取 ExecutionReport + 對應 step
- 自動塞入 step 的 screenshot URL 到 `attachments_json`
- step 的 `error_message` → `actual_result`
- testcase → `linked_testcase_id`
- report → `linked_report_id`
- 回傳 `DefectResponse`，前端跳到 detail drawer 確認後送出

#### 權限

- 看：project 成員
- 寫（新增 / 編輯）：`testcase.write`
- 刪除 / 強制關閉：`testcase.manage` 或 Admin

### 8.2 測試看版（#testkanban）

點頂部「**測試看版**」進入測試看版工作台。

**用途**：以「待辦」為單位追蹤每日進度，與測試案例 / 報告 / 缺陷雙向連結。

**5 個垂直泳道**（顏色由淺到深）：

| 欄位 | 顏色 | 說明 |
|---|---|---|
| 待辦 | slate | 尚未開始 |
| 進行中 | blue | 正在執行 |
| 待驗證 | violet | 完成等待驗證 |
| 已完成 | emerald | 已驗證完成 |
| 關閉 | stone | 不再追蹤 |

#### 新增待辦

點「**+ 新增待辦**」（深藍按鈕），填寫：
- 標題（必填）
- 描述
- 狀態（5 選 1）/ 優先級（P0–P3）/ 指派對象 / Sprint
- **開始時間 + 結束時間**（會同步顯示在「測試時程」Gantt 上）
- **連接項目**（可選，下拉選單）：可連到 testcase / report / defect / test_round / project
  - 選了類型後，ID 欄位變成下拉選單，自動列出該類型可選的項目

> v1.1.9 將「新增待辦」與「待辦詳細頁」的狀態下拉選單統一為相同 5 選項，且 modal 寬度統一為 880px。

#### 卡片操作

- **拖拽**：在泳道間拖動卡片，自動更新 status
- **點擊**：彈出待辦詳細視窗，可修改所有欄位 + 看連結項目
- **三點選單**：複製連結、刪除

#### 待辦 ↔ 連結項目

- 在 testcase 編輯器 / report 詳情 / defect drawer 都可看到「相關待辦」清單
- 點待辦的「跳轉」可直接前往連結的 testcase / report / defect

### 8.3 測試時程（#testschedule）

點頂部「**測試時程**」進入時程工作台。

**用途**：以 Gantt 風格規劃 Sprint / 階段 / 里程碑，並把當前 active 的所有待辦疊上去看排程。

**內容**：

1. **時程列**（`test_schedules` 表）：手動建立的 Sprint 區段（標題 / 開始 / 結束 / 顏色 / 階段標籤）
2. **待辦疊圖**：自動把所有有 `start_date` + `due_date` 的待辦疊到對應日期區間
3. **新增 / 編輯 / 刪除時程**：點時程列上的按鈕，與待辦一樣彈出 modal 編輯

> v1.1.9：新增待辦時的「開始 + 結束時間」會自動讓該待辦出現在測試時程 Gantt 上，不需要再到測試時程另外建立。

### 9.1 Mock 端點（#mock）

適合：後端 API 尚未完成，但需要前端 / 測試流程繼續推進。

點頂部「**Mock**」進入 Mock 工作台：

1. 點「**+ 新增 Mock**」
2. 填寫：
   - HTTP 方法（GET / POST / PUT / PATCH / DELETE…）
   - 路徑（例 `/api/users/:id`）
   - **Request**（描述 / 範例 headers / query / body）— v1.1.9 新增
   - Response：status code、headers、body（JSON / text）
   - 類型統一為 `Task`（v1.1.9 移除多型別下拉）
3. 儲存後，測試執行時呼叫到此 path 即回傳預設 response

> v1.1.9 修正：`POST /api/mock-endpoints` 422 容錯改善，headers 接受任意 dict / list 結構而不限定 `dict[str, Any]`；前端表單錯誤訊息也會詳細顯示哪一欄位問題。

### 9.2 檔案比對（#filecompare）

適合：跨環境匯出的 Excel / CSV 結果比對、回歸資料驗證。

點頂部「**檔案比對**」：

1. 上方拖入「左側檔案」與「右側檔案」（支援 `.xlsx` / `.xls` / `.csv`）
2. 系統自動：
   - 偵測 sheet（Excel 多 sheet 可選）
   - 比對欄位 header
   - row-by-row diff，顏色標示新增 / 移除 / 修改
3. 結果可下載為「差異報告」Excel

> 純客戶端執行（內嵌 SheetJS），**不上傳任何資料到伺服器**，敏感資料也可安心比對。

### 9.3 畫面比對（Picture Diff）

位於檔案比對頁下方，v1.1.9 新增。

1. 上傳兩張截圖（PNG / JPG）
2. 系統用 pixelmatch 演算法做 pixel-level diff
3. 三欄並排：原圖 A / 原圖 B / 差異圖（紅色標示）
4. 可調整：
   - **threshold**：差異容忍度
   - **alpha**：差異圖透明度
   - **anti-alias 容忍**：抗鋸齒誤判修正
5. 點「**下載差異圖**」存 PNG

> 100% 純客戶端跑，無任何資料上傳；底層演算法為內嵌的 pixelmatch port。

## 第十章 審核與通知

### 10.1 審核工作流

點頂部「**審核**」（顯示待審件數 badge）：

**分頁說明**：

| 分頁 | 顯示內容 |
|---|---|
| **待審核** | 已提交、等待審核的項目 |
| **已審核** | 審核通過的項目 |
| **已拒絕** | 被拒絕的項目 |

**類型篩選**：全部 / 測試案例 / 測試腳本 / 測試報告 / **缺陷** （v1.1.9 新增）

> 缺陷被建立 / 狀態變更時會自動建立審核項目，alembic migration 0031 已在 `reviewable_entity_type` enum 加入 `DEFECT`。

**審核流程**：

1. 提交者在 TestCase 編輯器點「**送審**」，進入 Pending 狀態
2. 審核者在「待審核」分頁看到新項目

4. 或點「**拒絕**」→ 填入拒絕原因 → 狀態變為 Rejected
5. 管理員可點「**回復**」將 Approved 退回 Pending 重新審核

審核項目若為**缺陷**類型，會在 detail 顯示「**前往缺陷**」按鈕，可直接跳到 `#defects` 對應的缺陷詳情。

**批次操作**：勾選多筆 → 「批次重新指派」指定審核者。

### 10.2 通知中心

點頂部導覽右側「**鈴鐺**」圖示：

- 顯示所有未讀通知（紅色數字 badge）
- 通知類型：審核事件（送審 / 核准 / 拒絕）、缺陷事件、系統事件
- 點單筆標記已讀，或點「**全部標記為已讀**」

### 10.3 環境變數

點頂部「**環境變數**」：

- **新增變數**：輸入 Key / Value，名稱只能包含字母、數字、底線，不能以數字開頭
- **敏感變數遮罩**：名稱含 TOKEN / SECRET / PASSWORD / KEY 的變數值自動遮罩顯示
- **Faker 生成**：點「🎲 Faker」按鈕，選擇類型（Name / Email / Phone / UUID / Address）隨機填入
- **批次匯入**：點「**匯入 .env**」，貼上 `.env` 格式文字（`KEY=VALUE`）批次建立

在步驟輸入值欄位使用 `${KEY}` 語法引用，執行時系統自動替換。

### 10.4 測試資料集

### 10.5 截圖基準比對（視覺回歸）

對有截圖的步驟設定「基準截圖」，用於視覺回歸測試：

1. 在報告的步驟截圖上，點「**設為基準**」

與第九章 9.3 的「畫面比對」不同：截圖基準是**自動**回歸（執行時觸發），畫面比對是**手動**比對任意兩張截圖。

## 第十一章 CI/CD 整合

AutoTest 提供完整 REST API（200+ 端點），可直接從 CI/CD pipeline 呼叫。完整 API 文件見 `http://localhost/api/docs`（Swagger UI）或 `http://localhost/api/redoc`。

### 11.1 取得 API Token

```bash
# 登入取得 access_token
TOKEN=$(curl -s -X POST http://localhost/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"your_password"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")
```

### 11.2 觸發執行

```bash
# NODE_ID：TestCase / Scenario / Feature 的節點 ID（從 UI URL 或 API 取得）
NODE_ID="abc123"

RESPONSE=$(curl -s -X POST http://localhost/api/executions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"node_id\": \"$NODE_ID\",
    \"trigger_type\": \"CI\",
    \"execution_mode\": \"docker\"
  }")

TASK_ID=$(echo $RESPONSE | python3 -c "import sys,json; print(json.load(sys.stdin)['task_id'])")
REPORT_ID=$(echo $RESPONSE | python3 -c "import sys,json; print(json.load(sys.stdin)['report_id'])")
echo "Task: $TASK_ID  |  Report: $REPORT_ID"
```

### 11.3 輪詢結果

```bash
while true; do
  STATUS=$(curl -s "http://localhost/api/executions/$TASK_ID/status" \
    -H "Authorization: Bearer $TOKEN" \
    | python3 -c "import sys,json; print(json.load(sys.stdin)['status'])")
  echo "Status: $STATUS"
  if [ "$STATUS" = "PASS" ] || [ "$STATUS" = "FAIL" ] || [ "$STATUS" = "ERROR" ]; then
    break
  fi
  sleep 10
done

[ "$STATUS" = "PASS" ] && echo "Tests PASSED" || { echo "Tests FAILED"; exit 1; }
```

### 11.4 自動建立缺陷（從 CI 失敗）

```bash
# 若報告失敗，從某 step 自動建立缺陷
curl -X POST http://localhost/api/defects/from-report \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"report_id\": \"$REPORT_ID\",
    \"step_id\": \"<failed_step_id>\",
    \"severity\": \"Major\",
    \"priority\": \"P1\",
    \"title_override\": \"[CI auto] 登入冒煙失敗\"
  }"
```

### 11.5 GitHub Actions 完整範例

```yaml
name: AutoTest CI

on: [push, pull_request]

jobs:
  run-tests:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger AutoTest
        id: trigger
        run: |
          TOKEN=$(curl -s -X POST ${{ secrets.AUTOTEST_URL }}/api/auth/login \
            -H "Content-Type: application/json" \
            -d '{"username":"${{ secrets.AUTOTEST_USER }}","password":"${{ secrets.AUTOTEST_PASS }}"}' \
            | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

          RESP=$(curl -s -X POST ${{ secrets.AUTOTEST_URL }}/api/executions \
            -H "Authorization: Bearer $TOKEN" \
            -H "Content-Type: application/json" \
            -d '{"node_id":"${{ vars.TEST_NODE_ID }}","trigger_type":"CI","execution_mode":"docker"}')

          TASK_ID=$(echo $RESP | python3 -c "import sys,json; print(json.load(sys.stdin)['task_id'])")
          echo "task_id=$TASK_ID" >> $GITHUB_OUTPUT
          echo "token=$TOKEN" >> $GITHUB_OUTPUT

      - name: Wait for Result
        run: |
          TASK_ID=${{ steps.trigger.outputs.task_id }}
          TOKEN=${{ steps.trigger.outputs.token }}
          for i in $(seq 1 60); do
            STATUS=$(curl -s "${{ secrets.AUTOTEST_URL }}/api/executions/$TASK_ID/status" \
              -H "Authorization: Bearer $TOKEN" \
              | python3 -c "import sys,json; print(json.load(sys.stdin)['status'])")
            echo "[$i/60] Status: $STATUS"
            if [ "$STATUS" = "PASS" ]; then exit 0; fi
            if [ "$STATUS" = "FAIL" ] || [ "$STATUS" = "ERROR" ]; then exit 1; fi
            sleep 10
          done
          echo "Timeout waiting for test result"; exit 1
```

### 12.3 角色與權限（RBAC）

AutoTest 採三層 RBAC：

| 層級 | 說明 |
|---|---|
| **Global** | Superuser，跨 Org 全局管理 |
| **Org Level** | Admin / QA / Viewer，對整個組織生效 |
| **Project Level** | Project-Admin / Project-Tester / Project-Reviewer / Project-Viewer，可覆蓋 Org 角色 |

**內建角色（Org）**：
- `Admin`：全部權限
- `QA`：讀寫測試案例 + 執行 + 查看報告 + 建缺陷 / 待辦
- `Viewer`：只讀

**內建角色（Project）**：
- `Project-Admin`：專案內全部權限（含成員管理）
- `Project-Tester`：寫案例 + 執行 + 查看報告 + 缺陷管理
- `Project-Reviewer`：讀全部 + 審核
- `Project-Viewer`：只讀

**權限 key（節錄）**：`testcase.read / write / manage`、`defect.read / write / manage`、`review.approve`、`schedule.write`、`testkanban.read / write`、`mock.write` 等，完整清單見 `backend/app/auth/permissions_catalog.py`。

### 12.5 備份與還原

```bash
docker exec autotest-postgres pg_dump -U admin autotest_db > backup_$(date +%Y%m%d).sql
```

```bash
# 手動觸發

# 查看備份清單
docker exec autotest-backup-cron ls -lh /backups/
```

### 12.7 釋放 VM 磁碟空間

當 VM 磁碟用量上升，安全清理方式：

```bash
# ✅ 安全清理（不會刪 spawn image）
docker container prune -f          # 停止的容器
docker volume prune -f             # 沒掛載的 volume（不會碰 named volume）
docker image prune -f              # dangling image（只刪沒 tag 的）
docker builder prune -af           # build cache

# ❌ 絕對不要跑這個
docker image prune -a              # 會把 autotest-robot-runner / autotest-recorder 砍掉！
```

備份保留天數可在 `.env` 中設定：

## 附錄

### 常見問題

**Q：執行後報告顯示「FAIL」但日誌看不出原因？**
點報告的「查看 Trace」按鈕，在 Playwright Trace Viewer 中逐步回放每個操作，可看到每步的截圖與 console 輸出。

**Q：Docker 模式執行時出現「image not found」？**
先執行 `docker compose --profile spawnable build` 建置 `autotest-robot-runner` image。**請勿執行 `docker image prune -a`**，會把它砍掉。

**Q：錄製器顯示「noVNC 無法連接」？**
`autotest-recorder` image 還在建置中（首次需 5–10 分鐘），等建置進度條完成後重試。

**Q：本機 Agent 無法接收任務？**
確認 `python local_agent.py --server http://<主機 IP>` 的主機 IP 可從本機訪問，且 `80` port 未被防火牆擋住。

**Q：測試資料集的 `{{= row.col }}` 沒有被替換？**
確認執行選項有勾選「DDT 展開」，且欄位名稱大小寫與資料集定義完全一致。

**Q：排程設定後沒有自動執行？**
確認排程的「啟用」開關已打開，且 `next_run_at` 時間正確（系統每 30 秒掃描一次）。

**Q：審核通過的 TestCase 能修改嗎？**
Approved 狀態的案例可被編輯，但儲存後狀態自動退回 Pending，需重新送審。管理員可用「回復」按鈕手動退回 Pending。

**Q：缺陷的 detail 點圖片後被其他 modal 擋住？**
v1.1.9 已修正：lightbox 用 `z-index: 2147483647`（最大整數）+ open 時把 element 移到 `document.body` 最後位置，確保永遠在最上層；右上角有叉叉按鈕可關閉。

**Q：新增 Mock 一直 422 怎麼辦？**
確認 headers 欄位的 JSON 格式正確（v1.1.9 後容錯放寬，但仍需是合法 JSON）。前端表單會詳細顯示哪一欄位失敗。

**Q：測試看版的卡片顏色和其他頁面差異很大？**
v1.1.9 已統一視覺風格：主題色從 sky → blue（Tailwind 預編譯 CSS 內 sky 系列未含 500/600），所有按鈕都有實際底色。

**Q：如何匯入大量 TestCase？**
使用 Markdown 匯入功能：準備符合格式的 `.md` 檔，在樹節點右鍵選「匯入 Markdown」。
