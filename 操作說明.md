## 1. 快速上手（5 分鐘）

### 1-1. 啟動平台

需要 Docker 24+ 與 Docker Compose v2.23+（Linux / macOS / Windows Docker Desktop）。

```bash
git clone https://github.com/ryanlin147188-commits/RL-for-Kapito.git
cd RL-for-Kapito

# 自動產生 .env（含隨機 secret；已存在則跳過）
docker compose --profile init run --rm bootstrap

# 預先建置 spawn-time image（首次約 5–10 分鐘）
docker compose --profile spawnable build

# 啟動主服務
docker compose up -d --build
```

完成後，開啟瀏覽器前往 [http://localhost](http://localhost)。

### 1-2. 登入

| 登入方式 | 帳號 | 密碼 |
|---|---|---|
| 帳密登入 | `admin` | `admin123` |
| Zoho SSO | 你的 Zoho 帳號 | — |

> **首次登入**：系統強制要求填寫顯示名稱（Display Name）、Email，並修改密碼後才能進入主介面。

### 1-3. 認識介面配置

```
┌─ 頂部導覽列 ────────────────────────────────────────────────┐
│ Logo | 測試案例 測試資料 環境變數 TestRun 測試排程 ... | 帳號 │
└──────────────────────────────────────────────────────────┘
┌─ 左側面板 ──────┐  ┌─ 主內容區 ────────────────────────────┐
│  專案下拉選單   │  │  依所在頁面顯示對應內容                  │
│  ─────────────  │  │  （首頁 / 測試案例 / 報告 / 審核 ...）  │
│  專案樹狀目錄   │  │                                        │
│  （六層結構）   │  │  ← 返回  麵包屑導覽                    │
└─────────────────┘  └────────────────────────────────────────┘
```

- **頂部功能列**：直接切換各功能頁面
- **左側樹**：顯示當前專案的六層測試目錄
- **← 返回按鈕**：頁面切換時出現，可回到上一個頁面

## 2. 核心概念

### 2-1. 六層測試樹

AutoTest 用六層結構組織所有測試，對應業務邏輯：

```
Project（專案）               ← 一個產品或系統
└── Feature（功能）            ← 業務功能，例：使用者認證
    └── Platform（平台）       ← WEB / API / E2E
        └── Page（頁面/端點）  ← 登入頁、/api/auth/login
            └── Scenario（情境）← 正向 / 反向 / 邊界
                └── TestCase  ← 真正的測試步驟在這層
```

每一層都有對應的圖示與顏色，讓你在複雜的測試庫中快速定位。

### 2-2. TestCase 的四個區塊

| 區塊 | 說明 |
|---|---|
| **驗收準則（AC）** | 用文字描述預期行為，供審核者閱讀；不直接執行 |
| **前置動作（Setup）** | 此案例執行前先跑的其他 TestCase 鏈（login setup、資料建立等）|
| **測試步驟（Steps）** | BDD 格式的實際步驟，每行對應一個 Robot Framework 關鍵字呼叫 |
| **資料驅動（DDT）** | 多組測試資料，讓同一步驟對 N 筆資料各跑一次 |

### 2-3. 步驟欄位說明

| 欄位 | 說明 | 範例 |
|---|---|---|
| BDD | Given / When / Then / And | `When` |
| 步驟說明 | 自然語言描述（給人讀的）| `輸入帳號` |
| 動作 | Robot Framework 關鍵字 | `Input Text` |
| 測試目標 | 元件定位符（Locator）| `id=username` |
| 輸入 | 輸入值或參數 | `admin` |
| 比較條件 | 斷言方式 | `==` |
| 預期值 | 斷言的預期結果 | `登入成功` |

## 3. 專案與目錄管理

### 3-1. 建立新專案

1. 點擊左側面板頂部的專案下拉選單旁「**+**」按鈕（或樹狀頂部的 🗁 圖示）
2. 在「新增專案」Modal 中：
   - 輸入**專案名稱**
   - 若要複製既有專案的完整結構：勾選「**複製自現有專案**」，再從下拉選單選擇來源
3. 按「確認儲存」

> **複製專案**會複製：完整的六層樹狀結構、每個 TestCase 的步驟（steps_json）、驗收準則（ac_text）、前置動作（setup_text）、DDT 資料（ddt_json），以及所有**前置案例連結**（TestcasePreconditionLink）。**不**複製：執行報告、錄製腳本、排程設定、成員。

### 3-2. 建立目錄結構

在左側樹上「**右鍵**」任一節點可以：

| 操作 | 說明 |
|---|---|
| 新增子節點 | 在此節點下新增下一層 |
| 修改名稱 | 重命名（Platform / Scenario 層為系統列舉值，不可改）|
| 刪除 | 刪除此節點及所有子孫（含 TestcaseContent，不可復原）|

每一層新增方式：
- **Feature**：點樹狀頂部的「新增功能」按鈕
- **Platform**：右鍵 Feature → 新增子節點，選 WEB / API / E2E
- **Page**：右鍵 Platform → 新增子節點，輸入頁面名稱
- **Scenario**：右鍵 Page → 新增子節點，選 正向 / 反向 / 邊界
- **TestCase**：右鍵 Scenario → 新增子節點，輸入檔案名稱（.md 結尾）

### 3-3. 拖曳排序

同一層的節點可以**拖曳**調整顯示順序，支援跨父節點（同深度內）移動。

## 4. 撰寫測試案例

### 4-1. 開啟編輯器

點擊左側樹中任一 **TestCase 節點**，主內容區即開啟案例編輯器。

### 4-2. 填寫驗收準則

在「驗收準則」區塊用自然語言描述測試目的，例如：

```
GIVEN  使用者在登入頁
WHEN   輸入正確的帳號 admin 與密碼，按下登入
THEN   系統跳轉首頁，Header 顯示使用者名稱
```

### 4-3. 新增步驟

點擊步驟表格末尾的「**+ 新增步驟**」，或按最後一行的 Enter。填入各欄：

**範例：登入測試**

| BDD | 步驟說明 | 動作 | 測試目標 | 輸入 | 比較條件 | 預期值 |
|---|---|---|---|---|---|---|
| Given | 開啟登入頁 | Navigate To | — | http://localhost | — | — |
| When | 輸入帳號 | Input Text | `id=username` | admin | — | — |
| When | 輸入密碼 | Input Text | `id=password` | admin123 | — | — |
| When | 點擊登入 | Click | `text=登入` | — | — | — |
| Then | 確認跳轉 | Wait Until Element | `id=mainHeader` | — | Contains | AutoTest |

### 4-4. 儲存案例

按右上角「**儲存**」（或 Ctrl/Cmd + S）。每次儲存皆建立一筆 entity_version 快照，可從「版本歷史」回溯或比對差異。

## 5. 執行測試與查看報告

### 5-1. 執行測試

**方式一：在案例編輯器**
- 右上角「**執行**」按鈕

**方式二：在左側樹**
- 右鍵任一節點 → 「執行」（自動展開所有 TestCase 子孫批次執行）

選擇執行模式：
- **Docker 模式**（預設）：後台 spawn 一個 `autotest-robot-runner` 容器執行
- **本機模式**：需先啟動本機 Agent（見 [§7-13](#7-13-本機-agent有頭模式)），測試在你的桌面瀏覽器上可見

執行中可即時看 **WebSocket 日誌**（每一步的 keyword 呼叫與結果）。

### 5-2. 查看執行報告

前往頂部「**測試報告**」頁面，點擊任一執行記錄進入詳情：

| 區塊 | 說明 |
|---|---|
| **KPI 卡片** | 總步驟數、通過數、失敗數、執行時間 |
| **步驟時間軸** | 展開每個步驟，查看截圖、Robot log、錯誤訊息 |
| **MP4 影片** | 完整錄影（點擊「播放」或「下載」）|
| **Playwright Trace** | 下載 `.zip`，可拖入 [trace.playwright.dev](https://trace.playwright.dev) 做互動式分析 |
| **PDF 匯出** | 右上角「匯出 PDF」，以 html2pdf 將報告渲染成 PDF |

### 5-3. 報告篩選

報告列表支援以下篩選：

- 狀態（Running / Passed / Failed / Cancelled）
- 觸發方式（手動 / 排程 / CI）
- 時間區間

## 6. 錄製器

### 6-1. Web 錄製（本機模式）

適合已安裝 Node.js / Playwright 的開發機。

1. 頂部「**測試錄製**」→ 選「Web」→「本機」
2. 點「建立 Session」，系統回傳啟動指令（`npx playwright codegen ...`）
3. 在本機終端機執行該指令，Playwright Inspector 視窗開啟
4. 在瀏覽器上操作完畢後關閉 Inspector
5. 回到 SPA，將 Inspector 產生的腳本貼回
6. 系統自動解析為 BDD 步驟陣列，供你套用到 TestCase

### 6-2. Web 錄製（Docker 模式）

無需在本機安裝任何依賴。

1. 選「Web」→「Docker」
2. 系統起一個含 noVNC 的 `autotest-recorder` 容器（首次約 20–30 秒）
3. SPA 內嵌 iframe 顯示遠端瀏覽器，使用工具列控制（點擊 / 輸入 / 截圖）
4. 結束後系統自動解析步驟

### 6-3. API 錄製（貼 cURL）

1. 選「API」→「貼 cURL」
2. 貼上 cURL 指令，例如：
   ```bash
   curl -X POST https://api.example.com/auth/login \
     -H 'Content-Type: application/json' \
     -d '{"username":"admin","password":"admin123"}'
   ```
3. 系統解析 method、URL、headers、body → 轉成 `HTTP Request` 步驟

支援多種 shell 語法、`-H`、`--data`、`--cookie`、bearer token 等常見參數。

### 6-4. API 錄製（mitmproxy Docker 模式）

適合需要完整擷取 SPA API 對話的情境。

1. 選「API」→「Docker」
2. 系統起一個含 mitmweb 的 `autotest-recorder-api` 容器
3. 將瀏覽器或 app 流量導向此 proxy（SPA 提供設定指引）
4. 操作完畢後系統解析所有 HTTP 流量為步驟陣列

### 6-5. App 錄製（Appium）

1. 選「App」
2. 使用 Appium Inspector 錄製 mobile app 操作，匯出 Python 腳本
3. 貼入 SPA，系統轉換為 `AppiumLibrary` keyword 的步驟陣列

### 6-6. 套用錄製結果至 TestCase

錄製解析完畢後，選擇套用方式：

- **建立新案例**：跳出 Scenario 選擇 dialog，一鍵建立新 TestCase 並套入步驟
- **套用至現有案例**：選擇一個既有的 TestCase，以錄製步驟覆蓋
- **手動微調後套用**：在預覽區逐行編輯，再套入

## 7. 進階功能

### 7-1. Capture — 從畫面或 API 擷取變數

把執行過程中取得的值儲存為變數，供後續步驟引用。

**從畫面元素擷取文字：**

| 動作 | 測試目標 | 輸入（變數名稱）|
|---|---|---|
| Get Text | `.user-greeting` | welcome_msg |

**從 API 回應的 JSON 擷取（JSONPath）：**

| 動作 | 測試目標（JSONPath）| 輸入（變數名稱）|
|---|---|---|
| Get Json Value | `$.data.user.id` | user_id |

**後續步驟引用：**

```
{{= welcome_msg }}     ← 文字變數
{{= user_id }}         ← 從 API 取得的 ID
```

### 7-2. 條件分支（If / ElseIf / Else / EndIf）

根據 Capture 結果或環境變數，讓測試流程分支。

| 動作 | 比較條件 | 預期值 |
|---|---|---|
| If | Captured `user_role` == | admin |
| （管理員專屬步驟群組）| | |
| ElseIf | Captured `user_role` == | viewer |
| （一般使用者步驟群組）| | |
| Else | | |
| （其他情況步驟群組）| | |
| EndIf | | |

平台將這些轉換成 Robot Framework 7 的 `IF / ELSE / END` 區塊，真正在 runner 內執行條件分支，不是預先計算再跑全部。

### 7-3. 動態運算式（Mini DSL）

在任何輸入欄位使用 `{{= expression }}` 語法，執行時即時計算。

| 類型 | 語法 | 說明 |
|---|---|---|
| DDT 欄位 | `{{= row.username }}` | 引用資料驅動欄位 |
| 環境變數 | `{{= env.STAGING_URL }}` | 引用環境變數表 |
| 字串拼接 | `{{= env.HOST + "/api/v1" }}` | 組合 URL |
| 唯一識別碼 | `{{= uuid() }}` | UUID v4 |
| 當前時間 | `{{= now() }}` | ISO 8601 時間戳記 |
| 隨機姓名 | `{{= fakerName() }}` | Faker 隨機假資料 |
| 隨機 Email | `{{= fakerEmail() }}` | — |
| JSON 路徑 | `{{= json(captured.resp, "$.id") }}` | 從 JSON 字串取值 |
| MD5 雜湊 | `{{= md5(captured.token) }}` | — |
| Base64 編碼 | `{{= b64encode(payload) }}` | — |

### 7-4. 資料驅動測試（DDT）

讓同一個測試案例對多組資料各跑一次，適合登入驗證、表單邊界、API 參數測試。

**步驟一：在 DDT 分頁建立資料表**

| username | password | expected_msg |
|---|---|---|
| admin | admin123 | 登入成功 |
| admin | wrongpw | 密碼錯誤 |
| blocked | any123 | 帳號已停用 |

**步驟二：在步驟中引用欄位**

```
Input Text    id=username    {{= row.username }}
Input Text    id=password    {{= row.password }}
Wait Toast    .toast-body    Contains: {{= row.expected_msg }}
```

**執行結果**：報告中該案例顯示 3 次獨立執行，各自保存截圖、日誌，pass / fail 個別計算。

> 也可以引用「**測試資料集**」（獨立建立的共用資料集），讓多個 TestCase 共用同一份資料。

### 7-5. 前置案例（Setup Chain）

在執行主要 TestCase 之前，先跑其他 TestCase 做環境準備（如：建立測試帳號、登入、建立測試資料）。

**設定方式：**

1. 在 TestCase 編輯器選「**前置動作**」分頁
2. 點「+ 新增前置案例」，搜尋並選擇目標 TestCase
3. 可設定多個前置，並透過排序調整執行順序
4. 勾選「啟用/停用」可暫時跳過特定前置而不刪除

**執行邏輯：**

- 前置案例依 `sort_order` 順序執行，任一前置失敗則立即中斷主案例（`on_failure=stop`）
- 同一前置若被多個 TestCase 引用，不重複執行（DFS 去重）
- 循環依賴（A → B → A）會被偵測並在執行前拒絕

**複製專案時**：前置案例連結會一起複製，新專案內兩端的 node_id 均自動對應到新節點。

### 7-6. 測試排程

自動在指定時間執行測試，無需人工觸發。

**建立排程：**

1. 頂部「**測試排程**」→ 右上角「+ 新增排程」
2. 填寫設定：

| 欄位 | 說明 |
|---|---|
| **排程名稱** | 識別用名稱（必填）|
| **目標節點** | 選擇要執行的 TestCase（可多選，僅顯示 `.md` 測試案例節點）|
| **重複類型** | ONCE / DAILY / WEEKLY / MONTHLY |
| **WEEKLY 設定** | 勾選星期幾（週日=0，週六=6，可多選）|
| **MONTHLY 設定** | 填入日期（1–31；超出該月天數自動補至月底，例如 31 → 2 月 28 日）|
| **執行時間** | 日期與時間（以伺服器本地時間為準）|
| **執行環境** | Docker（預設）或本機 Agent |

3. 按「確認儲存」

**排程清單操作：**

| 操作 | 說明 |
|---|---|
| **啟用/停用** | 切換排程的 active 狀態，停用後不會觸發 |
| **立即執行** | 不等下次觸發時間，馬上執行一次 |
| **查看上次報告** | 點擊「查看報告」連結，跳至最近一次的執行報告 |
| **刪除** | 移除排程（需確認）|

> 排程服務每 **30 秒**掃描一次，觸發誤差在 30 秒以內。ONCE 類型執行完畢後自動停用。

### 7-7. 測試回合（Test Round）

將多次執行群組成一個測試週期，例如「Sprint 23 Smoke Round」。

1. 頂部「**TestRun**」→「+ 新增 Round」，填入回合名稱
2. 在測試案例樹上右鍵節點 → 「加入 Round」，選擇目標 Round
3. 在 TestRun 頁面選擇 Round → 點「執行此 Round」，系統批次執行所有案例
4. Round dashboard 顯示：執行次數、整體通過率、各案例狀態、時間走勢

### 7-8. 審核中心

對 TestCase、錄製腳本（Script）、執行報告（Report）進行審核。

**審核狀態流程：**

```
（新建 / 修改）
    ↓ 自動進入 PENDING
PENDING（待審核）
    ↓ 通過審核           ↓ 退回
APPROVED（已通過）    REJECTED（已拒絕）
    ↓ 撤銷審核               ↓ 重新送審
  PENDING                PENDING
```

> APPROVED 的 TestCase 會被**鎖定**，無法編輯，必須先「撤銷審核」才能修改。

**操作說明：**

1. 頂部「**審核**」→ 切換分頁：
   - **待審核**：等待你審核的項目
   - **已審核**：你已通過的項目（顯示審核者與審核時間）
   - **已拒絕**：已退回的項目（顯示審核者、審核時間與退回原因）

2. 點擊任一列開啟**詳情 Modal**，可查看：
   - **REPORT**：執行狀態、通過/失敗案例數、總執行時間、觸發方式
   - **TESTCASE**：節點層級路徑、步驟總數
   - **SCRIPT**：錄製目標網址、主機名稱、指令數、腳本摘要
   - 完整審核歷程（誰、何時、做了什麼決定、填了什麼原因）

3. 在詳情 Modal 底部執行：
   - **通過**：僅限 PENDING 狀態
   - **退回**：需填寫退回原因（不能空白）
   - **查看報告 / 前往案例 / 前往腳本**：跳轉到對應實體頁面
   - **撤銷審核**：APPROVED → PENDING（需填理由）

### 7-9. 待辦清單（Backlog）

管理測試相關的任務、缺陷、探索項目。

**進入方式：** 頂部選單（若有），或直接輸入 `#backlog` hash 路由。

**階層結構：** Feature（上層容器）→ Task / Bug / Spike（項目）

**欄位說明：**

| 欄位 | 說明 |
|---|---|
| **類型** | Task（一般任務）/ Bug（缺陷）/ Spike（探索研究）|
| **Sprint 標籤** | 可選填 Sprint 編號或名稱 |
| **截止日期** | 超過截止日自動顯示過期 badge |
| **指派對象** | 可指定使用者（user）或群組（group）|
| **關聯連結** | 可與 TestCase / Report 建立連結 |

### 7-10. 環境變數

每個專案有獨立的環境變數表，在執行時以 Robot Framework 變數形式注入。

**設定方式：**

1. 頂部「**環境變數**」頁面
2. 選擇專案
3. 新增 Key = Value 組合，例如：

```
STAGING_URL = https://staging.example.com
TEST_USER   = qa-bot@company.com
API_TOKEN   = xxxxxxxxxxxxxxxxxxxx
```

**在步驟中引用：**

```
Navigate To    {{= env.STAGING_URL }}/login
Input Text     id=email    {{= env.TEST_USER }}
```

### 7-11. 測試資料集

獨立於 TestCase 之外的共用 DDT 資料表，可被多個 TestCase 引用。

1. 頂部「**測試資料**」頁面 → 「+ 新增資料集」
2. 定義欄位（Columns）與資料列（Rows）
3. 在 TestCase 的 DDT 分頁選擇「引用資料集」，選定後即可用 `{{= row.欄位名 }}` 引用

> 適合登入帳密、測試商品列表、國際化語系等需要多個 TestCase 共用的資料。

### 7-12. Mock 端點

為尚未完成的 API 後端提供假回應，讓前端測試可以先跑。

1. 左側（若有 Mock 入口）或直接從後端 API 管理
2. 新增一筆 mock：
   - **HTTP 方法**：GET / POST / PUT / DELETE
   - **路徑**：例如 `/api/users/me`
   - **回應 Body**：JSON 或純文字
   - **狀態碼**：200 / 404 / 500 等

3. 測試步驟中將請求指向 mock URL（`http://localhost/api/...`）

每筆 mock 屬於特定組織，不同組織之間完全隔離。

### 7-13. 本機 Agent（有頭模式）

讓測試在你的實體桌面瀏覽器上執行，而非 headless Docker，便於視覺化 debug。

**設定步驟：**

1. 「設定 → 使用者設定 → 本機 Agent」下載 agent 安裝包
2. 安裝後執行 agent，填入：
   - **Backend URL**：`http://<server-ip>`
   - **Agent Token**：從 SPA 使用者設定頁取得

3. Agent 顯示「Connected」後，執行測試時切換「**本機模式**」

測試過程中你可以親眼看到瀏覽器操作，方便觀察步驟失敗的原因。

### 7-14. Markdown 匯入 / 匯出

TestCase 的步驟以 7 欄 Markdown 表格格式儲存，可雙向轉換。

**匯出：**
1. 在樹狀節點上右鍵 → 「匯出為 Markdown」
2. 下載包含所有子節點案例的 `.zip` 壓縮檔

**匯入：**
1. 右鍵樹狀節點 → 「匯入 Markdown」
2. 上傳 `.md` 或 `.zip`，系統自動建立節點並填入步驟

**Markdown 格式：**

```markdown
Test Case: 登入成功
Documentation: 使用正確帳密登入應成功跳轉首頁

## Steps
| BDD | 步驟說明 | 動作 (官方指令) | 測試目標 | 輸入 | 比較條件 | 預期值 |
| --- | --- | --- | --- | --- | --- | --- |
| Given | 開啟登入頁 | Navigate To | | http://localhost | | |
| When | 輸入帳號 | Input Text | id=username | admin | | |
```

### 7-15. REST API 與 CI/CD 整合

**取得 JWT Token：**

```bash
TOKEN=$(curl -s -X POST http://<host>/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"ci-bot","password":"<password>"}' \
  | jq -r .access_token)
```

**觸發測試執行：**

```bash
# 觸發單一 TestCase 或整個子樹
curl -s -X POST http://<host>/api/executions \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{
    "node_id": "<testcase-uuid>",
    "trigger_type": "CI",
    "execution_mode": "docker"
  }'
# 回傳：{ "task_id": "...", "report_id": "..." }
```

**輪詢執行狀態：**

```bash
curl -s http://<host>/api/executions/<task_id>/status \
  -H "Authorization: Bearer $TOKEN"
# 回傳：{ "status": "running" | "passed" | "failed" | "cancelled" }
```

**GitHub Actions 範例：**

```yaml
- name: Run AutoTest Smoke
  run: |
    TOKEN=$(curl -s -X POST ${{ secrets.AUTOTEST_URL }}/api/auth/login \
      -H 'Content-Type: application/json' \
      -d "{\"username\":\"${{ secrets.AUTOTEST_USER }}\",\"password\":\"${{ secrets.AUTOTEST_PASS }}\"}" \
      | jq -r .access_token)

    RESULT=$(curl -s -X POST ${{ secrets.AUTOTEST_URL }}/api/executions \
      -H "Authorization: Bearer $TOKEN" \
      -H 'Content-Type: application/json' \
      -d "{\"node_id\":\"${{ vars.SMOKE_NODE_ID }}\",\"trigger_type\":\"GitHub Actions\",\"execution_mode\":\"docker\"}")

    TASK_ID=$(echo "$RESULT" | jq -r .task_id)

    # 輪詢，最多等 10 分鐘
    for i in $(seq 1 60); do
      STATUS=$(curl -s "${{ secrets.AUTOTEST_URL }}/api/executions/$TASK_ID/status" \
        -H "Authorization: Bearer $TOKEN" | jq -r .status)
      [ "$STATUS" = "passed" ] && exit 0
      [ "$STATUS" = "failed" ] && exit 1
      [ "$STATUS" = "cancelled" ] && exit 1
      sleep 10
    done
    exit 1  # 超時
```

**查看完整 API 文件：**

```bash
# 從 backend 容器內取得 OpenAPI JSON
docker compose exec backend curl -s localhost:8000/openapi.json | jq .
```

## 8. 帳號管理與權限

### 8-1. 建立新使用者

1. 「設定 → 組織成員」→「邀請成員」或「新增使用者」
2. 填入帳號（username）、顯示名稱、Email，選擇角色
3. 使用者首次登入時觸發「三欄首登 Modal」，強制設定顯示名稱、Email 及密碼

### 8-2. 三層 RBAC 架構

| 層級 | 設定位置 | 作用範圍 |
|---|---|---|
| **Global（全域）** | 使用者 CRUD | 整個平台的預設角色 |
| **Org（組織）** | 設定 → 組織成員 | 在此組織內的角色 |
| **Project（專案）** | 設定 → 專案協作成員 | 特定專案的角色，可 override 組織角色的權限 |

權限解析優先序：**Project > Org > Global**

### 8-3. 角色與權限

- **設定 → 角色權限**：查看所有角色及其權限清單
- 可**複製**既有角色建立自訂角色
- **Per-project override**：在「設定 → 專案協作成員」中針對特定專案勾選或取消特定權限，不影響其他專案

### 8-4. 群組管理

- **設定 → 群組**：建立群組（部門 / 小組）
- 群組支援**巢狀**（上層群組包含子群組）
- 群組可作為**指派對象**（assignee），在通知與待辦清單中使用

### 8-5. 推薦角色配置

| 人員類型 | 全域角色 | 專案角色 | 用途 |
|---|---|---|---|
| 平台管理員 | `is_superuser=True` | — | 全平台管理 |
| 測試主管 | — | `Project-Reviewer` | 審核案例與報告 |
| QA 工程師 | — | `Project-Tester` | 撰寫案例、執行測試 |
| 唯讀利害關係人 | — | `Project-Viewer` | 查看 dashboard 與報告 |
| 外部協作者 | — | `Project-Viewer` | 有限度檢視 |

### 8-6. 邀請管理

在「設定 → 邀請管理」可以：

- **發送邀請**：Email 邀請（含有效期限）
- **重新發送**：若對方未收到 email
- **延長有效期**：邀請連結快過期時延長
- **撤回邀請**：取消尚未接受的邀請
- **批次操作**：同時處理多筆邀請

### 8-7. Audit Log

「設定 → Audit Log」查看所有 mutation 操作記錄：
- 操作者（username）
- 時間戳記
- 動作類型（CREATE / UPDATE / DELETE）
- 影響的 entity type 與 ID

資料保留 90 天，符合 SOC 2 baseline 要求。

## 9. 常見問題 FAQ

**Q：修改了測試案例，執行時還是跑舊步驟？**
確認已按「儲存」。案例修改後若未儲存，執行會使用上次儲存的版本。

**Q：執行報告顯示 FAILED，但看不到明確錯誤訊息？**
點開報告詳情，展開失敗的步驟查看截圖與 Robot log。也可查看 `docker logs autotest-celery` 找 root cause；最常見原因是 `RUNNER_CONTAINER_TIMEOUT_SEC` 設定過短，或 docker-proxy 連線異常。

**Q：WebSocket 日誌一直空白？**
WebSocket 連線未建立。開啟瀏覽器 DevTools → Network → WS，確認連線狀態。常見原因是 nginx reverse proxy 未設定 `Upgrade: websocket` 標頭。

**Q：本機 Agent 執行後沒有反應？**
確認 Agent 程式正在執行且連線狀態為「Connected」。Agent token 有效期可能已過期，請重新從 SPA 取得新 token。

**Q：排程設定好了但沒有自動觸發？**
確認「啟用」開關為開啟狀態，以及「下次執行時間」未過期。排程服務每 30 秒掃描一次，正常情況下誤差不超過 30 秒。

**Q：TestCase 被鎖定，無法編輯？**
該案例已通過審核（APPROVED），需到「審核中心」找到對應記錄，點「撤銷審核」（需填原因），狀態回到 PENDING 後即可繼續編輯。

**Q：錄製完成後步驟沒有比對條件？**
Playwright codegen 只記錄操作動作，不記錄斷言。需要手動在步驟表格補填「比較條件」與「預期值」欄位。

**Q：複製專案時，哪些內容不會被複製？**
不複製：執行報告、錄製腳本、排程設定、成員列表、環境變數。複製：完整六層樹結構、TestCase 步驟與 DDT、驗收準則、前置案例連結。

**Q：可以同時執行多個 TestCase 嗎？**
可以。在樹狀節點上右鍵 Feature / Platform / Page / Scenario → 執行，系統自動展開底下所有 TestCase 批次執行，並在同一份報告中整合結果。

**Q：如何備份資料？**

```bash
# 備份 PostgreSQL（完整資料）
docker compose exec -T postgres \
  pg_dumpall -U admin > backup_$(date +%Y%m%d_%H%M%S).sql

# 備份 SeaweedFS（截圖 / 影片 / trace 等媒體檔）
docker run --rm \
  -v $(docker compose config --format json | python3 -c "import sys,json; d=json.load(sys.stdin); print(list(d['volumes'].keys())[0])"):/data \
  -v $(pwd)/backup:/backup \
  alpine tar czf /backup/seaweedfs_$(date +%Y%m%d).tar.gz /data
```

**Q：`docker image prune -a -f` 之後測試無法執行？**
`-a` 旗標會刪除所有沒有容器參考的 image，包含 `autotest-robot-runner` 和 `autotest-recorder`（這兩個是按需 spawn，平常沒有常駐容器）。需重新執行 `docker compose --profile spawnable build` 重建這些 image。
