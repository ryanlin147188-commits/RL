## 1. 快速上手（5 分鐘）

### 啟動平台

需要 Docker 24+ 與 Docker Compose v2.23+，支援 Linux / macOS / Windows（Docker Desktop）。

```bash
git clone https://github.com/ryanlin147188-commits/RL-for-Kapito.git
cd RL-for-Kapito

# 步驟 1：自動產生 .env（含隨機 secret；若已存在則不覆寫）
docker compose --profile init run --rm bootstrap

# 步驟 2：預先建置執行用 image（首次約需 5–10 分鐘）
docker compose --profile spawnable build

# 步驟 3：啟動主服務
docker compose up -d --build
```

完成後，開啟瀏覽器前往 [http://localhost](http://localhost)。

### 登入

| 方式 | 帳號 | 密碼 |
|---|---|---|
| 帳密登入 | `admin` | `admin123` |
| Zoho SSO | 你的 Zoho 帳號 | — |

> 首次以帳密登入時，系統會強制要求修改密碼（需填寫顯示名稱、Email、新密碼）。
> 自助註冊已停用，新使用者由管理員從「設定 → 組織成員」建立。

### 認識介面

登入後，畫面分為：

- **頂部導覽列**：左側 Logo（點選回首頁）、右側使用者資訊
- **左側功能選單**：點擊左上角「≡」展開，列出所有功能入口
- **返回上一頁**：頁面切換時，頂部會出現「← 返回」按鈕，點擊可回到上一個頁面

## 2. 核心概念

### 六層測試樹結構

AutoTest 以六層樹來組織測試案例，對應業務邏輯與技術層次：

```
Project（專案）
└── Feature（業務功能，例：登入）
    └── Platform（平台，例：Web / API / APP）
        └── Page（頁面或端點，例：登入頁）
            └── Scenario（測試情境，例：正常登入）
                └── TestCase（測試案例 ← 真正的步驟在這裡）
```

這樣的分層讓報告可以按任一層聚合統計，管理者也可以按 Feature 整段選取執行。

### 測試案例的組成

每個 TestCase 由四個區塊組成：

| 區塊 | 說明 |
|---|---|
| **驗收準則** | 描述預期行為的文字，供審核者閱讀 |
| **前置動作（Setup）** | 每次執行前必須完成的動作，例如登入、開啟瀏覽器 |
| **測試步驟（Steps）** | BDD 格式的步驟清單，Given / When / Then / And |
| **資料驅動（DDT）** | 多組測試資料，讓同一案例反覆執行 |

### 步驟格式

每一個步驟包含以下欄位：

| 欄位 | 說明 | 範例 |
|---|---|---|
| BDD | Given / When / Then / And | `When` |
| 步驟說明 | 自然語言描述 | `輸入帳號` |
| 動作 | Robot Framework keyword | `Input` |
| 測試目標 | 元件定位符（Locator） | `id=username` |
| 輸入 | 輸入值 | `admin` |
| 比較條件 | 斷言方式 | `Equals` |
| 預期值 | 斷言的預期結果 | `登入成功` |

## 3. 基本操作教學

### A. 建立專案與目錄結構

1. 點擊左側選單頂部的**專案下拉選單** → 選「+ 新增專案」
2. 輸入專案名稱後確認
3. 進入「**測試案例**」頁面，左側顯示空白的樹狀結構
4. 在樹狀結構上**按右鍵** → 選「新增子節點」，依序建立：
   - Feature（例：使用者認證）
   - Platform（例：Web）
   - Page（例：登入頁）
   - Scenario（例：正常登入）
   - TestCase（例：admin 帳號登入成功）

### B. 撰寫第一個測試案例

建立 TestCase 節點後，點擊節點即開啟編輯區。

**填寫驗收準則：**

```
GIVEN 使用者在登入頁面
WHEN  輸入正確的帳號 admin 與密碼 admin123，按下登入按鈕
THEN  系統跳轉至首頁，頁面標題顯示 AutoTest
```

**新增測試步驟：**

| BDD | 步驟說明 | 動作 | 測試目標 | 輸入 | 比較條件 | 預期值 |
|---|---|---|---|---|---|---|
| Given | 開啟登入頁 | Navigate | — | http://localhost | — | — |
| When | 輸入帳號 | Input | `id=username` | admin | — | — |
| When | 輸入密碼 | Input | `id=password` | admin123 | — | — |
| When | 點擊登入 | Click | `text=登入` | — | — | — |
| Then | 確認跳轉首頁 | Wait Element | `id=mainHeader` | — | Contains | AutoTest |

完成後按右上角「**儲存**」。

### C. 執行測試與查看報告

**執行：**

1. 在 TestCase 頁面按右上角「**執行**」
2. 選擇執行模式：
   - **Docker 模式**（預設）：後台自動起一個 robot-runner 容器執行
   - **本機模式**：需先啟動本機 Agent（見 [§5-11](#5-11-本機-agent有頭模式)）
3. 執行中可即時查看 WebSocket 日誌

**查看報告：**

1. 前往「**執行報告**」頁面
2. 點擊執行記錄進入報告詳情
3. 報告包含：
   - 整體統計（總步驟、通過數、失敗數、執行時間）
   - 每個步驟的截圖
   - 完整影片錄製（`.mp4`）
   - Playwright trace（可下載後拖入 [trace.playwright.dev](https://trace.playwright.dev) 分析）
   - 詳細 Robot Framework 日誌
4. 右上角「**匯出 PDF**」可下載報告

## 4. 錄製器

### D. Web 錄製

#### 本機模式

適合已安裝 Node.js / Playwright 的開發環境。

1. 前往「**錄製器**」頁面 → 選「Web」→「本機」
2. 點擊「建立 Session」
3. 系統回傳啟動指令（格式類似 `npx playwright codegen ...`）
4. 在本機終端機執行該指令，Playwright Inspector 視窗開啟
5. 在瀏覽器上操作你要錄製的流程
6. 操作完畢後關閉 Playwright Inspector
7. 回到 SPA，將 Inspector 產生的腳本貼回
8. 平台自動解析成 BDD 步驟陣列

#### Docker 模式

無需本機安裝任何依賴。

1. 選「Web」→「Docker」
2. 系統起一個含 noVNC 的 `autotest-recorder` 容器
3. SPA 內嵌 iframe 即可看到遠端瀏覽器畫面
4. 使用工具列控制（點擊、輸入、截圖）完成錄製
5. 結束後系統自動解析步驟

> 首次啟動容器約需 20–30 秒。

### E. API 錄製

#### 貼上 cURL（快速方式）

1. 選「API」→「貼 cURL」
2. 將 cURL 指令整段貼入，例如：
   ```bash
   curl -X POST https://api.example.com/auth/login \
     -H 'Content-Type: application/json' \
     -d '{"username":"admin","password":"admin123"}'
   ```
3. 平台解析 method、URL、headers、body，轉成 `HTTP Request` 步驟

支援 zsh / bash / Windows CMD 等多種語法，包含 `-H`、`--data`、`--cookie`、bearer token 等常見參數。

#### mitmproxy Docker 模式（完整擷取）

適合需要錄製完整 API 對話的複雜 SPA。

1. 選「API」→「Docker」
2. 系統起一個含 mitmproxy 的 `autotest-recorder-api` 容器
3. 將瀏覽器或 app 的流量導向此 proxy
4. 操作結束後，系統自動將所有 HTTP 流量解析為步驟陣列

### F. App 錄製

1. 選「App」
2. 使用 Appium Inspector 錄製你的 app 操作，並將 Python 腳本貼入
3. 平台解析成 `AppiumLibrary` keyword 的步驟陣列

### 套用錄製結果

錄製解析完畢後，有三種套用方式：

1. **建立新案例**：跳出 Scenario 選擇 dialog，選擇父節點後一鍵建立新 TestCase
2. **套用至現有案例**：選擇一個既有的 TestCase，以錄製步驟覆蓋
3. **手動微調後套用**：在預覽區逐行編輯，再套入

## 5. 進階功能

### 5-1. Capture — 從畫面或 API 擷取變數

將執行過程中的值儲存成變數，供後續步驟使用。

**從畫面元素擷取：**

| 動作 | 測試目標 | 輸入（變數名稱） |
|---|---|---|
| Capture Element Text | `.user-greeting` | `welcome_msg` |

**從 API 回應擷取（JSONPath）：**

| 動作 | 測試目標（JSONPath） | 輸入（變數名稱） |
|---|---|---|
| Capture API Response | `$.data.user.id` | `user_id` |

**後續引用：**

```
{{= welcome_msg }}
{{= user_id }}
```

### 5-2. 條件分支（If / ElseIf / Else / EndIf）

根據 Capture 的結果或環境變數，讓測試走不同路徑。

| 動作 | 比較條件 | 預期值 |
|---|---|---|
| If | (Captured `user_role`) Equals | admin |
| （管理員專屬步驟群組） | | |
| ElseIf | (Captured `user_role`) Equals | viewer |
| （一般使用者步驟群組） | | |
| Else | | |
| （其他情況步驟群組） | | |
| EndIf | | |

平台將這些轉換成 Robot Framework 7 的 `IF / ELSE / END` 區塊，在 runner 內真實執行條件分支。

### 5-3. 動態運算式

在步驟的任何輸入欄位使用 `{{= expression }}` 語法，執行時即時計算。

| 類型 | 範例 | 說明 |
|---|---|---|
| 變數引用 | `{{= row.username }}` | DDT 欄位 |
| 環境變數 | `{{= env.STAGING_URL }}` | 環境變數表 |
| 字串拼接 | `{{= env.HOST + "/api/v1" }}` | 組合 URL |
| 隨機資料 | `{{= fakerName() }}` | 每次執行產生不同值 |
| 唯一識別碼 | `{{= uuid() }}` | UUID v4 |
| 時間戳記 | `{{= now() }}` | 當前時間 |
| JSON 路徑 | `{{= json(captured.resp, "$.id") }}` | 從 JSON 取值 |

### 5-4. 資料驅動測試（DDT）

讓單一案例對多組資料重複執行，適合登入驗證、表單驗證等情境。

**步驟一：建立資料表**

在案例右側「DDT」分頁新增欄位與資料列，例如：

| username | password | expected |
|---|---|---|
| admin | admin123 | 登入成功 |
| admin | wrong | 密碼錯誤 |
| blocked | pass123 | 帳號已停用 |

**步驟二：在步驟中引用欄位**

```
輸入帳號欄位：{{= row.username }}
輸入密碼欄位：{{= row.password }}
驗證訊息包含：{{= row.expected }}
```

**執行結果**：報告中該案例將顯示 3 次執行記錄，各自的截圖與日誌分開保存，pass / fail 個別統計。

### 5-5. 測試排程

自動在指定時間執行測試，無需人工觸發。

**建立排程：**

1. 前往左側選單「**測試排程**」
2. 點擊右上角「+ 新增排程」
3. 填寫以下設定：
   - **排程名稱**：識別用名稱
   - **目標節點**：選擇要執行的測試案例（僅顯示 TestCase 層節點，支援多選）
   - **重複類型**：
     - **ONCE**：僅執行一次
     - **DAILY**：每天同一時間
     - **WEEKLY**：每週指定的星期幾（可多選）
     - **MONTHLY**：每月指定日期（1–31，超出該月天數自動補至月底）
   - **執行時間**：日期與時間（以伺服器本地時間為準）
   - **執行環境**：Docker（預設）或本機 Agent
4. 按「儲存」

**排程管理：**

| 操作 | 說明 |
|---|---|
| 啟用 / 停用 | 點擊「啟用」開關切換排程狀態 |
| 立即執行 | 點擊「立即執行」，不等下次觸發時間 |
| 查看上次報告 | 點擊「查看報告」連結至最近一次的執行報告 |
| 刪除 | 點擊「刪除」按鈕（需確認） |

> 排程每 30 秒掃描一次，誤差在 30 秒以內。

### 5-6. 測試回合（Test Round）

將多次執行群組在一起，形成一個測試週期（例如「Sprint 23 Smoke Round」）。

1. 在測試案例樹上右鍵節點 → 選「加入測試回合」
2. 選擇目標 Round 或建立新的
3. 進入「測試回合」頁面，可看到該 Round 的整體 dashboard：
   - 執行次數、通過率
   - 各案例最新狀態
   - 時間走勢圖

### 5-7. 審核中心

對測試案例、錄製腳本、執行報告進行審核流程管理。

**審核工作流程：**

```
送審（PENDING）→ 審核通過（APPROVED）
               → 審核退回（REJECTED）→ 重新送審（PENDING）
```

**操作說明：**

1. 前往「**審核中心**」
2. 切換分頁：
   - **待審核**：待你審核的項目
   - **已審核**：你已通過的項目（顯示審核者與審核時間）
   - **已拒絕**：已退回的項目（顯示審核者與審核時間）
3. 點擊任一列可開啟**詳情 Modal**，查看：
   - 送審的實體內容（報告的執行結果、測試案例的步驟、腳本的錄製內容）
   - 完整審核歷程（誰、何時、做了什麼決定、填寫了什麼原因）
4. 在詳情 Modal 底部可進行：
   - **通過**：將狀態改為 APPROVED（僅限 PENDING 狀態）
   - **退回**：將狀態改為 REJECTED，需填寫退回原因
   - **查看報告 / 前往案例 / 前往腳本**：直接跳轉到對應的實體頁面

> 被 APPROVED 的測試案例會被**鎖定**，無法修改，必須先「撤銷審核」才能繼續編輯。

### 5-8. 待辦清單（Backlog）

管理測試相關的任務、缺陷、探索工作。

**進入方式：** 左側選單「待辦清單」，或在測試案例 / 報告詳情頁的 linked-todos popover。

**項目類型：**
- **Task**：一般任務
- **Bug**：缺陷
- **Spike**：探索研究

**功能：**
- Feature → 項目的兩層階層
- Sprint 標籤（選填）
- 過期日期，自動顯示過期 / 即將過期 badge
- assigned_to（可指定人員或群組）

### 5-9. 環境變數

每個專案有獨立的環境變數表，在測試步驟中以 `{{= env.KEY }}` 引用。

**設定方式：**

1. 前往「**環境變數**」頁面
2. 選擇專案
3. 新增 key-value 對

**常見用途：**

```
STAGING_URL = https://staging.myapp.com
TEST_USERNAME = qa-bot@company.com
API_KEY = xxxxxxxxxxxxxx
```

在步驟中使用：

```
Navigate 到 {{= env.STAGING_URL }}/login
```

### 5-10. Mock 端點

為尚未完成的 API 後端提供假回應，讓前端測試可以繼續進行。

1. 前往「**Mock 端點**」頁面
2. 新增一個 mock：
   - **路徑**：例如 `/api/users/me`
   - **方法**：GET / POST / PUT / DELETE
   - **回應 Body**：JSON 或純文字
   - **HTTP Status Code**：例如 200、404
3. 測試步驟中將請求指向 `http://localhost/mock/<路徑>` 即可

每個 Mock 端點屬於特定組織範圍，不同組織之間不會互相干擾。

### 5-11. 本機 Agent（有頭模式）

讓測試在你的實體桌面瀏覽器上執行，而非 headless 容器，便於視覺化 debug。

**設定步驟：**

1. 前往「設定 → 使用者設定 → 本機 Agent」
2. 下載適合你作業系統的 Agent 安裝包
3. 安裝後執行 Agent，輸入：
   - Backend URL：`http://<server-ip>`
   - Token：從 SPA 使用者設定頁取得
4. Agent 連線成功後，執行測試時切換「**本機模式**」

### 5-12. Markdown 匯入 / 匯出

測試案例的步驟以標準 Markdown 格式儲存，支援雙向轉換，便於版本控制。

**匯出：**
- 在專案樹任一節點上右鍵 → 「匯出為 Markdown」
- 下載包含所有子節點案例的 `.zip` 檔案

**匯入：**
- 右鍵專案樹 → 「匯入 Markdown」
- 上傳 `.md` 檔案，系統自動建立對應的樹節點與案例內容

> Markdown 格式為 7 欄表格（BDD / 步驟說明 / 動作 / 測試目標 / 輸入 / 比較條件 / 預期值），可使用任何文字編輯器編輯後重新匯入。

### 5-13. REST API 與 CI/CD 整合

AutoTest 提供完整的 REST API，支援從 CI/CD pipeline 觸發執行。

**取得 JWT Token：**

```bash
TOKEN=$(curl -s -X POST http://<host>/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"ci-bot","password":"your-password"}' \
  | jq -r .access_token)
```

**觸發測試執行：**

```bash
curl -X POST http://<host>/api/executions \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{
    "node_id": "<testcase-uuid>",
    "trigger_type": "CI",
    "execution_mode": "docker"
  }'
# 回傳 task_id 與 report_id
```

**查詢執行狀態：**

```bash
curl http://<host>/api/executions/<task_id>/status \
  -H "Authorization: Bearer $TOKEN"
# 回傳 status: running / passed / failed / cancelled
```

**GitHub Actions 範例：**

```yaml
- name: Run AutoTest
  run: |
    TOKEN=$(curl -s -X POST ${{ secrets.AUTOTEST_URL }}/api/auth/login \
      -H 'Content-Type: application/json' \
      -d '{"username":"${{ secrets.AUTOTEST_USER }}","password":"${{ secrets.AUTOTEST_PASS }}"}' \
      | jq -r .access_token)
    curl -X POST ${{ secrets.AUTOTEST_URL }}/api/executions \
      -H "Authorization: Bearer $TOKEN" \
      -H 'Content-Type: application/json' \
      -d '{"node_id":"${{ vars.SMOKE_NODE_ID }}","trigger_type":"GitHub Actions","execution_mode":"docker"}'
```

完整 API 文件可從 backend 容器內取得：

```bash
docker compose exec backend curl -s localhost:8000/openapi.json | jq .
```

## 6. 帳號管理與權限

### 建立新使用者

1. 前往「設定 → 組織成員」
2. 點擊「邀請成員」或「新增使用者」
3. 輸入使用者名稱（帳號）、顯示名稱、Email
4. 選擇組織角色
5. 完成後系統發送邀請 Email（或由管理員告知初始密碼）

### 三層 RBAC 架構

| 層級 | 設定位置 | 說明 |
|---|---|---|
| **全域（Global）** | 使用者設定 | 適用整個平台的角色（Admin / QA / Viewer）|
| **組織（Org）** | 設定 → 組織成員 | 在此組織內的角色 |
| **專案（Project）** | 設定 → 專案協作成員 | 在特定專案的角色，可覆蓋組織角色權限 |

權限解析優先順序：**ProjectMember > OrgMembership > User**

### 推薦的角色配置

| 人員類型 | 全域角色 | 專案角色 | 用途 |
|---|---|---|---|
| 平台管理員 | is_superuser=True | — | 全平台管理 |
| 測試主管 | — | Project-Reviewer | 審核測試案例與報告 |
| QA 工程師 | — | Project-Tester | 撰寫案例、執行測試 |
| 唯讀利害關係人 | — | Project-Viewer | 查看 dashboard 與報告 |

### 群組管理

- 前往「設定 → 群組」建立群組
- 群組可以巢狀（部門 → 小組）
- 群組可作為 assignee（指派對象）

### Audit Log

- 前往「設定 → Audit Log」查看所有操作記錄
- 記錄內容：操作者、時間、動作類型、影響對象
- 資料保留 90 天，符合 SOC 2 baseline

## 7. 常見問題 FAQ

**Q：修改了測試案例，執行時還是跑舊的步驟？**
確認已按「儲存」。案例修改後若未儲存，執行會使用上次儲存的版本。

**Q：執行報告顯示 FAILED，但日誌看不到錯誤？**
通常是 runner 容器超時或被系統終止。查看 `docker logs autotest-celery` 或 Audit Log，最常見原因是 `RUNNER_CONTAINER_TIMEOUT_SEC` 設定過短。

**Q：WebSocket 日誌一直空白沒有輸出？**
WebSocket 連線未建立成功。開啟瀏覽器 DevTools → Network → WS，確認連線狀態。常見原因是 nginx reverse proxy 未設定 `Upgrade: websocket` 標頭。

**Q：本機 Agent 模式按執行但沒有反應？**
Agent 未啟動或未連線。確認 Agent 程式正在執行，且連線狀態顯示為「Connected」。

**Q：排程設定好了但沒有自動執行？**
確認排程「啟用」開關為開啟狀態，以及「下次執行時間」尚未過期。排程每 30 秒掃描一次，正常情況下誤差不超過 30 秒。

**Q：錄製完成後步驟的「預期值」欄位都是空白？**
Playwright codegen 只記錄操作動作，不記錄斷言。錄製完成後需要手動在步驟表格中補填「比較條件」與「預期值」欄位。

**Q：DDT 執行完看不到各筆資料的結果？**
在報告詳情頁，DDT 每一筆資料的執行結果以獨立折疊區塊顯示，點擊展開即可看到該筆的截圖與日誌。

**Q：測試案例被鎖定，無法編輯？**
該案例已通過審核（APPROVED 狀態），需先到「審核中心」對該記錄執行「撤銷審核」，才能重新編輯。

**Q：可以同時執行多個測試案例嗎？**
可以。在測試樹上右鍵任一 Feature / Platform / Page / Scenario 節點 → 執行，系統會展開該節點下的所有 TestCase 並批次執行。

**Q：想把資料庫備份起來，怎麼做？**

```bash
# 備份 PostgreSQL
docker compose exec postgres pg_dumpall -U postgres > backup_$(date +%Y%m%d).sql

# 備份 SeaweedFS（媒體檔案）
# 直接備份 Docker volume：seaweedfs_data
docker run --rm \
  -v rl-for-kapito_seaweedfs_data:/data \
  -v $(pwd):/backup \
  busybox tar czf /backup/seaweedfs_$(date +%Y%m%d).tar.gz /data
```
