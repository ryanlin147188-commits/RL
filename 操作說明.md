## 1. 5 分鐘快速上手

### Step 1：啟動平台

**需求**：Docker 24+ / Docker Compose v2（Windows / macOS / Ubuntu / Linux 三平台通用）。

**Windows (PowerShell)**
```powershell
cd <專案資料夾>
docker build -f backend/Dockerfile.runner -t autotest-robot-runner:latest backend/
docker compose up -d --build
start http://localhost
```

**macOS / Ubuntu / Linux (bash / zsh)**
```bash
cd <專案資料夾>
docker build -f backend/Dockerfile.runner -t autotest-robot-runner:latest backend/
docker compose up -d --build
open http://localhost       # macOS
xdg-open http://localhost   # Linux
```

看到登入頁就成功了。

> 🍎 **Apple Silicon (M 系列) Mac**：可正常使用，但 Runner 容器為 amd64，啟動會比原生慢 2-4 倍。
> ⚠ `.env` 檔必須有 `STORAGE_BACKEND=minio`，且 MySQL / MinIO 的預設帳密也在 `.env` 設定（預設 `admin / admin123`）。沒有檔案？複製 [README.md](README.md) 內的範本即可。

### Step 2：登入

第一次進入 <http://localhost/> 會看到登入畫面：

- **預設帳號**：`admin` / `admin123`（已預填於登入表單）
- **註冊**：點上方「註冊」分頁建立新帳號（使用者名稱 3-32 字元、密碼 ≥ 6 字元）
- **忘記密碼**：點「忘記密碼」分頁 → 輸入使用者名稱 + 註冊 Email + 新密碼，直接重設
- **暗黑模式**：右上角 🌙 按鈕切換亮 / 暗主題，偏好自動儲存到瀏覽器

登入後右上角顯示使用者頭像與名稱；點下拉可進入「使用者設定」或登出。

### Step 3：認識首頁

進入 <http://localhost/> 登入成功後看到：

- **頂部導覽列**（桌面：寬螢幕顯示文字，中寬螢幕自動切為 icon-only 節省空間）：設備資訊 / 測試案例 / 環境變數 / DB 資訊 / Mock / 測試回合 / 測試報告 / 測試排程 / 測試錄製 / 環境切換（Docker / 本機）
- **左上角 AutoTest Logo**：任何時候點擊可回首頁
- **左側專案目錄**：建立專案後顯示 5 層樹狀結構；預設節點全部收合，點 ▸ 展開
- **側邊欄工具列**：搜尋框 + ⬇ 全部展開 + ⬆ 全部收合；標題右側有橘色 📁+ 新增專案
- **當前專案概況**：專案數、測試案例數、節點總數、測試回合數一目了然
- **系統狀態面板**：Docker 狀態 / CPU / 記憶體 / 儲存空間 / 網路流量（每 5 秒即時更新）
- **快速導覽卡片**：9 張功能卡，一鍵跳轉
- **最近測試案例**：最近建立 / 編輯的最多 10 筆案例

### Step 4：建立第一個專案

1. 左側「專案目錄」標題右邊的橘色 <kbd>📁+</kbd> 按鈕
2. 輸入專案名稱（例：`官網測試`）→ 確定
3. 系統自動切到新專案，自動回首頁顯示新專案的概況

### Step 5：建立目錄與第一個測試案例

按照 5 層結構建立（**由上到下**）：

```
功能 (Feature)：帳號管理
 └─ 平台 (Platform)：WEB
     └─ 頁面 (Page)：登入頁
         └─ 情境 (Scenario)：正向
             └─ 測試案例 (TestCase)：成功登入.md   ← 點這個開始編輯
```

**操作方式**：滑鼠移到節點上，右邊出現 <kbd>➕</kbd> <kbd>✏</kbd> <kbd>🗑</kbd> 按鈕。點 ➕ 逐層建立子節點，到 TESTCASE 層就能點進去編輯。

### Step 6：跑起來

1. 點開 `成功登入.md`
2. 右側編輯器出現四個區塊（驗收準則 / 前置動作 / BDD 步驟 / DDT 資料）
3. 在「測試步驟設計」區新增一列，動作選 `Navigate`，輸入值填 `https://www.google.com`
4. 按右上角 <kbd>💾 儲存</kbd>
5. 按 <kbd>▶ 執行測試</kbd>
6. 底部自動彈出終端機，看到 `🐳 啟動容器` → `🏁 容器結束` 就完成了
7. 切到「測試報告」看結果

🎉 恭喜，你剛跑完人生第一個自動化測試。

## 2. 核心概念速懂

花 3 分鐘看完這段，後面的旅程會順很多。

### 五層目錄：為什麼要這樣分？

| 層級 | 代表 | 例子 | 用途 |
|---|---|---|---|
| **Feature（功能）** | 一個業務功能 | 帳號管理 / 購物車 / 會員中心 | 按需求領域切分 |
| **Platform（平台）** | 測試的技術平台 | WEB / API / APP / DB / E2E / UI | 決定用哪個底層 Library |
| **Page（頁面）** | UI 頁面 / API 資源 / 資料表 | 登入頁 / 商品 API / users 表 | 更細的分類 |
| **Scenario（情境）** | 測試意圖 | 正向 / 反向 / 邊界 | 區分 happy path 與錯誤情境 |
| **TestCase（測試案例）** | 實際跑的測試 `.md` | 成功登入.md / 忘記密碼.md | **只有這一層可以跑** |

💡 **重點**：**Platform 會決定你能用哪些動作**。把案例放在 WEB 底下 → 只能用 WEB 動作；放在 DB 底下 → 只能用 `Db.*` 動作；E2E 平台則全部可用。

### 測試案例的四區塊

每個 TESTCASE 打開後都有四塊：

| 區塊 | 內容 | 給誰看 |
|---|---|---|
| **1. 驗收準則 (AC)** | 業務描述「怎樣算通過」 | PM / BA / QA 對齊需求 |
| **2. 前置動作 (Pre-Setup)** | 執行前需要準備的事（文字說明） | QA 手動準備或寫在步驟裡 |
| **3. BDD 步驟表** | 具體執行什麼、驗證什麼 | 系統自動執行 |
| **4. DDT 資料表** | 同一個測試跑多組資料 | 擴增覆蓋率 |

### 「動作」與「比對條件」

BDD 步驟表的兩大欄位：

- **動作 (Action)**：你要做什麼（`Click` / `Fill` / `Http.POST` / `Db.Query` ...）— 共 **100+ 種**
- **比對條件 (Condition)**：怎麼驗證（`Equals` / `Contains` / `Regex` / `GreaterThan` ...）— 共 **12 種**

### 存資料的格式

- **測試案例內容** → 儲存於後端資料庫（MySQL）
- **截圖 / 錄影 / Trace** → 存到 MinIO（物件儲存），跨機器可看
- **環境變數 / 設備 / Mock / DB 連線** → 環境變數與設備存後端；Mock 與 DB 連線目前存瀏覽器 `localStorage`

### A. 寫出你的第一個 WEB UI 測試（登入案例）

**目標**：驗證官網登入頁用正確帳密可成功登入並跳轉。

**預期時間**：5-10 分鐘。

#### 建立案例

參考快速上手 Step 4，建立到 `帳號管理 → WEB → 登入頁 → 正向 → 成功登入.md`。

#### 填驗收準則

```
使用者輸入正確帳號與密碼後，應導向 /dashboard
並顯示歡迎訊息「Hi, {username}」。
```

#### 填前置動作

```
- 確認 staging 環境可連線
- 已建立測試帳號：demo / Pa$$w0rd
```

#### 寫 BDD 步驟

按 <kbd>+ 新增步驟</kbd> 逐列填入（每列可多行描述）：

| # | BDD | 步驟描述 | 動作 | 定位器 | 輸入值 | 比對條件 | 預期結果 |
|---|---|---|---|---|---|---|---|
| 1 | Given | 開啟登入頁 | `Navigate` | | `https://staging.example.com/login` | | |
| 2 | When | 輸入帳號 | `Fill` | `#username` | `demo` | | |
| 3 | When | 輸入密碼 | `Fill` | `#password` | `Pa$$w0rd` | | |
| 4 | When | 點擊登入按鈕 | `Click` | `button[type=submit]` | | | |
| 5 | Then | 驗證 URL 跳轉 | `AssertURL` | | | `Contains` | `/dashboard` |
| 6 | Then | 驗證歡迎訊息 | `AssertText` | `.welcome` | | `Contains` | `Hi, demo` |

#### 儲存與執行

1. 按 <kbd>💾 儲存</kbd>
2. 按 <kbd>▶ 執行測試</kbd>
3. 底部終端機會即時串流每個步驟的 log
4. 跑完切到「測試報告」看結果

> 🎯 **技巧**：不確定定位器怎麼寫？用下一段的「錄製器」就不用手寫。

### B. 用錄製器 10 倍加速

手寫定位器（`#username`、`button[type=submit]`）很費時。**錄製器**讓你用真的瀏覽器操作一次，系統自動產生步驟。

#### B-1. WEB 錄製

1. 先在左側樹**點選一個 TESTCASE**（錄製結果會合併到這個案例）
2. 頂部點「錄製」
3. 輸入目標 URL → <kbd>建立錄製階段</kbd>
4. 畫面顯示兩段一鍵指令（**依你的作業系統選一個複製**）：
   - **Windows**：PowerShell 一鍵指令
   - **macOS / Linux**：bash 一鍵指令
5. 貼到終端機 Enter → Playwright Inspector 自動打開
6. **在瀏覽器中操作**：點、輸入、送出...
7. 想自動產生斷言？點 Inspector 工具列的 <kbd>Assert visibility</kbd> / <kbd>Assert text</kbd> / <kbd>Assert value</kbd>，再點頁面元素
8. 關閉瀏覽器 → 自動上傳
9. 回到錄製頁點 <kbd>套用至當前案例</kbd> → 步驟合併到編輯器

#### B-2. API 錄製（貼 cURL）

1. 在瀏覽器 DevTools Network 面板，對 API 請求按右鍵 → Copy → Copy as cURL
2. 切到錄製頁「API」分頁
3. 貼上（可一次貼多個，空行分隔）
4. 按 <kbd>解析為 Http.\* 步驟</kbd> → 自動轉成 `Http.GET` / `Http.POST` + Headers + Body

#### B-3. APP 錄製（貼 Appium 腳本）

1. 從 Appium Inspector 匯出 Python 腳本
2. 切到錄製頁「APP」分頁，貼上
3. 按 <kbd>解析為 Mobile.\* 步驟</kbd> → 自動轉成 `Mobile.Click` / `Mobile.Input`

### C. 讓一個案例跑 10 組資料（DDT）

**情境**：手機號碼驗證，想一次測 10 種格式（台灣 / 美國 / 國際 / 亂碼 / 空值...）。

#### 建立 DDT 資料列

1. 打開案例，拉到最下面「測試資料集 (DDT)」
2. 按 <kbd>+ 新增資料列</kbd> 加入多筆資料
3. 填入每列的 `Key` / `Value`，例如：

| No | 檔案名稱 | Key | Value |
|---|---|---|---|
| 1 | TW 正常 | phone | 0912345678 |
| 2 | TW 空格 | phone | 0912 345 678 |
| 3 | US 正常 | phone | +1-555-0100 |
| 4 | 亂碼 | phone | abcdefg |
| 5 | 空值 | phone | |

#### 在步驟裡引用

在 BDD 步驟的欄位中用 `${phone}` 語法：

| 動作 | 定位器 | 輸入值 |
|---|---|---|
| `Fill` | `#phone` | `${phone}` |

#### 執行時展開

**預設**：只拿第一列資料當變數跑一次。
**想每列都跑一次**：目前前端預設不開展開，需透過 API 送出 `ddt_expand=true`：

```bash
curl -X POST http://localhost/api/executions \
  -H "Content-Type: application/json" \
  -d '{"node_id": "<case_id>", "ddt_expand": true}'
```

每列會獨立產生錄影與 Trace，報告中可個別檢視。

### D. 安排每天自動跑回歸

**情境**：每天凌晨 3 點自動跑一次全套回歸測試，發現問題早上進公司前就看得到結果。

#### 建立排程

1. 頂部「排程」→ <kbd>新增排程</kbd>
2. 填入：
   - **名稱**：每日回歸
   - **重複規則**：`每天` → 時間填 `03:00`
   - **執行環境**：`Docker`（無頭、穩定）
   - **測試案例**：在清單中搜尋並勾選要跑的 `.md`（可多選）
3. 啟用 → 儲存

#### 確認運作

- 後端每 30 秒掃一次到期排程，最多 30 秒內觸發
- 排程列表的 <kbd>立即</kbd> 按鈕可手動觸發一次驗證
- 執行結果會進「測試報告」，依時間倒序顯示

#### 對整個 Feature 或 Scenario 排程

目前 UI 只能多選 TESTCASE；想對 FEATURE / PAGE 整批排程需直接呼叫 API。

### E. 讀懂執行報告

跑完後到頂部「測試報告」，你會看到三層畫面。

#### E-1. 儀表板（最上層）

- **通過率圓環圖**：一眼看今天或最近範圍過多少
- **趨勢折線**：看每天表現有沒有變差
- **Feature 通過率長條**：哪個功能最不穩定

#### E-2. 最近執行紀錄（中層）

每列欄位：**執行時間 / 專案 / 平台 / 頁面 / 測試案例 / 觸發者 / 成功·失敗 / 結果 / 耗時**。

- 狀態徽章：`RUNNING`（執行中，會每 5 秒自動更新） / `PASS` / `FAIL`
- 點列的測試案例 → 進入報告詳情

#### E-3. 報告詳情（下層）

左側是步驟時間軸，右側依動作類型自動切換面板：

- **UI 步驟** → 顯示前/後截圖（目標元素紅框高亮）
- **API 步驟** → 顯示 Request / Response JSON 雙欄 + 狀態碼 + 耗時
- **APP 步驟** → 手機直式框架內的 Appium 截圖
- **DB 步驟** → SQL 程式碼 + 結果表 / 影響列數

#### E-4. 進階檢視

**完整錄影**：任一步驟點開 → 右側「軌跡 / 錄影」面板 → <kbd>▶ 完整錄影</kbd> 播整個案例 `.webm`。

**Playwright Trace**：
- <kbd>下載 Trace</kbd> → 本機用 `playwright show-trace trace.zip` 開啟
- <kbd>Trace Viewer ↗</kbd> → 新分頁開啟線上版
- <kbd>嵌入檢視</kbd> → 報告頁內 iframe 嵌入完整 Trace Viewer（DOM 快照、Network、Console）

**Screenshot Diff**：
- 用 `AssertScreenshotMatch` 動作的步驟會多出紫色面板
- 三聯比對：**Baseline | Actual | Diff（紅色覆蓋差異）**
- 顯示實際差異 %
- 如果「現在這個畫面才對」，點 <kbd>📷 把當下 actual 設為新 baseline</kbd>

#### E-5. 匯出 PDF

在報告詳情頁用瀏覽器「列印 → 另存 PDF」即可得到一份分類呈現的報告（UI 截圖 / API JSON / DB 結果表 / APP 截圖各自分段）。

## 4. 進階功能

走完五段旅程後，這些是你可能會需要的補充能力。

### 使用者帳號 / 登入管理

**預設帳號**：`admin` / `admin123`（首次啟動自動建立於瀏覽器 localStorage）。

- **註冊新使用者**：登入頁點「註冊」分頁 → 填入使用者名稱（3-32 字元英數底線）、Email、密碼（≥ 6 字元、兩次確認）→ 建立帳號後自動切回登入頁
- **忘記密碼**：輸入使用者名稱 + 註冊 Email + 新密碼 → 驗證相符即時重設
- **使用者設定**（點右上角使用者頭像 → 使用者設定）：
  - **帳戶資訊**：變更 Email、顯示名稱（顯示名稱會出現在報告「觸發者」欄位）
  - **變更密碼**：舊密碼驗證 + 新密碼兩次確認
  - **個人偏好**：預設執行環境（Docker / 本機）、自動刷新、切換未儲存提示
  - **登入資訊**：最近登入時間、登入次數

> ⚠ 目前認證是 demo 等級（前端 localStorage，未串接後端）；正式部署請改用後端 JWT / session。

### 暗黑模式

右上角使用者頭像左側有 🌙 / ☀ 按鈕，一鍵切換亮 / 暗主題；選擇自動儲存到 localStorage，下次開頁直接套用（不閃爍）。

暗黑模式覆蓋：body 背景 / 卡片 / 表單 / 邊框 / 捲軸；保留功能色（紅綠藍紫橙）與漸層按鈕維持辨識度。

### 首頁系統狀態面板

首頁「當前專案概況」下方的「系統狀態」區塊即時顯示 **AutoTest 平台的資源使用量**（以所有 `autotest-*` 容器加總），並對照 Docker host 資訊：

| 卡片 | 實際內容 | 資料來源 |
|---|---|---|
| **Docker** | Docker Engine 版本、容器 running / total、映像數量 | `docker.info()` + `docker.version()` |
| **CPU 使用率** | **所有 autotest-\* 容器 CPU % 加總**（多核系統可能 > 100%）| docker SDK `container.stats()` 差值計算 |
| **記憶體** | **平台容器實際使用量 / Host 總記憶體**（百分比以 Host 總量為分母）| `container.stats()` 記憶體工作集 + `info().MemTotal` |
| **儲存空間** | backend 容器根檔案系統（與 Docker data volume 同 data root） | `os.statvfs('/')` |
| **網路流量** | 所有平台容器 RX / TX 累計 + 即時速率 | `container.stats().networks` 差值計算 |
| **Host 對照列** | Docker host 作業系統 / CPU 核心數 / 記憶體總量 / 平台容器數量 | `docker.info()` |

**輪詢策略**：只在首頁可見時每 5 秒更新一次，離開首頁自動停止節省資源。
**API 端點**：`GET /api/system/status`（在 Swagger 有完整 schema；`scope: "autotest-platform"` 欄位明示資料範圍）。

> 💡 **為什麼不是 Host 整機？** 因為 backend 在容器內跑，直接讀 `/proc` 會混合 host 與 cgroup 限制，語意不明。
> 改用 Docker SDK 加總「所有 `autotest-*` 容器」後，數字即是「**測試平台佔用了你的伺服器多少資源**」，商業評估更直覺。

### 測試回合 (Test Round)

把一組 `.md` 測試案例**命名**後存成一個「回合」，一鍵執行全部並彙總為一份報告。
適合做 **Smoke Test**、**Release 驗證**、**特定 Feature 回歸**。

1. 頂部「測試回合」→ <kbd>新增回合</kbd>
2. 取名字（例：`上線前 Smoke`）、勾選案例、選預設執行環境
3. 儲存後在列表可 <kbd>執行</kbd> / <kbd>編輯</kbd> / <kbd>刪除</kbd>

### 本機 Agent（有頭模式）

想看瀏覽器跑？適合 demo、教育訓練、debug 困難案例。

1. 頂部環境切到「本機」→ 出現 <kbd>?</kbd> 按鈕 → 開啟說明
2. 下載 `local_agent.py`
3. 安裝依賴：
   ```bash
   pip install playwright requests
   playwright install chromium
   ```
4. 啟動：
   ```bash
   python local_agent.py --server http://localhost
   ```
5. 後續執行會由 agent 認領，開啟有頭 Chromium

**限制**：本機 Agent 目前只支援 Web UI；`Http.*` 與 `Mobile.*` 請改用 Docker 模式。

### 環境變數 + Faker 隨機資料

頂部「環境變數」→ 維護全專案共用的 Robot suite variables：

- 每列填 `NAME=VALUE`，在步驟中用 `${NAME}` 引用
- 含 `TOKEN` / `SECRET` / `PASSWORD` 字樣自動遮蔽
- **🎲 Faker 按鈕**：28 種隨機資料類型（姓名、Email、UUID、手機、日期、信用卡...）一鍵產生
- **🎲 批次 Faker**：一次勾選多種類型批次加入

### 設備資訊 + 虛擬機啟動

頂部「設備資訊」→ 管理 Android / iOS 虛擬機設定：

- 每組設備注入成 `&{DEVICE_<label>}` dict，Mobile.* 步驟可引用
- **🟢 啟動虛擬機** 按鈕：直接啟 AVD / iOS Simulator（後端未實作時，會把指令複製到剪貼簿）
- 狀態徽章：未啟動 / 啟動中 / 執行中 / 啟動失敗

### DB 資訊（資料庫連線管理）

頂部「DB 資訊」→ 維護多組資料庫連線（**7 種**：MySQL / PostgreSQL / MSSQL / Oracle / MongoDB / Redis / SQLite）：

- 切換 type 自動帶預設 port（3306 / 5432 / 1433 ...）
- 🔌 **測試連線** 按鈕即時驗證
- **SQL 測試區**：直接下 SQL 看結果，不必先寫 `Db.Query` 步驟
- 命名規則：`name` 會成為 `&{DB_<name>}` Robot 變數

### Mock 端點管理

頂部「Mock」→ 管理假的 REST 端點，不等後端準備好就能先測前端：

- 支援 GET / POST / PUT / PATCH / DELETE / HEAD / OPTIONS
- **Faker 佔位符**：Body / Headers 內可寫 `{{name}}` / `{{uuid}}` / `{{int:1,100}}`
- <kbd>🛰 試打</kbd> 按鈕：渲染佔位符後顯示完整 Request / Response
- Mock Server 啟停控制

### Markdown 匯入 / 匯出

- **匯出**：案例編輯頁右上 <kbd>匯出 MD</kbd> → 下載 `.md`，可進 git / PR review
- **匯入**：`POST /api/testcases/{node_id}/import-md`（目前 UI 未提供按鈕，需呼叫 API）
- **CLI 獨立執行**：`python run_tests.py -f tests/<your>.md`，完全脫離 web 平台

### REST API（for CI/CD 整合）

啟動後打開 <http://localhost:8000/docs> 查看 Swagger 完整 API。常用端點：

| 端點 | 用途 |
|---|---|
| `POST /api/executions` | 觸發執行 |
| `GET /api/reports` | 查詢報告列表 |
| `GET /api/reports/{id}` | 單筆報告詳情 |
| `WS /ws/executions/{task_id}/logs` | 即時日誌串流 |
| `POST /api/schedules/{id}/trigger-now` | 手動觸發排程 |

## 5. 給 QA 主管的導入建議

### 建議的專案樹結構

```
<產品線>
├─ Feature: 帳號管理
│   ├─ Platform: WEB
│   │   ├─ Page: 登入頁       → 正向 / 反向 / 邊界 → *.md
│   │   └─ Page: 註冊頁
│   ├─ Platform: API          → 使用者 API 端點
│   └─ Platform: APP          → iOS/Android App
├─ Feature: 購物車
└─ Feature: 結帳
```

**原則**：
- **一個 Feature 一個業務功能**（對應 PRD 或 JIRA Epic）
- **Platform 分層** 讓 UI / API / APP 各自獨立，也能放 E2E 案例整合
- **Scenario 命名固定**：正向 / 反向 / 邊界，團隊溝通一致

### 導入 3 階段

**階段 1（第 1-2 週）：建立基礎**
- 建 1 個重要 Feature（例：登入流程）作為 pilot
- 選 5-10 個最常手動跑的測試，用錄製器轉成自動化
- 讓團隊熟悉平台

**階段 2（第 3-6 週）：擴充覆蓋率**
- 每個 Sprint 加 20-30 個案例
- 開始用 DDT 擴大資料覆蓋
- 設定每日夜間回歸排程

**階段 3（第 7 週+）：整合 CI/CD**
- 透過 REST API 把 Smoke Test 接到 CI Pipeline
- PR 合併前自動跑關鍵案例
- 失敗自動發通知（Slack / Email via Webhook）

### 團隊角色分工建議

| 角色 | 在平台做什麼 |
|---|---|
| **PM / BA** | 寫 AC（驗收準則）作為測試起點，用 Markdown 匯出審閱 |
| **QA 工程師** | 補 BDD 步驟、建 DDT 資料、維護錄製、看報告、建排程 |
| **開發** | 跑本機 Agent debug 失敗案例；接 CI/CD |
| **QA 主管** | 管理測試回合、看儀表板趨勢、定導入節奏 |

## 6. 常見問題 FAQ

### Q1. 改了案例看不到最新結果？
請確認畫面上方是否顯示「● 未儲存」。按 <kbd>💾 儲存</kbd> 後才會生效。

### Q2. 為什麼錄製後沒有 trace.zip？
- 使用頁面提供的一鍵指令（內含 `--save-trace`）最穩定
- 自己改寫 codegen 指令沒帶 `--save-trace` 就不會有 trace
- Playwright 版本需 ≥ 1.35；舊版會自動退回無 trace 模式

### Q3. 為什麼錄製結果沒有比對條件 / 預期結果？
- Click / Fill / Goto 這類純操作不會自動產生斷言
- 請在 Playwright Inspector 用 `Assert visibility` / `Assert text` / `Assert value` 工具產生 `expect(...)`，平台才會解析成 Condition / Expected

### Q4. 測試還在跑，報告列表卻顯示 FAIL？
新版已多層防護：
1. 前端對「還在等 WebSocket done」的 row 強制顯示 RUNNING
2. 後端明確 RUNNING / PASSED / FAILED 才照實顯示
3. `duration_ms > 0` 才依步驟聚合判斷
4. 無資料時顯示 `—` 而不是 FAIL

若看到 > 2 分鐘沒變的 RUNNING：可能 worker 卡住，可在編輯頁底部終端機按 <kbd>停止</kbd>，或 `docker logs autotest-celery` 觀察。

### Q5. 本機模式按執行卻沒反應？
- 確認頂部環境已切到「本機」
- 確認 `local_agent.py` 已啟動，且 `--server` 指到正確平台網址
- Api / Mobile 類步驟請改回 Docker 模式

### Q6. WebSocket 日誌一直空白？
- DevTools → Network → WS，確認連線已建立
- 路徑是 `/ws/executions/{task_id}/logs`（不是 `/ws/v1/...`）
- 重啟後端：`docker compose restart backend`

### Q7. 改了 Schema / Model 後 API 報錯？
- 後端容器需重建：`docker compose build backend && docker compose up -d backend`
- 新增 DB 欄位需執行 `ALTER TABLE` 或 `docker compose down -v` 重建（會刪資料）

### Q8. 想清空所有資料重新開始？
```bash
docker compose down -v
docker compose up -d --build
```
> ⚠ `-v` 會刪除 mysql / minio volume，所有專案、報告、截圖都會消失。

### Q9. 可以重新命名專案或節點嗎？
可以。專案下拉右側有「重新命名目前專案」；每個節點列右側的 ✏ 圖示可改名。

### Q10. 為什麼 DDT 沒有每列都重跑？
預設前端固定送出 `ddt_expand=false`；需要逐列展開請用 API 直接送 `ddt_expand=true`（見第 3-C 節）。

### Q11. Apple Silicon Mac 很慢怎麼辦？
`ppodgorsek/robot-framework` 只有 amd64 image，在 M 系列 Mac 會走 Rosetta/QEMU 模擬，啟動會慢 2-4 倍。建議：
- 短期使用：忍受速度或減少並行
- 長期使用：換原生 amd64 機器，或自行建 arm64 base image

### Q12. 怎麼讓 PM 看到測試結果？
三種方法擇一：
1. **瀏覽器分享網址**：把報告詳情頁網址貼給 PM
2. **匯出 PDF**：在報告頁列印 → 存 PDF
3. **匯出 Markdown**：案例編輯頁 <kbd>匯出 MD</kbd>，放進 PR / Confluence
