## 1. 啟動與停止平台

需求：**Docker Desktop 24+** 且已啟用。

```powershell
cd C:\Demo\autotest_v1.0_20260420

# 啟動（首次或修改後加 --build）
docker compose up -d --build

# 停止
docker compose down

# 連同資料一起清空
docker compose down -v
```

啟動後請在瀏覽器開啟 <http://localhost/>。

| 服務 | 對外網址 | 用途 |
|---|---|---|
| 前端 SPA | <http://localhost/> | 主要操作介面 |
| Swagger 文件 | <http://localhost:8000/docs> | REST API 互動文件 |
| MinIO 主控台 | <http://localhost:9001/> | 截圖／報告檔案瀏覽 |

```

├──────────────┬──────────────────────────────────────────┤
│              │                                          │
│  ProjectTree │            EditorPanel                   │
│   (左側目錄樹)│   ┌─ 驗收準則 (AC)                        │

```

- **TopNav**：切換專案、切換頁面（編輯器 / 報告儀表板）。
- **ProjectTree**：5 層樹狀目錄（Feature → Platform → Page → Scenario → TestCase）。
- **EditorPanel**：點選 TESTCASE 葉節點後出現的右側編輯區。

## 3. 建立專案與目錄樹

### 3.1 建立專案

1. 點 TopNav 的「專案下拉」→ **新增專案**。
2. 輸入專案名稱 → 確定。
3. 系統會自動切換至新專案，目錄樹為空。

### 3.2 新增節點

於樹狀目錄上 **滑鼠右鍵** 任一節點，可選擇：

| 動作 | 說明 |
|---|---|
| 新增子節點 | 在目前節點下建立下一層 |
| 重新命名 | 修改節點名稱 |
| 刪除 | 連同子節點一併刪除（不可復原）|

層級由系統自動管理：每往下一層即往下一個 level_type。
**只有最底層的 TESTCASE 節點**，點選後才會出現右側編輯器。

## 4. 撰寫測試案例

點選任一 TESTCASE 節點後，右側編輯器顯示四個區塊。
任何修改都會把按鈕區的「💾 儲存」變為高亮狀態，**離開節點前請務必按儲存**。

### 4.1 驗收準則 (Acceptance Criteria)

純文字框，記錄使用者驗收測試案例通過所需滿足的條件。

範例：
```
使用者輸入正確帳密後，應導向 /dashboard 並顯示歡迎訊息「Hi, {username}」。
```

### 4.2 ⭐ 前置動作 (Pre-Setup)

> v1.0 新增。位於驗收準則下方。

純文字框，用來記錄 **執行此測試案例前需要手動準備** 的環境/資料事項，例如：

```
- 啟動 mock server (port 4000)
- 在資料庫塞入 seed 使用者: alice / Pa$$w0rd
- 預先取得 OAuth token 並設為環境變數 TOKEN
```

> ⚠️ 目前此欄位 **僅作為說明文字**，不會自動執行。
> 若要自動化前置動作，請改寫到 BDD 步驟表（4.3）。

### 4.3 BDD 步驟表

每一列代表一個動作。欄位說明：

| 欄位 | 必填 | 說明 |
|---|---|---|
| 關鍵字 | ✓ | `Given` / `When` / `Then` / `And`（僅作可讀性） |
| 描述 | ✓ | 此步驟在報告中顯示的文字 |
| Action | ✓ | 系統可識別的動作（見下表） |
| Locator | 視 action 而定 | CSS / XPath / API URL / SQL |
| Input | 視 action 而定 | 輸入文字、JSON body、SQL 片段 |
| Condition | 預設 Equals | 斷言比較方式 |
| Expected | 視 action 而定 | 預期值、HTTP status、列數 |

#### Action 速查（常用）

| 類別 | Action | 說明 |
|---|---|---|
| 瀏覽器 | `Goto` / `Click` / `Fill` / `Select` / `Hover` | UI 操作 |
| 瀏覽器 | `AssertVisible` / `AssertText` / `AssertUrl` | UI 斷言 |
| HTTP  | `Http.Get` / `Http.Post` / `Http.Delete` | API 呼叫 |
| SQL   | `Db.Connect` / `Db.Query` / `Db.RowCount` | 資料庫驗證 |
| Mobile| `Mobile.Open` / `Mobile.Click` / `Mobile.Input` | App 操作 |

> 完整 action 清單請見 [README.md](README.md#測試案例步驟steps_json支援的-action)。

#### 操作步驟列

- **➕ 新增步驟**：在表格底部插入空白列。
- **▲ ▼**：上下移動列順序。
- **🗑️**：刪除該列。

### 4.4 DDT 資料表

Data-Driven Testing：同一份步驟搭配多筆資料各執行一次。

1. 按「➕ 新增欄位」設定欄位名稱（例如 `username`、`password`）。
2. 按「➕ 新增資料列」填入每組測試資料。
3. 在 BDD 步驟表的 Locator / Input / Expected 內，使用 `${欄位名}` 引用，例如：
   ```
   Action:   Fill
   Locator:  #username
   Input:    ${username}
   ```
4. 執行時，runner 會 **逐列展開為多個 Test Case**。

也可以按「📤 匯入 JSON」從檔案載入：
```json
{ "headers": ["username", "password"],
  "rows":    [["alice", "p1"], ["bob", "p2"]] }
```

## 5. 執行測試

1. 在編輯器右上方按「▶ 執行」。
2. 系統會建立一筆 Execution，並把工作丟到 Celery Worker。
3. 編輯器底部會自動展開 **即時日誌抽屜**（透過 WebSocket 串流）。
4. 過程中每個步驟會自動截圖（PRE / POST），存放於 MinIO。

> 想停止執行：按底部抽屜的「⏹ 停止」（會送出 cancel 訊號，但已開始的步驟仍會執行完）。

## 6. 查看報告

切到 TopNav 的「📊 報告」頁。

### 6.1 儀表板 (Dashboard)

- 通過率圓環圖
- 最近 N 次執行的趨勢折線圖
- 各 Feature 通過率長條圖

### 6.2 報告列表 (ReportList)

依時間倒序列出所有執行紀錄，支援以狀態（PASS/FAIL/RUNNING）過濾。

### 6.3 報告詳情 (ReportDetail)

點進任一筆報告：

- 上方：執行統計、總時長、瀏覽器版本
- 中間：步驟時間軸（每行可點開查看 PRE/POST 截圖）
- 下方：完整 Robot Framework log / report.html 連結

## 7. 常見問題

### Q1. 改了「前置動作」之後沒看到？
請確認是否按了「💾 儲存」。離開節點前若未儲存會跳警示，請儲存後再切換。

### Q2. 執行時看不到截圖？
- 確認 MinIO 容器健康：`docker compose ps`
- 確認 `.env` 的 `BASE_URL` 與實際對外網址一致（預設 `http://localhost`）

### Q3. WebSocket 日誌一直空白？
- 檢查瀏覽器 DevTools → Network → WS 連線是否建立
- 重啟 backend：`docker compose restart backend`

### Q4. 改了 Schema/Model 後 API 報錯？
- 後端容器需要重建：`docker compose build backend; docker compose up -d backend`
- 若是新增 DB 欄位，需先對 MySQL 執行 ALTER TABLE 或 `docker compose down -v` 重建

### Q5. 想清空所有資料重新開始？
```powershell
docker compose down -v
docker compose up -d --build
```
> ⚠️ `-v` 會刪除 mysql / minio volume，所有專案、報告、截圖都會消失。
