## 1. 5 分鐘快速上手

### Step 1:啟動平台

**需求**:Docker 24+ / Docker Compose v2.23+(Windows / macOS / Ubuntu / Linux 三平台通用)。

#### 🚀 純 Docker Compose 部署

四個指令完成首次部署,跨平台行為一致。Linux / macOS 用 bash,Windows 用 PowerShell,**指令相同**:

```bash
cd <專案資料夾>

# 1) 自動產 .env(含隨機 secret;若已有 .env 不覆寫)
docker compose --profile init run --rm bootstrap

# 2) 預 build 4 個 spawn-time image(robot runner / 錄製 web / 錄製 api / mcp)
#    這些 image 是 backend 在 runtime 動態 docker run 出來的容器,但必須先存在
docker compose --profile spawnable build

# 3) 啟動主服務(13 個常駐容器)
docker compose up -d --build

# 4) 建立第一位 admin(平台沒有預設 admin/admin123)
docker compose exec backend python -m app.cli create-admin
```

**日常維運子命令**:

| 用途 | 指令 |
|---|---|
| 部署 / 啟動 | `docker compose up -d` |
| 查看容器狀態 | `docker compose ps` |
| 跟隨即時 log | `docker compose logs -f` |
| 停止(保留資料)| `docker compose down` |
| 重置(**清空所有資料**)| `docker compose down -v` |

**Profile 切換**:
- `--profile obs`:啟動 Prometheus + Jaeger 觀察性堆疊
- `-f docker-compose.dev.yml` overlay:暴露內部 port 給 host(DBeaver 等工具直連 postgres / valkey)

> 🍎 **Apple Silicon Mac**:可正常使用,但 runner / recorder 容器為 amd64,啟動會比原生慢 2-4 倍。
> ⚠ 第一次部署後務必執行 `python -m app.cli create-admin` 建立第一位管理員 — 平台沒有任何預設帳號。

### Step 2:建立第一位 admin 並登入

平台**沒有**預設帳號(早期版本曾有 `admin/admin123`,已移除以避免公開部署留後門)。第一次部署完一定要先建管理員:

```bash
docker compose exec backend python -m app.cli create-admin
```

互動模式會提示輸入使用者名稱 + 密碼 + email。或非互動模式:
```bash
AUTOTEST_ADMIN_USERNAME=alice \
AUTOTEST_ADMIN_PASSWORD='at-least-8-chars' \
AUTOTEST_ADMIN_EMAIL=alice@example.com \
docker compose exec -T backend python -m app.cli create-admin --non-interactive
```

進 <http://localhost/> 後就能用剛建的帳號登入。其他選項:

- **註冊**:點上方「註冊」分頁讓其他人自助建帳號(使用者名稱 3-32 字元、密碼 ≥ 6 字元)
- **忘記密碼**:點「忘記密碼」分頁 → 輸入使用者名稱 + 註冊 Email + 新密碼,直接重設
- **暗黑模式**:上方列 🌙 按鈕切換亮 / 暗主題,偏好自動儲存到瀏覽器
- **語系切換**:上方列 `EN` / `中` 按鈕一鍵切繁中 / 英文,介面文字、placeholder、tooltip 全部即時翻譯
- **大頭貼**:登入後到設定 → 使用者 → 帳戶資訊,點頭像旁邊 📷 上傳(≤ 5 MB)

**登入後**:點上方列右側的 Avatar + 名字 → **直接進「設定」頁**;名字右邊是獨立的紅色「↩ 登出」按鈕。

### Step 3:認識首頁與導覽

平台採用 **「上方列 + 左側彈出選單(Nav Drawer)」** 的雙層導覽設計。

#### 上方列(單列、永遠顯示)

```
[≡ 功能選單] [📁 專案樹(行動)] [Logo · RL v1.0]   ─    [環境 Docker] [🔔] [🌙] [EN] | [Avatar 名字] [↩ 登出]
```

| 元件 | 用途 |
|---|---|
| **≡ 功能選單** | 點擊彈出左側 drawer,內含全部功能分頁 |
| **📁 專案樹**(僅行動版顯示)| 行動裝置上開啟左側專案目錄 |
| **Logo / RL** | 點擊隨時回首頁 |
| **環境(Docker / 本機)** | 一鍵切換執行環境 |
| **🔔 通知** | 站內通知中心,紅點顯示未讀數;含 toast 訊息歷史 |
| **🌙 / ☀** | 切換亮 / 暗主題 |
| **EN / 中** | 切換英文 / 繁中介面 |
| **Avatar + 名字** | 點擊直接進設定頁 |
| **↩ 登出** | 獨立紅色按鈕,點擊登出 |

#### 左側 Nav Drawer(分群顯示全部功能)

點上方列 ≡ 後從左側滑入,內含 4 個分群:

| 分群 | 功能 |
|---|---|
| **規劃** | 測試專案 / WBS / 測試計畫 / 需求-RTM / 測試案例 / 測試資料集 / 測試文件 / **待辦清單(Backlog)** |
| **設定** | 設備資訊 / 環境變數 / DB 資訊 / Mock |
| **執行 / 監控** | 測試時程 / 測試回合 / 測試排程 / 測試錄製 / 測試看版 / 測試報告 / 缺陷管理 / **審核中心** |
| **審核** | 審核中心(獨立分群快速入口;同個 view 的另一條捷徑)|

**操作**:點背景 / 按 Esc / 點選任一功能後 drawer 自動關閉。

#### 首頁主畫面

- **左側專案目錄**(桌面常駐 / 行動裝置抽屜):5 層樹狀結構,搜尋框 + 全部展開 / 收合
- **Hero**:當前專案 + 大字「自動化測試平台」漸層卡
- **當前專案概況**:專案、測試案例、節點總數、測試回合數
- **系統狀態面板**:Docker / CPU / 記憶體 / 儲存空間 / 網路流量(每 5 秒輪詢)
- **待辦事項 + 行事曆**:依到期日分桶(過期 / 7 天內 / 待辦 / 已完成),首頁 read-only 預覽
- **最近測試案例**:最近 10 筆
- **快速導覽**:20 張功能卡(規劃 7 + 設定 5 + 執行/監控 8),點擊一鍵跳轉

### Step 4:建立第一個專案

1. 左側「專案目錄」標題右邊的橘色 <kbd>📁+</kbd> 按鈕
2. 輸入專案名稱(例:`官網測試`)→ 確定
3. 系統自動切到新專案,自動回首頁顯示新專案的概況

### Step 5:建立目錄與第一個測試案例

按照 5 層結構建立(**由上到下**):

```
功能 (Feature)   :帳號管理
 └─ 平台 (Platform):WEB
     └─ 頁面 (Page) :登入頁
         └─ 情境 (Scenario):正向
             └─ 測試案例 (TestCase):成功登入.md   ← 點這個開始編輯
```

**操作方式**:滑鼠移到節點上,右邊出現 <kbd>➕</kbd> <kbd>✏</kbd> <kbd>🗑</kbd> 按鈕。點 ➕ 逐層建立子節點,到 TESTCASE 層就能點進去編輯。

### Step 6:跑起來

1. 點開 `成功登入.md`
2. 右側編輯器出現四個區塊(驗收準則 / 前置動作 / BDD 步驟 / DDT 資料)
3. 在「測試步驟設計」區新增一列,動作選 `Navigate`,輸入值填 `https://www.google.com`
4. 按右上角 <kbd>💾 儲存</kbd>
5. 按 <kbd>▶ 執行測試</kbd>
6. 底部自動彈出終端機,看到 `🐳 啟動容器` → `🏁 容器結束` 就完成了
7. 切到「測試報告」看結果

🎉 恭喜,你剛跑完人生第一個自動化測試。

## 2. 核心概念速懂

花 3 分鐘看完這段,後面的旅程會順很多。

### 五層目錄:為什麼要這樣分?

| 層級 | 代表 | 例子 | 用途 |
|---|---|---|---|
| **Feature(功能)** | 一個業務功能 | 帳號管理 / 購物車 / 會員中心 | 按需求領域切分 |
| **Platform(平台)** | 測試的技術平台 | WEB / API / APP / DB / E2E / UI | 決定用哪個底層 Library |
| **Page(頁面)** | UI 頁面 / API 資源 / 資料表 | 登入頁 / 商品 API / users 表 | 更細的分類 |
| **Scenario(情境)** | 測試意圖 | 正向 / 反向 / 邊界 | 區分 happy path 與錯誤情境 |
| **TestCase(測試案例)** | 實際跑的測試 `.md` | 成功登入.md / 忘記密碼.md | **只有這一層可以跑** |

💡 **重點**:**Platform 會決定你能用哪些動作**。把案例放在 WEB 底下 → 只能用 WEB 動作;放在 DB 底下 → 只能用 `Db.*` 動作;E2E 平台則全部可用。

### 測試案例的四區塊

每個 TESTCASE 打開後都有四塊:

| 區塊 | 內容 | 給誰看 |
|---|---|---|
| **1. 驗收準則 (AC)** | 業務描述「怎樣算通過」 | PM / BA / QA 對齊需求 |
| **2. 前置動作 (Pre-Setup)** | 執行前需要準備的事(文字說明)| QA 手動準備或寫在步驟裡 |
| **3. BDD 步驟表** | 具體執行什麼、驗證什麼 | 系統自動執行 |
| **4. DDT 資料表** | 同一個測試跑多組資料 | 擴增覆蓋率 |

### 「動作」與「比對條件」

BDD 步驟表的兩大欄位:

- **動作 (Action)**:你要做什麼(`Click` / `Fill` / `Http.POST` / `Db.Query` ...)— 共 **100+ 種**
- **比對條件 (Condition)**:怎麼驗證(`Equals` / `Contains` / `Regex` / `GreaterThan` ...)— 共 **12 種**

#### 進階動作:Capture / If / ElseIf / Else / EndIf ✨

除了純動作,還有 5 種「特殊 step action」讓你寫條件式 / 變數綁定的進階案例:

| 動作 | 用途 | 範例 |
|---|---|---|
| **Capture** | 從畫面 / API 抓變數 | 從 `.token` 抓 text 存進 `${token}` |
| **If** | 條件分支起點 | `If` + 條件式 `{{= ${type} == 'vip' }}` |
| **ElseIf** | 多分支 | `ElseIf` + 另一條件 |
| **Else** | 預設分支 | — |
| **EndIf** | 結束分支(必填) | — |

詳見 [第 5 章 — Capture step / 條件分支](#capture-step--從畫面--api-抓變數-)。

#### 動態運算式 `{{= ... }}` ✨

step 的 `expected` / `input` 欄位可寫運算式:`{{= ${count}+1 }}` / `{{= ${name} | upper }}` / `{{= ${rows} | len > 5 }}`。
詳見 [第 5 章 — 動態運算式](#動態運算式mini-dsl)。

### 資料儲存原則 — 全部都進伺服器,不留本機

| 資料種類 | 落地位置 |
|---|---|
| **測試案例內容**(步驟、AC、DDT)| PostgreSQL |
| **截圖 / 錄影 / Trace / 缺陷附件 / 錄製 trace.zip** | SeaweedFS(S3 相容物件儲存) |
| **環境變數、設備、Mock 端點、DB 連線設定** | PostgreSQL(密碼欄位 Fernet 加密)|
| **Backlog / 需求 / 缺陷 / 計畫 / 文件 / 通知** | PostgreSQL |
| **使用者偏好**(主題、語系、最近專案)| 瀏覽器 localStorage |

> ⚠ 換言之,**只有 UI 偏好留在瀏覽器**;所有業務資料、附件、密碼都在伺服器端,合規敏感資料不會外洩。

### A. 寫出你的第一個 WEB UI 測試(登入案例)

**目標**:驗證官網登入頁用正確帳密可成功登入並跳轉。

**預期時間**:5-10 分鐘。

#### 建立案例

參考快速上手 Step 4,建立到 `帳號管理 → WEB → 登入頁 → 正向 → 成功登入.md`。

#### 填驗收準則

```
使用者輸入正確帳號與密碼後,應導向 /dashboard
並顯示歡迎訊息「Hi, {username}」。
```

#### 填前置動作

```
- 確認 staging 環境可連線
- 已建立測試帳號:demo / Pa$$w0rd
```

#### 寫 BDD 步驟

按 <kbd>+ 新增步驟</kbd> 逐列填入(每列可多行描述):

| # | BDD | 步驟描述 | 動作 | 定位器 | 輸入值 | 比對條件 | 預期結果 |
|---|---|---|---|---|---|---|---|
| 1 | Given | 開啟登入頁 | `Navigate` | | `https://staging.example.com/login` | | |
| 2 | When | 輸入帳號 | `Fill` | `#username` | `demo` | | |
| 3 | When | 輸入密碼 | `Fill` | `#password` | `Pa$$w0rd` | | |
| 4 | When | 點擊登入按鈕 | `Click` | `button[type=submit]` | | | |
| 5 | Then | 驗證 URL 跳轉 | `AssertURL` | | | `Contains` | `/dashboard` |
| 6 | Then | 驗證歡迎訊息 | `AssertText` | `.welcome` | | `Contains` | `Hi, demo` |

#### 儲存與執行

1. 按 <kbd>💾 儲存</kbd>
2. 按 <kbd>▶ 執行測試</kbd>
3. 底部終端機會即時串流每個步驟的 log
4. 跑完切到「測試報告」看結果

> 🎯 **技巧**:不確定定位器怎麼寫?用下一段的「錄製器」就不用手寫。

### B. 用錄製器 10 倍加速(本機 / Docker / 自動建案)

手寫定位器(`#username`、`button[type=submit]`)很費時。**錄製器**讓你用真的瀏覽器操作一次,系統自動產生步驟。

#### B-1. 兩種錄製模式 ✨

| 模式 | 適合 | 怎麼用 |
|---|---|---|
| **本機錄製** | 已裝 Node.js / Playwright,熟悉終端機 | 終端機貼指令 → Playwright Inspector 開本機 chromium |
| **Docker 錄製** | 不想裝任何東西、想直接在瀏覽器頁內操作 | 平台啟動 recorder 容器 → noVNC iframe 顯示桌面 → 直接點操作 |

建立錄製階段時 modal 上方有 **「執行模式」切換鈕**:`💻 本機` / `🐳 Docker`,選好再按建立。

#### B-2. WEB 錄製(本機模式)

1. 先在左側樹**點選一個 TESTCASE**(錄製結果會合併到這個案例),或 ✨ 先不選,等錄完再「主動建為新案例」
2. 頂部點「錄製」
3. 切到「💻 本機」模式 → 輸入目標 URL → <kbd>建立錄製階段</kbd>
4. 畫面顯示兩段一鍵指令(**依你的作業系統選一個複製**):
   - **Windows**:PowerShell 一鍵指令
   - **macOS / Linux**:bash 一鍵指令
5. 貼到終端機 Enter → Playwright Inspector 自動打開
6. **在瀏覽器中操作**:點、輸入、送出...
7. 想自動產生斷言?點 Inspector 工具列的 <kbd>Assert visibility</kbd> / <kbd>Assert text</kbd> / <kbd>Assert value</kbd>,再點頁面元素
8. 關閉瀏覽器 → 自動上傳 trace.zip 到 SeaweedFS

#### B-3. WEB 錄製(Docker 模式) ✨

1. 切到「🐳 Docker」模式 → 輸入目標 URL → 可選擇 **「📦 自動建案模式」+ 選 SCENARIO**
2. <kbd>建立錄製階段</kbd> → 平台 spawn `recorder-web` 容器
3. 頁面下方出現 **noVNC iframe**(直接是容器內桌面),你看到的就是被錄的瀏覽器
4. 在 iframe 內操作 → 操作會被 codegen 即時錄成 script
5. 關閉容器內的 Playwright Inspector → trace.zip 自動回傳 + 容器自動回收(`auto_remove=True`)
6. 若有開「自動建案」→ 平台自動建新節點 + 自動跳轉到新案例編輯頁,**0 點擊建出案例**

#### B-4. 套用錄製結果到案例(三條路徑) ✨

錄完後底下「Preview」section 顯示已抽取的 steps,有 3 種建案路徑:

| 按鈕 | 行為 | 適合 |
|---|---|---|
| <kbd>套用至當前案例</kbd> | 合併到「進入錄製前選的 TESTCASE」 | 補強既有案例 |
| <kbd>➕ 直接建為新案例</kbd> ✨ | 跳出 mini Modal 選 SCENARIO + 取名 → 新建一個 TESTCASE | 想保留原案、開新分支 |
| <kbd>🤖 AI 增強</kbd> ✨ | 把 trace + 截圖丟 vision LLM,推斷意圖、補斷言、加變數、加分支 | 詳見 [旅程 H](#h-用-ai-增強錄製vision-llm) |

> 💡 **錄製結束沒看到 trace.zip?** 確認指令含 `--save-trace`(平台一鍵指令已內建);Playwright 版本要 ≥ 1.35。

#### B-5. API 錄製(貼 cURL)

1. 在瀏覽器 DevTools Network 面板,對 API 請求按右鍵 → Copy → Copy as cURL

### F. 用 AI 助理 ⚡ 一鍵生案例

**情境**:接到新需求「我們要做購物車從加入到結帳的測試」,你不想從零打字 — 給 AI 描述一遍,直接生 30 個案例先鋪量。

1. 設定 → AI Token → <kbd>新增 Token</kbd>
2. 公司選 OpenAI / Anthropic / Google / DeepSeek 任一(或本地 Ollama)
3. 貼 API Key → <kbd>用 token 拉模型清單</kbd> → 選一個模型
4. 勾「設為預設」→ 儲存

#### 步驟

1. 點右下角浮動圓點(漸層紫色)→ AI 對話框滑入
2. 輸入需求,例如:
   > 「我要測購物車流程。從首頁點商品 → 加入購物車 → 改數量 → 進入結帳頁 → 填地址 → 送出。請涵蓋正向、未登入跳轉、庫存不足三種情境」
3. 等 AI 回應 → **訊息泡泡左下會多一顆 ⚡ 按鈕**
4. 點 ⚡ → 跳出選擇:
   - **套用到當前案例**:現在編輯器開的這個 TESTCASE 直接覆蓋 / 合併 steps

- **錯誤訊息友善化**:401 會講「key 過期請重設」、429 講「速率限制」、找不到模型講「請按重抓清單」

### G. 讓 AI 透過 MCP 自己操作瀏覽器產生案例

**情境**:接到新客戶但沒寫過半個案例,讓 AI 直接打開客戶網站「自己」走一遍,你只在旁邊看 + 補斷言。

### H. 用 AI 增強錄製(Vision LLM)

**情境**:錄完一段流程,但只有 Click/Fill 沒有斷言;你想讓 AI 看著 trace 截圖幫你補多條件斷言、補 Capture、加 if 分支。

- 已完成一次錄製(本機 / Docker 任一模式),trace.zip 上傳成功
- AI Token 設好,**選一個支援 Vision 的模型**:
  - OpenAI:`gpt-4o` / `gpt-4o-mini`
  - Anthropic:`claude-3-5-sonnet` / `claude-3-opus`
  - Google:`gemini-1.5-pro` / `gemini-1.5-flash`

#### 步驟

1. 錄製頁 Preview section 點 <kbd>🤖 AI 增強</kbd>
2. 進度 modal 顯示:「抽取 trace 截圖 → 餵給 LLM → 推斷意圖 → 改寫 steps」(10-30 秒)
3. 完成後跳 **diff view**:
   - 左欄:原始錄製 steps(灰底)
   - 右欄:AI 增強版 steps(彩色高亮新增 / 修改的部分)
4. 你可以**逐 step 接受 / 拒絕**:
   - ✅ 接受所有:右欄全套用
   - 🎯 部分接受:勾選想要的 step → <kbd>套用所選</kbd>
   - ❌ 全部拒絕:關 modal,原稿不動
5. 套用後仍走「套用至當前案例」/「建為新案例」其中一條

#### 平台會做哪些增強

| 原始 step | AI 可能增強為 |
|---|---|
| `Click('登入')` | `Click('登入')` + `WaitForSelector('text=歡迎')` + `AssertText('歡迎', '張三')` |
| `Fill('#token', 'abc')` | `Capture(.token-display, text, var=token)` + `Fill('#input', '${token}')` |
| 順序執行 | `If($status == 'success') { steps... } ElseIf { ... }` 分支 |

#### 注意

- 接受了哪些建議全進 audit log:設定 → 審計 → 篩選 action=`ai_enhance.accept`
- 若原稿已經很完整,AI 可能不會改太多 — 這是預期行為
- **Vision 增強很吃 token**,建議用 `gpt-4o-mini` / `gemini-1.5-flash` 等便宜款先試

## 4. ALM 工作流

平台不只跑測試 — 還整合了 Backlog、需求、缺陷、追溯鏈,讓 PM 寫的 AC 跟 QA 跑的案例真的串起來。

### 4-1. Backlog 階層 — 精簡版 Agile

```
🎯 Feature (產品功能 / 主題)
    │
    ├─ ⚙ Task   (開發 / 測試工作)
    ├─ 🐛 Bug    (缺陷修復)
    └─ 🔬 Spike  (技術研究 / 探勘)
```

**規則**:
- `Feature` 一定是頂層
- `Task / Bug / Spike` 可選 Feature 為父,也可獨立存在
- 用 `sprint_label` 把任務分到 Sprint(自由文字,例 `Sprint 23` / `2026-W18`)
- **沒有 Story / AC** — 這兩者改去「需求 / RTM」分頁,平台統一用 `parent_id` 結構表達 Story → AC,RTM 追溯鏈直接讀那邊,**不再雙頭管理**

#### 建立任務

1. 左側 nav drawer「規劃」→「待辦清單」
2. 右上「+ 新增」按鈕,從下拉選類型(Feature / Task / Bug / Spike)
3. 填標題、優先、到期、指派、Sprint label
4. 若選 Task / Bug / Spike,父項可選一個既有 Feature(也可留空)
5. 儲存

#### 在 Backlog modal 內看 / 加 / 移連結 ✨ 新功能

打開既有 Task 後,modal 下方會出現「🔗 連結」面板:

- 可連 9 種實體:**需求 / 案例 / 缺陷 / 測試計畫 / 測試回合 / 里程碑 / WBS / 測試文件 / 專案**
- 每筆顯示 icon + 類型 badge + 目標 code + 標題
- 右側 ✕ 一鍵移除

> 🎯 **使用情境**:在「實作登入功能」這個 Task 上,連結對應的 User Story(需求)、相關測試案例、修復過程冒出來的 Bug — 一個 Task 就把整個工作脈絡串起來。

### 4-2. 需求 / RTM(User Story → AC)

「規劃」分群 → 「需求 / RTM」分頁。三個子分頁:

#### 📋 需求清單

- 表格顯示所有需求,**頂層需求 = User Story**,**子需求(parent_id 非空)= AC**
- 欄位:代號 / 標題 / 來源 / 優先 / 狀態 / 關聯案例數 / **連結 Backlog** / 操作
- **連結 Backlog** ✨ 新欄位:點數字開 popover 看哪些 Backlog 任務連到這個需求
- 編輯需求 modal 內可勾選關聯的測試案例(多選)、選父需求(空 = User Story、選 = AC)

#### 📊 RTM 矩陣

- 行 = 需求,列 = 測試案例,格子顏色 = 該案例最新執行狀態(綠/紅/灰)
- 顯示「需求覆蓋率 %」(已關聯案例的需求數 / 總需求數)

#### 🔗 追溯鏈 (Story → AC → 案例 → 缺陷)

- 樹狀展開:每個 User Story 下的 AC、AC 下的測試案例、案例下發現的缺陷
- **每一層都會渲染 linked Backlog chip** ✨ 新功能 — 讓你一眼看到「這個 AC 還有什麼任務在做」「這個缺陷有人在追」
- KPI: User Stories 數 / ACs 數 / 鏈內案例數 / 鏈內缺陷數 / 孤兒缺陷數
- 「尚未關聯測試案例的缺陷」面板:點下方任一缺陷即可編輯,設定「**關聯測試案例**」後就會出現在追溯鏈中

### 4-3. 缺陷管理

「執行 / 監控」→ 「缺陷管理」。

- 8 種狀態工作流:`新建立 → 已指派 → 處理中 → 已修復 → 已驗證 → 已關閉`(+ `重新開啟` / `不予修復`)
- 嚴重性 4 級 + 優先 P0–P3
- **附件**(圖、PDF、ZIP 等),儲存於 SeaweedFS,缺陷編輯頁拖拉上傳即可
- **「關聯測試案例」下拉** ✨ 新增 — 設了之後該缺陷會出現在 RTM 追溯鏈中該案例下方
- 「測試看版」分頁是缺陷專屬 Kanban,可拖拉變更狀態,卡片上顯示 🔗 N 徽章標示「有 N 個 Backlog 連到這個缺陷」

### 4-4. 端對端流程示範

從一個業務需求到測完關閉:

```
1. PM 在「需求 / RTM」建一個 User Story:「使用者可用 OAuth 登入」
     │
     ├─ 拆兩個 AC(子需求):
     │     • Google OAuth flow 通過
     │     • 失敗時跳轉錯誤頁
     │
2. QA 在「測試案例」建對應的 *.md,並在「需求清單」修改 modal 把案例勾選進該 AC
     │
3. QA 在「待辦清單」建一個 Task「實作 Google OAuth 自動化」
     │
     ├─ 開 Task modal → 「🔗 連結」加上:
     │     • 連到 User Story (需求)
     │     • 連到剛建的測試案例
     │
4. CI 跑案例失敗 → QA 在「缺陷管理」開 Bug,設「關聯測試案例」
     │
     ├─ Bug 自動進入 RTM 追溯鏈該案例下方
     ├─ 在 Backlog Task modal 加「🔗 連結 → defect」
     │
5. 開「需求 / RTM → 追溯鏈」→ 完整看到:
     User Story
        └─ AC × 2
              └─ TestCase
                    └─ Bug (linked back to Task in Backlog)
              └─ linked_todos chip:Task「實作 Google OAuth 自動化」
```

## 5. 進階功能

走完上面四段後,這些是你可能會需要的補充能力。

### 使用者帳號 / 登入管理

**預設帳號**:`admin` / `admin123`(首次啟動由後端 seeder 建立到 PostgreSQL,bcrypt hash)。

- **註冊新使用者**:登入頁點「註冊」分頁 → 填入使用者名稱(3-32 字元英數底線)、Email、密碼(≥ 6 字元、兩次確認)→ 建立帳號後自動切回登入頁
- **忘記密碼**:輸入使用者名稱 + 註冊 Email + 新密碼 → 驗證相符即時重設
- **設定**(點上方列右側 Avatar + 名字 → 直接開設定頁,共 6 個分頁):
  - **使用者**:含「帳戶資訊」、「變更密碼」、「個人偏好」、「登入資訊」
    - **帳戶資訊** ✨:**大頭貼上傳**(點頭像旁的 📷 → 選圖片,SeaweedFS 落地;支援 PNG/JPG/WebP/GIF,≤ 5 MB);可改 Email、顯示名稱;**角色下拉**(從 `/api/settings/roles` 載入,Admin / QA / Viewer 等)
    - **變更密碼**:舊密碼驗證 + 新密碼兩次確認
    - **個人偏好**:預設執行環境(Docker / 本機)、自動刷新、切換未儲存提示
    - **登入資訊**:最近登入時間、登入次數
  - **權限**:全平台 30+ 種權限唯讀清單(供分配給角色用)
  - **角色**:CRUD 角色 + 勾選權限矩陣
  - **通知**:per-event 選擇站內通知 / Email channel
  - **電子郵件 (SMTP)**:寄通知用的 SMTP 設定(host / port / TLS / 帳密)
  - **AI Token**:管理多家 AI provider(見下節)

### 動態運算式(Mini DSL)✨

**情境**:案例的 expected 欄位需要根據變數運算,例如「點按鈕後 count 應該 +1」、「token 取大寫後比對」。

平台支援 `{{= 表達式 | filter }}` 語法(寫在 step 的 `expected` 或 `input` 欄位):

| 寫法 | 解釋 | 結果範例 |
|---|---|---|
| `{{= ${count}+1 }}` | 算術 | `count=3` → `4` |
| `{{= ${name} \| upper }}` | 套用 upper filter | `name='abc'` → `ABC` |
| `{{= ${rows} \| len }}` | 取長度 | `rows=[1,2,3]` → `3` |
| `{{= ${rows} \| len > 5 }}` | 比較式(回 bool) | `rows=[1,2]` → `False` |
| `{{= ${api_resp.id} }}` | JSON path 取 capture 變數值 | 取 `api_resp` 的 `id` 欄位 |
| `{{= ${name} \| strip \| lower }}` | 鏈式 filter | `name=' AB '` → `ab` |

#### 內建 filter

`upper` / `lower` / `len` / `strip` / `int` / `float` / `str`

#### 安全保證

- **不用 `eval()`** — 改用 Python `ast` 模組白名單解析
- **禁止函式呼叫**(防 `os.system(...)` 注入)
- **禁止屬性存取**(防 `().__class__.__bases__...` 沙盒逃逸)
- 純運算式 + filter pipeline 一次完成

#### 向後相容

純字串(沒有 `{{= }}`)照舊處理,**舊案例零異動**。

### Capture step — 從畫面 / API 抓變數 ✨

**情境**:登入後拿到 token,要帶到下一個 API request 的 header。

#### Step action:`Capture`

| 欄位 | 範例值 | 用途 |
|---|---|---|
| 動作 | `Capture` | — |
| 定位器 | `.token-display` | 從哪抓 |
| `capture_target` | `text` / `attr:href` / `value` / `json:user.id` | 抓什麼 |
| `var_name` | `token` | 取名,後續 `${token}` 引用 |

#### 範例

```
| BDD | 動作    | 定位器           | 輸入值 / capture_target | var_name | expected |
|-----|---------|-----------------|------------------------|----------|----------|
| When | Click   | text=登入       |                        |          |          |
| Then | Capture | .session-token  | text                   | token    |          |
| And  | Http.GET| /api/me         | Authorization=Bearer ${token} | |          |
```

#### 注意

- Capture 的變數預設是 **suite-level**,後續所有 step 都看得到
- 跨案例不會互通(每個 testcase 獨立 suite)
- API 回應抓 JSON path:`capture_target = json:data.items.0.id`

### 條件分支 — If / ElseIf / Else / EndIf ✨

**情境**:登入步驟之後,根據是否為新使用者跳不同頁,要寫條件分支。

#### Flat 語法(平面結構)

不用樹狀編輯器,用 4 個特殊 step action 平鋪即可:

```
| BDD | 動作       | 定位器 / 條件式               | 輸入值 / expected |
|-----|-----------|-----------------------------|-------------------|
| When| Click     | text=登入                    |                   |
| And | If        | {{= ${user_type} == 'new' }} |                   |
| Then| AssertText| h1                          | 歡迎新朋友        |
| And | Click     | text=開始導覽                |                   |
| And | ElseIf    | {{= ${user_type} == 'vip' }} |                   |
| Then| AssertText| h1                          | 歡迎回來,VIP    |
| And | Else      |                              |                   |
| Then| AssertText| h1                          | 歡迎              |
| And | EndIf     |                              |                   |
```

#### 平台會做什麼

- 自動轉成 Robot Framework 5.0+ 的 `IF / ELSE IF / ELSE / END` 語法
- 支援巢狀(If 內可以有 If)
- 條件式可用任何 mini DSL 表達式

#### 注意

- 必須**配對**:有 `If` 就必須有 `EndIf`,否則錯誤
- `ElseIf` / `Else` 可省略
- 巢狀深度建議不超過 3 層,可讀性會下降

## 6. 給 QA 主管的導入建議

### 建議的專案樹結構

```
<產品線>
├─ Feature: 帳號管理
│   ├─ Platform: WEB
│   │   ├─ Page: 登入頁       → 正向 / 反向 / 邊界 → *.md
│   │   └─ Page: 註冊頁
│   ├─ Platform: API          → 使用者 API 端點
│   └─ Platform: APP          → iOS/Android App
├─ Feature: 購物車
└─ Feature: 結帳
```

**原則**:
- **一個 Feature 一個業務功能**(對應 PRD 或 JIRA Epic)
- **Platform 分層** 讓 UI / API / APP 各自獨立,也能放 E2E 案例整合
- **Scenario 命名固定**:正向 / 反向 / 邊界,團隊溝通一致

### 導入 3 階段

**階段 1(第 1-2 週):建立基礎**
- 建 1 個重要 Feature(例:登入流程)作為 pilot
- 選 5-10 個最常手動跑的測試,用錄製器轉成自動化
- 同步把對應的 User Story / AC 在「需求 / RTM」建好,跟測試案例綁定
- 讓團隊熟悉平台、跑出第一張追溯鏈

**階段 2(第 3-6 週):擴充覆蓋率 + 把 Backlog 串起來**
- 每個 Sprint 加 20-30 個案例
- 開始用 DDT 擴大資料覆蓋
- 設定每日夜間回歸排程
- **把 Sprint 任務搬進「待辦清單」,每個 Task 用「🔗 連結」綁對應的需求 / 案例 / 缺陷**

**階段 3(第 7 週+):整合 CI/CD + RTM 報告儀表化**
- 透過 REST API 把 Smoke Test 接到 CI Pipeline
- PR 合併前自動跑關鍵案例
- 失敗自動發通知(Slack / Email via Webhook)
- 月會用「需求 / RTM 追溯鏈」直接看「這個版本的需求覆蓋率 / 缺陷追蹤狀況」

| 角色 | 在平台做什麼 |
|---|---|
| **PM / BA** | 「需求 / RTM」寫 User Story + AC、用 Markdown 匯出審閱 |
| **QA 工程師** | 補 BDD 步驟、建 DDT 資料、維護錄製、看報告、建排程、把缺陷的「關聯測試案例」設好 |
| **開發** | 跑本機 Agent debug 失敗案例;Backlog 任務領 Task,完成後改狀態 |
| **QA 主管** | 管理測試回合、看儀表板趨勢、定導入節奏、看追溯鏈確認需求覆蓋率 |

## 7. 常見問題 FAQ

### Q1. 改了案例看不到最新結果?
請確認畫面上方是否顯示「● 未儲存」。按 <kbd>💾 儲存</kbd> 後才會生效。

### Q2. 為什麼錄製後沒有 trace.zip?
- 使用頁面提供的一鍵指令(內含 `--save-trace`)最穩定
- 自己改寫 codegen 指令沒帶 `--save-trace` 就不會有 trace
- Playwright 版本需 ≥ 1.35;舊版會自動退回無 trace 模式

### Q3. 為什麼錄製結果沒有比對條件 / 預期結果?
- Click / Fill / Goto 這類純操作不會自動產生斷言
- 請在 Playwright Inspector 用 `Assert visibility` / `Assert text` / `Assert value` 工具產生 `expect(...)`,平台才會解析成 Condition / Expected

### Q4. 測試還在跑,報告列表卻顯示 FAIL?
新版已多層防護:
1. 前端對「還在等 WebSocket done」的 row 強制顯示 RUNNING
2. 後端明確 RUNNING / PASSED / FAILED 才照實顯示
3. `duration_ms > 0` 才依步驟聚合判斷
4. 無資料時顯示 `—` 而不是 FAIL

若看到 > 2 分鐘沒變的 RUNNING:可能 worker 卡住,可在編輯頁底部終端機按 <kbd>停止</kbd>,或 `docker logs autotest-celery` 觀察。

### Q5. 本機模式按執行卻沒反應?
- 確認頂部環境已切到「本機」
- 確認 `local_agent.py` 已啟動,且 `--server` 指到正確平台網址
- Api / Mobile 類步驟請改回 Docker 模式

### Q6. WebSocket 日誌一直空白?
- DevTools → Network → WS,確認連線已建立
- 路徑是 `/ws/executions/{task_id}/logs`
- 重啟後端:`docker compose restart backend`

### Q7. 改了 Schema / Model 後 API 報錯?
- 後端容器需重建:`docker compose up -d --build backend`
- 新增 DB 欄位的 migration 寫在 `database.py::init_db()`,啟動時 idempotent 執行;若改了 enum 而舊資料有舊值,需手動 UPDATE 清掉

### Q8. 想清空所有資料重新開始?
```bash
docker compose down -v
docker compose up -d --build
```
> ⚠ `-v` 會刪除 postgres / seaweedfs volume,所有專案、報告、截圖、Backlog 都會消失。

### Q9. 可以重新命名專案或節點嗎?
可以。專案下拉右側有「重新命名目前專案」;每個節點列右側的 ✏ 圖示可改名。

### Q10. 為什麼 DDT 沒有每列都重跑?
預設前端固定送出 `ddt_expand=false`;需要逐列展開請用 API 直接送 `ddt_expand=true`(見第 3-C 節)。

### Q11. Apple Silicon Mac 很慢怎麼辦?
`ppodgorsek/robot-framework` 只有 amd64 image,在 M 系列 Mac 會走 Rosetta/QEMU 模擬,啟動會慢 2-4 倍。建議:
- 短期使用:忍受速度或減少並行
- 長期使用:換原生 amd64 機器,或自行建 arm64 base image

### Q12. 怎麼讓 PM 看到測試結果?
三種方法擇一:
1. **瀏覽器分享網址**:把報告詳情頁網址貼給 PM
2. **匯出 PDF**:在報告頁列印 → 存 PDF
3. **匯出 Markdown**:案例編輯頁 <kbd>匯出 MD</kbd>,放進 PR / Confluence
4. **追溯鏈**:給 PM 看「需求 / RTM → 追溯鏈」分頁,直接看到他寫的 User Story 跑出什麼結果

### Q13. 缺陷怎麼進到 RTM 追溯鏈?
要在缺陷編輯 modal 內把「**關聯測試案例**」下拉設好(在「實際結果」跟「附件」中間)。設了之後該缺陷會自動出現在 RTM 追溯鏈中該案例下方;沒設的話會在「尚未關聯測試案例的缺陷」面板裡。

### Q14. Backlog Task 的「🔗 連結」可以連什麼?
9 種:**需求 / 案例 / 缺陷 / 測試計畫 / 測試回合 / 里程碑 / WBS / 測試文件 / 專案**。
連結後會在 RTM 追溯鏈、需求清單、測試看版的對應位置出現徽章 / chip。

### Q15. 我有舊資料是 Epic / Story 類型,升級後會壞掉嗎?
不會。`init_db()` 啟動時自動 migrate:
- `Epic` → `Feature`(階層位置不變)
- `Story` → `Feature`(parent_id 設 NULL,升頂);其下的 Task / Bug / Spike 子節點仍指向原列(現在是 Feature),父子鏈條保持合法
- 真的要保留 Story / AC 的追溯關係,請到「需求 / RTM」分頁重建為 Requirement(parent_id 為空 = User Story、設了父需求 = AC)

### Q16. AI 生的案例 / MCP 操作出來的案例,能不能跑?

可以。
- AI Chat ⚡ 生案例走的是 `steps_json` schema 化路徑,後端有 `GeneratedStep` model 驗證 → 落地後等同手動建的案例
- MCP 跑出來的步驟也會被轉成同一份 schema → <kbd>▶ 執行</kbd> 直接送 Robot Framework
- **但 LLM 推斷可能不準**:建議先建為新案例跑一次,通過後再決定要不要保留 / 合併到主案

### Q17. AI 對話 / MCP 容器 用哪種模型成本最低?

| 用途 | 推薦模型 | 為什麼 |
|---|---|---|
| AI Chat 一般對話 | `gpt-4o-mini` / `claude-3-5-haiku` / `gemini-1.5-flash` | 便宜、速度快 |
| MCP 操作瀏覽器 | `gpt-4o` / `claude-3-5-sonnet` | tool calling + vision 都需要強 |
| Vision 增強錄製 | `gpt-4o-mini` / `gemini-1.5-flash` | 截圖看圖能力夠 + 省錢 |
| 完全本地化 / 內網 | Ollama:`llama3.1:70b` 或 LM Studio | 0 成本但需 GPU 機器 |

### Q18. 動態運算式 `{{= ... }}` 會不會被注入攻擊?

不會。平台內部用 Python `ast` 模組解析,**白名單** 只允許算術 / 比較 / 字串 / list / dict literal,
**禁止** 函式呼叫(`Call`)、屬性存取(`Attribute`)、import — 不會被`os.system(...)` / `().__class__...` 等沙盒逃逸技巧打破。

### Q19. MCP 容器忘記關會不會吃資源?

不會。後端有 idle sweeper 背景任務,**閒置 30 分鐘**自動 stop + remove 容器(每個使用者獨立判斷)。
要強制停:設定 → AI → MCP 區塊 → <kbd>⏹ 停止我的容器</kbd>。

### Q20. 我同事用 `@clienta.com` Email 註冊,結果跑到我的組織?

檢查設定 → 組織 → 確認你的組織「Email 網域」欄位是不是設了 `clienta.com`。
如果是 → 同事自動歸屬到你的組織。要分開請改用**邀請碼**(每個組織一組,精細控管)。

### Q21. If/ElseIf/Else 寫了但跑出來只有第一段執行?

最常見原因:
1. **沒寫 `EndIf`** → 平台會把後面所有 step 都當成 If 的 then 區塊,不會跳出
2. **條件式語法錯**:`{{= ${x} == 'a' }}` 必須有 `{{= }}` 包住,否則被當純字串(永遠 false)
3. **變數未 capture**:條件用 `${user_type}` 但前面沒 Capture 過 → 用 undefined 變數比對

可以開報告詳情看 Robot 翻譯出來的 `.robot` 檔(下載 trace.zip 內附)確認 IF/END 配對。
