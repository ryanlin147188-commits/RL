## 1. 5 分鐘快速上手

### Step 1:啟動平台

需要 Docker 24+ 與 Docker Compose v2.23+,Linux / macOS / Windows(Docker Desktop)都可以。

```bash
git clone https://github.com/ryanlin147188-commits/RL-for-Kapito.git && cd RL-for-Kapito

# 1) 自動產 .env(含隨機 secret;若已有 .env 不覆寫)
docker compose --profile init run --rm bootstrap

# 2) 預 build spawn-time image(Robot runner / 錄製 web / 錄製 api / mcp)
#    這些 image 是 backend 在 runtime 動態 `docker run` 起來的,但必須先存在
docker compose --profile spawnable build

# 3) 啟動主服務
docker compose up -d --build

# (可選) 4) 改 seed admin 的初始密碼;沒設就用內建的 admin/admin123
# echo "AUTOTEST_DEFAULT_ADMIN_PASSWORD=Op3rator-Init" >> .env
```

第一次跑大概 5–10 分鐘 build image。完成後打開 <http://localhost>。

### Step 2:登入

預設兩條路徑並存:

| URL | 帳號 | 密碼 |
|---|---|---|
| `http://<host>/` 帳密登入 | `admin` | `admin123`(第一次登入會強制改密碼) |
| 「使用 Zoho 登入」按鈕 | (你的 Zoho 帳號) | — |

Zoho SSO 預設沒接 — 要啟用見 [README.zh-TW.md](README.zh-TW.md) 的 v1.1.5 章節。

### Step 3:認識首頁與導覽

進來後最上面是橫向 nav bar,Logo 點下去回首頁;右上角是當前使用者、語言切換、暗黑模式。

**左側 Nav Drawer**(點左上角的 ≡):分群列出全部功能。v1.1.9 之後的功能群組大致如下:

- 測試管理:測試案例、測試回合
- 環境設定:環境變數、Mock 端點
- 報告:執行報告
- 設定:使用者設定、組織成員、專案協作成員、群組、角色權限、邀請、Email config、Audit log、Notification 偏好
- 錄製器
- 待辦清單(Backlog)、審核中心

### Step 4:建立第一個專案

側邊欄上方有「專案」下拉,點「+ 新增專案」填名稱即可。所有測試案例都掛在某個專案下;每個專案是一個 RBAC 邊界。

### Step 5:建立目錄與第一個測試案例

進「測試案例」分頁後,左側是專案樹。專案是頂層,底下五層:

```
Project
└── Feature (例:登入)
    └── Platform (Web / API / APP)
        └── Page (登入頁)
            └── Scenario (正常登入)
                └── TestCase ← 真正的測試在這層
```

按右鍵某個節點 → 新增子節點,逐層往下建到 TestCase。

### Step 6:跑起來

打開測試案例 → 右上角「執行」按鈕。
- 預設用 Docker mode:後台會 spawn 一個 `autotest-robot-runner` 容器跑這個案例。
- 想看著瀏覽器跑(有頭模式)→ 切到本機 Agent(見 [§4-9](#4-9-本機-agent有頭模式))。

執行中可以即時看 WebSocket log,跑完點「報告」進去看截圖 / 影片 / trace。

## 2. 核心概念速懂

### 五層目錄:為什麼要這樣分?

從業務角度切:`Feature`(這是哪個業務功能)→ `Platform`(在哪個平台)→
`Page`(哪一頁面 / 端點)→ `Scenario`(這個場景)→ `TestCase`(實際步驟)。

五層讓報告可以按任一層 group by;管理者寫測試計畫也可以按 Feature 或
Platform 整段勾選執行。

### 測試案例的四區塊

每個 `TestCase` 由四塊組成:

1. **Acceptance Criteria**(驗收準則):Markdown 文字,給人看的、給審核者讀的。
2. **Setup**(前置動作):每次跑案例之前要做的事(登入、開瀏覽器、建測試帳號)。
3. **Steps**(BDD 步驟):真正的測試步驟,Given / When / Then / And 開頭,
   每行是一個 step;每個 step 對應一個 Robot Framework 關鍵字呼叫。
4. **DDT**(資料驅動):如果這個 case 要對多組資料重複跑,把資料表填這裡。

步驟支援動態運算式(`{{= ... }}`)、Capture(把畫面 / API 結果存進變數)、
If / ElseIf / Else / EndIf 條件分支。

### 「動作」與「比對條件」

step 表格內每一列至少有:
- **Keyword**(Given / When / Then / And)
- **Description**(自然語言描述)
- **Action**(實際要做的動作 — Navigate / Click / Input / Wait / Capture / If / EndIf / …)
- **Locator**(在哪個元件上操作)
- **Input**(輸入值)
- **Condition + Expected**(比對條件 + 預期值)

「比對條件」是個 dropdown:`Equals`、`Contains`、`Not Equals`、`Greater Than`、`Matches Regex`、…
每個 Action 後面都可以接 Condition + Expected,等於是行內斷言。

### 資料儲存原則 — 全部都進伺服器

所有測試案例、執行報告、截圖、影片、trace、log、設定 **都進 PostgreSQL + SeaweedFS**,
不留本機。即使你關掉瀏覽器、換一台電腦,所有資料都在;不會因為單一機器丟失資料而出事。

## 3. 使用者旅程教學

### A. 寫第一個 WEB UI 測試(登入案例)

#### 建立案例

側邊樹右鍵 `Web (Platform)` 下的 `Login (Page)` → `新增子節點(Scenario)`
→ 名「正常登入」。再右鍵 Scenario → `新增子節點(TestCase)`,叫
「使用 admin 登入成功跳轉首頁」。

#### 填驗收準則

打開案例 → 上方 「驗收準則」 區塊填:

```
GIVEN 使用者在登入頁
WHEN 輸入正確的帳號密碼(admin / admin123)並按登入
THEN 系統跳轉到首頁,Header 顯示登入者名稱
```

#### 填前置動作

「前置動作」區塊填:

```
1. 開啟瀏覽器並導覽到 http://localhost
```

或留空 — Step 1 直接 Navigate 也可以。

#### 寫 BDD 步驟

進「測試步驟」表格,新增列:

| Keyword | Desc | Action | Locator | Input | Condition | Expected |
|---|---|---|---|---|---|---|
| Given | 在登入頁 | Navigate | — | http://localhost | — | — |
| When | 輸入帳號 | Input | `id=username` | admin | — | — |
| When | 輸入密碼 | Input | `id=password` | admin123 | — | — |
| When | 按登入 | Click | `text=登入` | — | — | — |
| Then | 看到首頁 | Wait Element | `id=mainHeader` | — | Contains | "AutoTest" |

#### 儲存與執行

按右上「儲存」→ 右上「執行」。後台用 Docker mode 起一個 Robot runner
容器跑 Robot Framework 7,Browser Library + Playwright。

跑完去報告頁,可以看到:
- 每一步的截圖
- 整個錄影 (`.webm`)
- Playwright trace (`.zip`,可下載拖進 trace.playwright.dev)
- 任何 fail 的詳細 log

### B. 用錄製器 10 倍加速(本機 / Docker)

#### B-1. 三種錄製模式

平台支援 web / api / app 三種錄製:
- **Web**:Playwright codegen(本機或 Docker 內)
- **API**:貼 cURL 直接解析 / mitmproxy Docker mode(進階)
- **App**:貼 Appium 腳本

#### B-2. WEB 錄製(本機模式)

進「錄製器」分頁 → 選「Web」+「本機」→ 按「建立 Session」→
- 平台會回一個 codegen 啟動指令(類似 `npx playwright codegen ...`)
- 本機跑這個指令,Playwright 視窗會打開讓你錄
- 操作完關掉 Playwright Inspector
- 在 SPA 上貼回 Inspector 產的腳本 / HAR
- 平台解析成 BDD 步驟陣列

#### B-3. WEB 錄製(Docker 模式)

選「Docker」→ 平台會起一個 `autotest-recorder` 容器,內含完整的
Playwright + noVNC,你可以直接在 SPA iframe 內看到瀏覽器畫面,
透過 SPA 的工具列控制(Click / Input / Screenshot)。

優點:跨平台、無需在本機裝 Playwright。
缺點:第一次起容器 20–30 秒;有頭模式比本機慢。

#### B-4. 套用錄製結果到案例

解析出 BDD 步驟後,有三條路徑:

1. **直接建為新案例** — 自動跳出 Scenario 選擇 dialog,選好之後一鍵建立新 TestCase + 套上步驟。
2. **套用到既有案例** — 選一個 TestCase,把錄製的步驟覆蓋進去。
3. **手動微調再套** — 在預覽區直接編輯每一行,再套。

#### B-5. API 錄製(貼 cURL)

切「API」分頁 → 選「貼 cURL」→ 把 `curl -X POST ...` 整段貼進去
→ 平台解析出 method、url、headers、body、verify host → 變成
`HTTP Request` action 的步驟。

支援多種 shell 語法(zsh / bash / Windows CMD)、含 `-H`、`--data` / `-d`、
`--cookie`、`-u`、bearer token 等常見參數。

#### B-6. API 錄製(mitmproxy Docker mode)

選「Docker」→ 平台起一個 `autotest-recorder-api` 容器,內含 mitmproxy +
透明 proxy。用 SPA 工具列控制你的瀏覽器 / app 流量導到這個 proxy,
所有 API 流量自動錄成 HAR;結束後解析成 step 陣列。

適合 SPA 行為複雜、要錄完整 API 對話的情境。

#### B-7. APP 錄製(貼 Appium 腳本)

選「APP」→ 貼 Appium Inspector 產的 Python 腳本 → 解析成
`AppiumLibrary` keyword 的 step 陣列。

### C. 一個案例跑 N 組資料(DDT)

某些案例只是「同一個流程,輸入不同資料」(註冊 / 登入 / 查詢)。
DDT(Data-Driven Testing)讓單一案例對 N 筆資料重複跑。

#### 建立 DDT 資料列

案例頁右側「DDT」分頁 → 編輯欄位 + 列。例:

| username | password | expected_msg |
|---|---|---|
| admin | admin123 | 登入成功 |
| admin | wrong | 密碼錯誤 |
| guest | guest123 | 帳號被停用 |

#### 在步驟裡引用

把步驟改成用 `{{=row.column}}` 引用 DDT 欄位:

| Keyword | Action | Locator | Input |
|---|---|---|---|
| When | Input | `id=username` | `{{=row.username}}` |
| When | Input | `id=password` | `{{=row.password}}` |
| Then | Wait Toast | `.toast-body` | (Contains: `{{=row.expected_msg}}`) |

#### 執行時展開

按執行 → 報告會看到該 case 跑了 3 次(對應 3 筆 DDT),每次的截圖 / log
分開記錄;total / pass / fail 分開統計。

### D. 讀懂執行報告

#### 儀表板(最上層)

進「報告」分頁,最上層是 dashboard:總執行數、成功率、最近 7 天的
pass / fail 走勢圖(Chart.js)、最常失敗的 Top 10 案例。

#### 最近執行紀錄

往下是執行列表,每列一筆 execution。可篩選:
- 案例 / Feature
- 觸發人
- 狀態(running / passed / failed / cancelled)
- 時間區間

#### 報告詳情

點任一筆 execution → 進報告詳情:
- 上方:KPI(總步驟、pass、fail、執行時間)
- 中段:每個案例的 timeline,點開看每一步:截圖、log、Robot output
- 底部:整段執行的影片 + Playwright trace 下載

#### 匯出 PDF

報告右上角「匯出 PDF」按鈕。前端用 html2pdf(動態 lazy-load)把當前
report 渲染成 PDF。

## 4. 進階功能

### 4-1. Capture step — 從畫面 / API 抓變數

需要把某個畫面值 / API 回傳值記下來,在後面步驟引用?用 Capture step。

設 step:

| Action | Locator | Input |
|---|---|---|
| Capture Element Text | `.user-greeting` | `welcome_msg` |

→ 把 `.user-greeting` 文字存進變數 `welcome_msg`。

之後步驟可以引用:

| Action | Input |
|---|---|
| Assert Equals | `{{= welcome_msg }}` = `"Hello, admin"` |

也可以從 API 回傳的 JSON 抓:

| Action | Locator (JSONPath) | Input |
|---|---|---|
| Capture API Response | `$.data.user.id` | `user_id` |

### 4-2. 條件分支(If / ElseIf / Else / EndIf)

想根據前一個 Capture 的結果分流?用條件分支 step。

| Action | Condition | Expected |
|---|---|---|
| If | (Captured `user_role`) Equals | `admin` |
| (步驟群組 A) | | |
| ElseIf | (Captured `user_role`) Equals | `viewer` |
| (步驟群組 B) | | |
| Else | | |
| (步驟群組 C) | | |
| EndIf | | |

平台會把這些轉成 Robot Framework 7 的 IF / ELSE / END 區塊 — 真正在
runner 內跑成條件分支,不是「先全跑、再判斷」。

### 4-3. 動態運算式(Mini DSL)

`{{= expression }}` 在執行時即時計算。支援:

- **變數引用**:`{{= row.username }}`、`{{= captured.user_id }}`
- **環境變數**:`{{= env.STAGING_URL }}`
- **算術 / 字串拼接**:`{{= "https://" + env.HOST + "/api/users/" + captured.user_id }}`
- **內建函式**:`now()`、`uuid()`、`md5(s)`、`b64encode(s)`、`fakerName()`、…
- **JSON path**:`{{= json(captured.response, "$.data.items[0].id") }}`

### 4-4. 審核中心

進「審核中心」分頁,看到 pending / approved / rejected 三個 tab。
testcase / script / report 的審核請求(進這個工作流的條件由設定決定)
都集中在這裡。

每筆審核可以:
- 看送來的 diff
- 加 reason
- approve / reject
- 留 audit trail(誰、什麼時候、為什麼)

### 4-5. 待辦清單(Backlog)

`#backlog` 路徑或從 testcase / report detail modal 內的 linked-todos
popover 進入。

支援:
- Feature → Task / Bug / Spike 階層
- Sprint label(可選)
- 過期 / 即將過期 / 完成的 badge
- 完整 CRUD,assigned_to + assigned_to_type(user / group)

⚠ **指派功能已在 v1.1.9 拿掉** — 你可以**設定** `assigned_to` 欄位,
但跨 entity 的「我的工作」inbox 與 bulk reassign UI 已經沒了。

### 4-6. 環境變數 + Faker 隨機資料

每個專案有自己的環境變數表(類似 .env),在步驟裡用 `{{= env.X }}`
引用。在「設定 → 環境變數」分頁編輯。

Faker 整合:`{{= fakerName() }}` / `fakerEmail()` / `fakerPhone()` /
`fakerCompany()` / … 每次執行產不同的隨機資料。

### 4-7. Mock 端點

「Mock 端點」分頁可以註冊 path + 預先回傳的 JSON / 文字 / status code。
測試案例可以打這個 mock URL 來測「後端還沒做好但前端要先測」的情境。

每個 mock 端點 per-org scope;不會跨組織洩漏。

### 4-8. 測試回合

把多筆 execution 群組成一個 Round,例如「Sprint 23 Smoke Round」。
共用一個 dashboard、共用 round-level KPI(pass rate、平均時間、…)。

從測試案例樹節點右鍵「加入 Round」即可。

### 4-9. 本機 Agent(有頭模式)

預設執行是 Docker headless,看不到 browser。要看著瀏覽器跑(debug 用):

1. 從「設定 → 使用者設定 → 本機 Agent」下載 agent 安裝包
2. 本機跑 agent,輸入 backend URL + token
3. 執行時切「本機模式」,測試會在你本機 desktop 啟有頭 Playwright

### 4-10. Markdown 匯入 / 匯出

每個 TestCase 在後端都是一份 Markdown(`steps_json` 用 7-column DSL
序列化)。從專案樹節點右鍵:

- **匯出**:下載整個 sub-tree 為 `.md` 檔 zip
- **匯入**:上傳 `.md`,自動建出對應的樹節點 + 案例內容

跨環境搬遷 / 版控 / Code review 用。

### 4-11. REST API(CI/CD 整合)

```bash
# 從 host 看 OpenAPI JSON(經 gateway 只看 public 端點)
curl http://localhost/openapi.json

# 完整 Swagger UI 從 backend 容器內看:
docker compose exec backend curl localhost:8000/docs

# host 不對外暴露 port 8000;外部統一走 http://localhost/
```

**CI/CD 觸發測試**:

```bash
# 1. 拿 JWT token
TOKEN=$(curl -X POST http://<host>/api/auth/login \
  -d '{"username":"ci-bot","password":"..."}' \
  -H 'Content-Type: application/json' | jq -r .access_token)

# 2. 觸發一個 testcase
curl -X POST http://<host>/api/executions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"node_id":"<testcase-uuid>","trigger_type":"jenkins","execution_mode":"docker"}'

# 回 task_id + report_id,可以 polling /api/executions/{task_id}/status
```

整合範例見 [docs/](docs/) 內的 Jenkins / GitHub Actions / GitLab CI 樣本。

## 5. 設定 / 帳號 / 多租戶

### 使用者登入管理

進「設定 → 使用者設定」可以改個人資料、密碼。

admin / 平台擁有者進「設定 → 組織成員」可以:
- 看 org 內所有使用者
- 改 role
- 停用 / 啟用
- 重設密碼(會發 token via email)
- 邀請新使用者(send / resend / extend / revoke)

### 三層 RBAC

| 層 | 表 | 用途 |
|---|---|---|
| Global | `users.role_id` | 全平台角色(Admin / QA / Viewer) |
| Organization | `org_memberships.role_id` | 在某個 org 內的角色 |
| Project | `project_members.role_id` | 在某個專案內的角色(可被 `project_role_permissions` override) |

外部協作者預設 `role_id=NULL`,看不到任何東西。平台擁有者邀請他們進
個別專案後才看得到。權限解析順序:**ProjectMember > OrgMembership > User**。

### 群組(可巢狀)

「設定 → 群組」可以建群組(team / department / committee),群組可以巢狀。
群組本身可以作為 assignee(待辦清單、testcase assigned_to)。

### 集中式帳號管理

admin 可以幫使用者代開帳號,使用者第一次登入觸發「三欄首登 modal」
(display_name + email + 新密碼)。

### Audit Log

「設定 → Audit log」看所有 mutation 動作的紀錄(誰、何時、改了什麼)。
資料留 90 天,符合 SOC 2 baseline。

## 6. 常見問題 FAQ

### Q1. 改了案例看不到最新結果?
重新整理瀏覽器,SPA 不會 long-polling。或關 / 開那個 case 的編輯頁。

### Q2. 為什麼錄製後沒有 trace.zip?
本機模式 Playwright 的 codegen 預設不啟 trace。要 trace 就用 Docker
mode,或本機跑 codegen 時加 `--save-trace=...`。

### Q3. 為什麼錄製結果沒有比對條件 / 預期結果?
原始 Playwright codegen 只記 action,不記斷言。要在錄製完成後手動編輯
step 表,把 Condition / Expected 欄位補上。

### Q4. 測試還在跑,報告列表卻顯示 FAIL?
通常是 runner container 超時或被 SIGKILL。看 `audit_logs` 或
`docker logs autotest-celery` 找 root cause;最常見是 `RUNNER_CONTAINER_TIMEOUT_SEC`
不夠長,或 `docker-socket-proxy` 連線斷。

### Q5. 本機模式按執行卻沒反應?
本機 Agent 沒啟動 / 沒登入。檢查 agent 的 log,確認連線狀態為 connected。

### Q6. WebSocket 日誌一直空白?
nginx / APISIX / backend WebSocket 路徑沒接通。檢查瀏覽器 DevTools
Network → WS,看 connection 是否 connecting。常見是 reverse-proxy 沒設
`Upgrade: websocket` 標頭。

### Q7. v1.1.9 拿掉的功能(AI / 看板 / RTM / 缺陷 / WBS / …)可以加回來嗎?
可以。每個功能在 git 上都是獨立 commit(`6283deb..e64fc59` 區間)。
單獨 revert 對應的 commit 就能還原該功能。DB schema 全部保留所以
不會丟資料。

### Q8. 想換成 PostgreSQL 之外的 DB?
不支援。平台的 storage backend(S3-compatible 走 SeaweedFS,DB 走 PG)
是設計選擇,不是抽象層,所有 alembic migration 都用 PG-specific 語法。

### Q9. 我可以把 backend 直接 expose 到 Internet 嗎?
**不要。** 預設 `backend:8000` 不對 host expose,所有對外流量必須經過
nginx + APISIX。看 [SECURITY.md](SECURITY.md) 的 production 硬化檢查表。

### Q10. 想自己 fork 改一改、合進 main 怎麼辦?
[CONTRIBUTING](README.md#contributing) 有完整流程。本地跑 `gitleaks` +
`pip-audit` + `bandit`(CI 也跑這三個);看 [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)。
