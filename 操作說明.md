## 1. 帳號管理

### 1.1 自助註冊(v1.1.10 新)

任何人都能在登入頁切換到「**建立帳號**」分頁,填四個欄位:

- 使用者名稱(可中文,顯示用)
- 電子信箱
- 帳號(英數 / `_` / `.` / `-`,3–80 字元)
- 密碼(6 字元以上)

送出後**立即可登入**(不再寄驗證信)。系統會自動:
1. 建立你的個人 Organization,名稱為「`{使用者名稱}` 的工作空間」
2. 把你設為該 org 的 **admin**,擁有完整 73 個權限
3. 把該 org 設為你的「預設組織」

> Email 驗證只保留兩個場景使用:**忘記密碼**、**被邀請加入別人專案**。

### 1.2 內建管理員 / Zoho SSO

- 預設 superuser:`admin` / `admin123`(首次登入會強制改密碼)
- Zoho SSO:管理員設定 `.env` 的 `ZOHO_CLIENT_ID` / `ZOHO_CLIENT_SECRET` 後,登入頁會出現「使用 Zoho 登入」按鈕

### 1.3 忘記密碼

登入頁 → 「忘記密碼」→ 輸入使用者名稱 + Email → 系統寄重設連結(需管理員已設 SMTP)。

### 1.4 個人設定(設定 → 設定)

- **帳戶資訊**:改顯示名稱 / Email / 大頭貼(≤ 5MB)
- **變更密碼**(本機帳號才有;Zoho SSO 帳號隱藏)
- **個人偏好**:預設執行環境、自動重整報告、執行時啟用 Trace + 影片
- **登入資訊** → **刪除帳號**:永久刪除你的帳號 + 個人工作空間,**不可復原**;需手動輸入帳號名稱確認;Superuser 不能自刪
- **退出專案**:列出你所屬的所有專案,點「退出」即離開該專案;若你是唯一成員會被擋住,避免孤兒;全退完後若你有個人 org 且當前不在它身上,會自動切回去

## 3. 專案與測試樹

### 3.1 樹狀結構(五層)

```
Project(專案)
  └── Feature(功能模組)
        └── Page(頁面 / API 群組)
              └── Scenario(情境)
                    └── TestCase(測試案例,葉節點)
```

### 3.2 建立新專案

1. 左側 sidebar 頂端 → 「**+ 新增專案**」
2. 輸入名稱,(可選)勾「複製自現有專案」+ 選來源 → 連同樹結構 / TestCase 內容 / 前置案例都一起複製

### 3.3 操作節點

在任一節點上**右鍵**:

| 操作 | 說明 |
|---|---|
| 新增子節點 | Feature / Page / Scenario / TestCase |
| 重新命名 | 改節點名稱 |
| 刪除 | 連同所有子節點刪除(DB CASCADE) |
| 移動 | Drag-and-drop |

葉節點(TestCase)儲存時自動建立 `entity_version` 快照,預設 `content_status = approved`。

### 3.4 切換專案

左側 sidebar 頂端的下拉選單。**v1.1.10**:你看得到的 project 包含:
- 你自己 org 內的 project
- 你被邀請進去其他 org 的 project(透過 ProjectInvite)

## 4. 撰寫測試案例

### 4.1 編輯器結構

點 TestCase 葉節點,右側打開編輯器:

- **基本資訊**:驗收標準(Markdown)、前置動作描述
- **前置案例(Setup Chain)**:可掛 N 個前置 TestCase,設 `sort_order`、可勾「失敗即中止」、可暫時停用
- **步驟表格(BDD / KDT)**

### 4.2 步驟表格欄位

| 欄位 | 說明 |
|---|---|
| No. | 自動編號 |
| BDD 關鍵字 | Given / When / Then / And / But |
| 說明 | 自然語言描述 |
| 動作 / Action | Robot Framework keyword;依當前案例平台動態下拉 |
| 定位器 / Locator | CSS / XPath / 元素描述 |
| 輸入值 / Input | 支援 `${ENV_KEY}`、`{{= expr }}` |
| 斷言條件 | 等於 / 包含 / 大於 / ... |
| 預期結果 | 斷言的預期值 |

右上角 `ALL / BDD / KDT` 切換顯示模式。

### 4.3 動態運算式(Mini DSL)

「輸入值」或「預期結果」欄位用 `{{= expr }}`:

```
{{= env.BASE_URL }}        取環境變數
{{= row.username }}        取 DDT 資料列欄位
{{= uuid() }}              隨機 UUID
{{= now() }}               時間戳
{{= fakerEmail() }}        隨機 Email
{{= fakerName() }}         隨機姓名
{{= 1 + 2 }}               算術
{{= "x_" + row.id }}       字串拼接
```

### 4.4 Capture 步驟

從畫面元素 / API 回傳值取值存入變數,後續步驟用 `${var_name}` 引用。例:Capture 「登入後的使用者 ID」存進 `${user_id}`,後面 API 步驟用它組 path。

### 4.5 條件分支

`If` / `ElseIf` / `Else` / `EndIf`,由 Robot Framework 7 真實執行(不是前端模擬)。

### 4.6 資料驅動(DDT)

兩種來源:
- **內建資料表**:此 TestCase 專屬,在編輯器右下角「資料」分頁編輯
- **獨立資料集**:從「測試資料集」分頁建立後,引用到多個 TestCase 共用

執行時系統會對每一列跑一次測試,`{{= row.col }}` 解析成該列欄位值。

### 4.7 匯入匯出

| 格式 | 用途 |
|---|---|
| Markdown(`.md`) | 整個子樹雙向轉換,適合版控 / 跨環境搬遷 |
| Excel(`.xlsx`) | 步驟表格批次編輯,Excel 內建下拉選單防呆 |
| CSV | 純步驟 |

## 5. 錄製測試操作

進「錄製」分頁,選模式:

### 5.1 Web 模式(Playwright + noVNC)

1. 點「新增錄製」→ 選「Web」+ 輸入起始 URL
2. 系統 spawn 一個 recorder 容器,內含 Chromium + Playwright codegen
3. 點「打開 noVNC」在瀏覽器內操控真實 Chromium
4. 你的點擊 / 輸入會被即時轉成 Robot Framework + Playwright keyword
5. 點「停止 + 匯入」→ 選目標 TestCase 節點 → 步驟自動寫入

### 5.2 API 模式(mitmproxy)

1. 「新增錄製」→ 選「API」
2. 系統提供 proxy 設定指令(`http://recorder-host:8080`)
3. 在 SPA / Postman / app 設這個 proxy,正常操作
4. mitmweb 介面內可以濾掉雜訊、選你要的請求
5. 「匯入」→ 自動轉成 Robot Framework HTTP keyword

### 5.3 App 模式(Appium)

外部 Appium Inspector 錄完後匯入腳本,系統轉為 Robot Framework 格式。

> Recorder 容器是動態 spawn,結束後自動清理。

## 6. 執行測試

### 6.1 兩種執行模式

| 模式 | 適用場景 |
|---|---|
| **Docker**(預設) | CI 整合 / 規律跑;headless,完全隔離 |
| **本機 Agent** | Debug 用;有頭模式,在你桌面的 Chrome / Edge 上跑 |

右上角「執行環境切換」切換預設。

### 6.2 跑單一 TestCase

編輯器右上角「**執行**」按鈕 → 即時 WebSocket log 浮在右側 →
跑完自動跳 report 頁。

### 6.3 跑 TestRun(批次)

進「TestRun」分頁 → 「新增 TestRun」→ 勾選多個 TestCase → 啟動。
共用一份 dashboard 與 KPI。

### 6.4 排程

「排程」分頁 → 新增 → 選 ONCE / DAILY / WEEKLY / MONTHLY + 時間 + 目標 TestCase / TestRun。系統每 30 秒掃描一次。

### 6.5 執行細節

- 每次跑都 spawn 獨立的 `autotest-robot-runner` 容器,跑完即刪
- 截圖每步驟一張,影片整段 MP4,Playwright trace 一份
- 失敗會把 error message + 截圖直接帶到報告詳情

## 7. 報告與比對

### 7.1 報告詳情頁

- **執行摘要**:總步驟 / 通過 / 失敗 / 跳過 / 耗時
- **步驟列表**:每步驟有 status / log / 截圖 thumbnail
- **影片 + Trace Viewer**:點任一步驟可定位到該秒
- **失敗 step**:旁邊有「**開缺陷**」按鈕,自動帶入該 step 截圖 / error message / testcase 連結

### 7.2 趨勢圖

報告列表頁上方有近 N 次趨勢條(通過 / 失敗),點任一條跳該次報告。

### 7.3 視覺回歸基準

任一有截圖的步驟可「設為基準」。之後同步驟跑出來的截圖會 pixel diff,超過閾值就在報告標紅警告。

### 7.4 檔案比對

進「檔案比對」分頁:
- **Excel / CSV diff**:選兩個檔,純客戶端 diff(不上傳)
- **畫面比對**:選兩張截圖,pixel-level diff,可下載 diff PNG

## 9. 協作:邀請 / 退出 / 跨組織

### 9.1 邀請別人加入專案(管理員)

設定 → 專案協作成員 → 「**邀請使用者**」:
1. 輸入 invitee email
2. 選 role(預設 user)
3. 設定過期天數(預設 7 天)

系統會:
- 寄一封邀請信給該 email(含一鍵兌換連結 + 邀請碼)
- 在「待兌換邀請」section 顯示 pending 邀請

### 9.2 兌換邀請(受邀者)

兩種方式:
- **點信內連結**:`http://your-host/?invite_code=XXXXXXXX` 自動兌換
- **手動兌換**:登入 → 設定 → 兌換邀請碼 → 貼入

**重要**:你登入的帳號 email **必須等於**邀請信寄到的 email(後端嚴格校驗),否則 403。

### 9.3 跨組織協作如何運作(v1.1.10)

受邀者 redeem 成功後:
- 系統建立 `ProjectMember` 記錄(role 由邀請方指定)
- **不**會把你加入邀請方的 Organization(只加 project-level 成員)
- 你的 sidebar 開始能看到該專案,可進去操作
- 權限以該 `ProjectMember.role_id` 為主(覆寫你 org-level 的 role)

### 9.4 退出專案

設定 → 設定 → 「退出專案」section:
- 列出你所屬的**所有 project**
- 點任一專案旁的「退出」→ 輸入「退出」確認 → DELETE `/api/projects/{id}/leave`
- 後端會擋:你是該 project 唯一 active member(避免孤兒,需先「刪除專案」或邀請別人接手)

### 9.5 自動切回個人 org

退出最後一個 project 後,如果:
- 你有個人 org(`personal-{username}`)
- 當前 active org 不是它

→ 系統自動 `POST /api/auth/switch-org` 切回 personal org + 全頁 reload。

## 10. 審核與通知

### 10.1 審核工作流

四種實體可設定審核流程:

| 實體 | 觸發 |
|---|---|
| TestCase | 修改後送審 → admin Approved / Rejected |

- 審核事件(被指派 / 通過 / 拒絕)
- 缺陷事件(被指派 / 狀態變更)
- 邀請事件
- 點通知跳到對應頁面
- 「標記全部已讀」一次清空

### 10.3 Email 通知設定

設定 → 通知設定:
- per-user / per-event 開關
- per-org 設定 SMTP server(Email 通知分頁)
- 沒對應寫權限的 user 看到唯讀 banner(以 org admin 設定為主)

## 11. CI/CD 整合

### 11.1 REST API 觸發

```bash
# 1) 拿 JWT
TOKEN=$(curl -s -X POST http://host/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"ci-user","password":"xxx"}' \
  | jq -r '.access_token')

# 2) 跑一個 TestCase
curl -X POST http://host/api/executions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"testcase_id":"<uuid>","mode":"docker"}'

# 3) 查狀態
curl -H "Authorization: Bearer $TOKEN" http://host/api/executions/<exec_id>
```

### 11.2 API Key(取代 JWT)

設定 → API Key → 產生新 key(`ak_xxxxx...`),在 CI 上用:

```bash
curl -H "X-API-Key: ak_xxxxx..." http://host/api/executions
```

Gateway 會把 API key 換成 5 分鐘短命 JWT,後端流程一致。

### 11.3 GitHub Actions 範例

```yaml
- name: Run AutoTest TestRun
  run: |
    curl -X POST $AUTOTEST_HOST/api/executions \
      -H "X-API-Key: ${{ secrets.AUTOTEST_API_KEY }}" \
      -H "Content-Type: application/json" \
      -d '{"testrun_id":"${{ vars.TESTRUN_ID }}"}'
```

### 11.4 Webhook(進階)

執行完成後可設定 webhook 推送結果到 Slack / 自己的後端。在「設定 → 通知設定」設目標 URL。

## 12. 管理員作業

### 12.1 RBAC 三層權限

| 層 | 範圍 | 例 |
|---|---|---|
| Global | 全系統 | superuser(`*`) |
| Organization | 一個 org 內 | 註冊時自動套用的「admin」role(73 個權限) |
| Project | 一個 project 內 | 邀請時指定的 role,覆寫 org-level |

新增 / 編輯角色:設定 → 角色與權限,以 12 資源 × 4 動作 = 48 個矩陣權限勾選。

### 12.2 群組(Group)

設定 → 群組:可巢狀,把 user 包成群組後一次性加進專案。

### 12.3 Organization 切換

右上角組織下拉:列出你所屬的所有 org;切換時:
- 後端 `POST /api/auth/switch-org` 重新簽 JWT(`org_id` claim 改)
- 設定 `active_org_id` cookie(簽章過,middleware 短路驗證)
- SPA 整頁 reload 套用新權限

### 12.4 備份 / 還原

- 自動每日 03:00:`backup-cron` 從 replica `pg_dump` + SeaweedFS tar,保留 7 天
- 手動觸發:`docker exec autotest-backup-cron sh /backup.sh`
- 還原 SOP:見 [docs/ops/data-safety.md](docs/ops/data-safety.md)

### 12.5 監控

- 主機資訊在首頁系統狀態卡(每 5 秒刷新)
- `/healthz`(public)和 `/readyz`(需 token)可給 K8s liveness probe / 外部監控
- Backend log:`docker logs autotest-backend -f`
- Gateway log:`docker logs autotest-gateway -f`(限速 / 熔斷紀錄)

### 12.6 Casbin RBAC 同步

新增 / 修改 user / role / OrgMembership / ProjectMember 後,系統自動排程 Casbin policy resync(非同步,~1 秒生效)。手動強制重 sync:重啟 backend container。

## 附錄 B:疑難排解

| 症狀 | 解決 |
|---|---|
| 登入 429 Too Many Requests | Gateway 限速(login `100/minute`,register `10/hour`)— 等 1 分鐘 / 改 `gateway/routes.yaml` |
| `/api/auth/me` 401 | JWT 過期 / 帳號被改密碼 — 重登 |
| Zoho SSO 回 callback 401 | 檢查 `.env` 的 `ZOHO_REDIRECT_URL` 跟 Zoho console 是否完全相符 |
| 註冊信沒寄 | 看 admin 是否設了 enabled 的 SMTP config;Celery log `docker logs autotest-celery` |
| 邀請 redeem 403「email 不符」 | 你登入的帳號 email 必須等於邀請信寄到的地址 |
| 唯讀 banner 出現 | 你在當前 active org 不是 admin — 切回個人 org 或請 admin 升級 role |
| 「退出專案」400「唯一成員」 | 你是該 project 唯一 active member — 改用「刪除專案」或邀請別人 |
| 「刪除帳號」400「Superuser 不能自刪」 | 找另一位 superuser 替你刪或先把你降級 |
