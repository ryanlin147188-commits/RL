> 對應版本：v1.0（含「前置動作 (Pre-Setup)」、排程、測試回合、本機 Agent 執行、多來源錄製、專案 / 節點重新命名與 Markdown 匯出）
> 預設網址：<http://localhost/>

## 1. 啟動與停止平台

需求：**Docker Desktop 24+** 且已啟用。

```powershell
cd C:\Demo\autotest_v1.0_20260420

# 第一次或修改 backend/tasks/robot_*.py / Dockerfile.runner 後，要先 build spawn 容器用的 image
docker build -f backend/Dockerfile.runner -t autotest-robot-runner:latest backend/

# 啟動（首次或修改後加 --build；不會 build autotest-robot-runner，那是另一張 image）
docker compose up -d --build

# 停止
docker compose down

# 連同資料一起清空
docker compose down -v
```

> ⚠ `.env` 必須包含 `STORAGE_BACKEND=minio`（spawn 模式只走 MinIO；設成 `local` 會在執行時報錯）。`.env.example` 不存在，請手動建立或複製 README 內的範本。

啟動後請在瀏覽器開啟 <http://localhost/>。

目前 Docker Compose 預設載入的是根目錄 `index.html` 單頁介面；案例編輯、測試回合、執行報告、排程與錄製都是在同一頁內切換，不是不同網址頁面。

補充：

- backend 啟動時會自動建立資料表，並同時啟動每 30 秒一次的排程背景輪詢。
- `docker-compose.yml` 已將 backend / celery 時區固定為 `Asia/Taipei`，排程與報表時間請以此為準。
- 點「執行測試」後，celery worker 會在背景透過 docker socket spawn 一個短命的 `autotest-robot-runner` 容器跑 robot；可在另一視窗以 `docker ps` 觀察到 `robot-<task>-<case>` 容器一閃即逝。

| 服務 | 對外網址 | 用途 |
|---|---|---|
| 前端 SPA | <http://localhost/> | 主要操作介面 |
| Swagger 文件 | <http://localhost:8000/docs> | REST API 互動文件 |
| MinIO 主控台 | <http://localhost:9001/> | 截圖／錄影／Trace 檔案瀏覽（帳密 `minioadmin / minioadmin`）|

## 2. 介面總覽

```
┌──────────────────────────────────────────────────────────────────────┐
│ TopNav  [設備資訊][環境變數][DB資訊][Mock][測試回合][測試報告][環境][排程][錄製] │
├──────────────┬───────────────────────────────────────────────────────┤
│              │                                                       │
│  ProjectTree │            EditorPanel                                │
│   (左側目錄樹)│   ┌─ 驗收準則 (AC)                                     │
│              │   ├─ 前置動作 (Pre-Setup)                              │
│              │   ├─ BDD 步驟表                                        │
│              │   └─ DDT 資料表                                        │
│              │                                                       │
└──────────────┴───────────────────────────────────────────────────────┘
```

- **TopNav**：切換頁面（設備資訊 / 環境變數 / DB 資訊 / Mock / 測試回合 / 測試報告 / 排程 / 錄製），並可切換執行環境（Docker / 本機）。
- **ProjectTree**：5 層樹狀目錄（Feature → Platform → Page → Scenario → TestCase），含搜尋欄與專案工具列。
- **EditorPanel**：點選 TESTCASE 葉節點後出現的右側編輯區。
- **DeviceInfoPage**：主從式維護 Android / iOS 虛擬機清單；編輯區可「啟動虛擬機」與「關閉」。
- **EnvVarsPage**：表格式維護全專案共用環境變數；支援每列 🎲 Faker 隨機產生與批次 Faker modal。
- **DbConfigPage**：主從式維護 DB 連線（MySQL / Postgres / MSSQL / Oracle / Mongo / Redis / SQLite），含測試連線與 SQL 測試區。
- **MockPage**：主從式維護 REST Mock 端點；可同時編輯發出（Request）與回應（Response），支援 Faker 佔位符與 Mock Server 啟停。
- **RecorderPage**：支援 WEB 錄製、API cURL 解析與 APP Appium Python 腳本解析。
- **SchedulePage**：建立一次性或循環排程，並可立即手動觸發一次。
- **TestRoundPage**：建立具名稱與說明的案例集合，集中執行多筆測試案例。

補充：

- 專案工具列提供「新增 Feature」、「重新命名目前專案」、「刪除目前專案」。
- 每個樹節點右側都有快捷按鈕，可直接新增子節點、修改名稱或刪除。
- 搜尋欄會即時過濾目前專案的樹狀節點名稱。
- 切換到「本機」模式後，右上角會出現 `?` 按鈕，可開啟 local agent 使用說明並下載腳本。

## 3. 建立專案與目錄樹

### 3.1 建立專案

1. 點 TopNav 的「專案下拉」→ **新增專案**。
2. 輸入專案名稱 → 確定。
3. 系統會自動切換至新專案，目錄樹為空。

### 3.2 新增、修改與刪除節點

將滑鼠移到樹狀目錄節點上，右側會出現操作按鈕：

| 動作 | 說明 |
|---|---|
| 新增子節點 | 在目前節點下建立下一層 |
| 修改名稱 | 直接修改該節點名稱 |
| 刪除 | 連同子節點一併刪除（不可復原）|

層級由系統自動管理：每往下一層即往下一個 level_type。
**只有最底層的 TESTCASE 節點**，點選後才會出現右側編輯器。

補充：

- 左上角「目錄樹」工具列的 `+` 用來在專案根下新增第一層 Feature。
- 節點列右側的鉛筆圖示可直接修改名稱；垃圾桶圖示則會刪除該節點與全部子孫。

### 3.3 搜尋與專案管理

- 左側搜尋欄可依名稱即時篩選節點，適合在大型專案中快速定位 Feature / Scenario / TestCase。
- 專案下拉選單右側共有三個按鈕：新增 Feature、重新命名目前專案、刪除目前專案。
- 刪除專案時，系統會一併刪除該專案下所有節點、測試案例與執行報告，且不可復原。

## 4. 撰寫測試案例

點選任一 TESTCASE 節點後，右側編輯器顯示四個區塊。
任何修改都會讓畫面上方顯示「● 未儲存」，且「儲存」按鈕高亮；**執行測試前請先儲存**。右上角另提供「匯出 MD」按鈕，可把目前案例下載成 Markdown。

### 4.1 驗收準則 (Acceptance Criteria)

純文字框，記錄使用者驗收測試案例通過所需滿足的條件。

範例：
```
使用者輸入正確帳密後，應導向 /dashboard 並顯示歡迎訊息「Hi, {username}」。
```

### 4.2 ⭐ 前置動作 (Pre-Setup)

> v1.0 新增。位於驗收準則下方。

純文字框，用來記錄 **執行此測試案例前需要手動準備** 的環境/資料事項，例如：

```
- 啟動 mock server (port 4000)
- 在資料庫塞入 seed 使用者: alice / Pa$$w0rd
- 預先取得 OAuth token 並設為環境變數 TOKEN
```

> ⚠️ 目前此欄位 **僅作為說明文字**，不會自動執行。
> 若要自動化前置動作，請改寫到 BDD 步驟表（4.3）。

### 4.3 BDD 步驟表

每一列代表一個動作。欄位說明：

| 欄位 | 必填 | 說明 |
|---|---|---|
| 關鍵字 | ✓ | `Given` / `When` / `Then` / `And`（僅作可讀性） |
| 描述 | ✓ | 此步驟在報告中顯示的文字 |
| Action | ✓ | 系統可識別的動作（見下表） |
| Locator | 視 action 而定 | CSS / XPath / API URL / SQL |
| Input | 視 action 而定 | 輸入文字、JSON body、SQL 片段 |
| Condition | 預設 Equals | 斷言比較方式 |
| Expected | 視 action 而定 | 預期值、HTTP status、列數 |

#### Action 速查（常用）

| 類別 | Action | 說明 |
|---|---|---|
| 瀏覽器 | `Goto` / `Click` / `Fill` / `Select` / `Hover` | UI 操作 |
| 瀏覽器 | `AssertVisible` / `AssertText` / `AssertUrl` | UI 斷言 |
| 瀏覽器 | `AssertScreenshotMatch` | Pillow 像素 diff vs baseline（見 4.6 節）|
| HTTP  | `Http.Get` / `Http.Post` / `Http.Delete` | API 呼叫 |
| SQL   | `Db.Connect` / `Db.Query` / `Db.RowCount` | 資料庫原始 SQL |
| SQL   | `Db.Insert` / `Db.Update` / `Db.Delete` | 寫入語意明確化 |
| SQL   | `Db.AssertRowExists` / `Db.AssertNoRow` / `Db.AssertValue` | 資料庫斷言 |
| Mobile| `Mobile.Open` / `Mobile.Click` / `Mobile.Input` | App 操作 |

> E2E 平台的案例可同時使用上述所有動作（含 Mobile + DB），跨領域 end-to-end 測試適用。
>
> 完整 action 清單請見 [README.md](README.md#測試案例步驟steps_json支援的-action)。

#### 操作步驟列

- **➕ 新增步驟**：在表格底部插入空白列。
- **▲ ▼**：上下移動列順序。
- **🗑️**：刪除該列。
- **長 Locator 檢視**：若欄位內容較長（例如 `role=heading[name="登入"]`），可用表格水平捲動查看，或把滑鼠移到欄位上看完整 Tooltip。

### 4.4 DDT 資料表

目前單頁 UI 的 DDT 編輯器採固定四欄：`No`、`檔案名稱`、`Key`、`Value`。

使用方式：

1. 按「新增資料列」加入一筆資料。
2. 在 `Key` / `Value` 欄輸入資料，或使用「匯入 JSON」把 JSON 展平成多筆資料列。
3. DDT 目前主要作為平台的資料來源表管理；前端按「執行測試」時，預設不會依每一列重新跑一次整個 testcase。

補充：

- 若需要逐列展開執行，需直接呼叫執行 API，並把 `ddt_expand=true` 一併送出。
- 若你是從舊版文件沿用泛型 `{ headers, rows }` 資料格式，後端仍能接受，但目前預設 UI 顯示為固定四欄。

### 4.5 Markdown 匯出

1. 先在左側選取一筆 TESTCASE。
2. 點右上角「匯出 MD」。
3. 系統會下載一份 `.md` 檔，內容包含該案例的 AC 與步驟表。

補充：

- 目前預設 UI 僅提供匯出按鈕。
- 若要把 Markdown 匯回平台，需透過後端 API `POST /api/testcases/{node_id}/import-md`。

### 4.6 Screenshot Diff（UI 前後差異比對）

UI / WEB / E2E 案例可以加入 **Playwright 風格的截圖比對**，自動驗證畫面與基準圖一致。

#### 加入比對步驟
1. 在步驟表新增一列，**動作 (Action)** 選 `AssertScreenshotMatch`
2. **Locator**：留空 = 截整頁；填 CSS / XPath = 只截單一元素
3. **預期結果**：填容忍 %（如 `1.5` 或 `1.5%`，預設 `1.0`）
4. **儲存案例**

#### 設定 baseline 的三種方式

**方式 A：自動（最常用）**
- 第一次執行時，listener 會自動把當下截圖存為 baseline，該步驟通過
- 之後執行才會做比對

**方式 B：手動上傳**
- 步驟列右側出現紫色 📷 按鈕，點開「設定 Screenshot Baseline」Modal
- 預覽現有 baseline、調整容忍 %、上傳新檔（PNG / JPEG / WebP，10MB 上限）
- 按「儲存」即覆蓋

**方式 C：報告中設定**
- 跑完 → 報告詳情頁 → 點該步驟 → 右側「Screenshot Diff」面板
- 三聯比對：**Baseline | Actual | Diff（紅色覆蓋）**
- 點「把當下 actual 設為新 baseline」→ 一鍵把這次跑的截圖設為新基準（適合「現在的畫面才對」情境）

#### 失敗時看什麼
- 報告詳情頁的 Screenshot Diff 面板會顯示實際 diff %
- Diff 那格顯示「紅色覆蓋」差異圖（差異像素以 60% 不透明紅色蓋在 Actual 上）
- 點圖可放大檢視

> 像素比對閾值：每像素 RGB 距離 > 30 才視為「真的有差」（避開 JPEG 雜訊與抗鋸齒微小差異）。如果想調整可改 [backend/tasks/assert_screenshot_lib.py](backend/tasks/assert_screenshot_lib.py) 的 `differ_mask = diff > 30`。

## 5. 錄製與腳本轉換

錄製頁目前分成三個平台頁籤：`WEB`、`API`、`APP`。

- `WEB`：以 Playwright codegen / rfbrowser codegen 錄製實際瀏覽器操作。
- `API`：貼上 DevTools 的 `Copy as cURL`，轉換成 `Http.*` 步驟。
- `APP`：貼上 Appium Python 腳本，轉換成 `Mobile.*` 步驟。

### 5.1 WEB：進入錄製頁前的準備

1. 先切到「案例編輯」，從左側目錄樹選取一筆 TESTCASE。
2. 確認目前案例已載入右側編輯器。
3. 再切到 TopNav 的「錄製」。

> 若沒有先選 TESTCASE，之後按「套用至當前案例」時系統會提示先返回案例編輯頁。

### 5.2 WEB：建立錄製階段

1. 在「① 建立錄製階段」輸入目標網站 URL。
2. 點「建立錄製階段」。
3. 畫面會顯示四種錄製指令：
   - A) Node.js `npx playwright codegen`
   - B) Python `python -m playwright codegen`
   - C) `rfbrowser codegen`
   - D) 一鍵 PowerShell（自動上傳）

### 5.3 WEB：產生 `recorded.py` 與 `trace.zip`

1. 在本機終端機貼上其中一種指令。
2. Playwright Inspector 會開啟真實瀏覽器。
3. 在瀏覽器中執行你的測試操作。
4. 關閉錄製視窗後，本機會產生：
   - `recorded_xxxx.py`
   - `trace_xxxx.zip`

> `trace.zip` 可在 <https://trace.playwright.dev> 開啟，分析 DOM、步驟時間線、network 與 screenshot。

### 5.4 WEB：讓系統自動帶出「比對條件 / 預期結果」

若你只做 Click / Fill / Goto，系統只會產生操作步驟，不會自動填 Condition / Expected。

若你要讓錄製結果自動帶出斷言欄位，請在 Playwright Inspector 工具列使用：

- `Assert visibility`
- `Assert text`
- `Assert value`

之後再點頁面元素，腳本會產生 `expect(...)` 斷言；平台解析後會自動填入：

- Action：`AssertVisible` 或 `AssertText`
- Condition：`IsVisible`、`Equals`、`Contains`
- Expected：預期文字或 `true`

### 5.5 WEB：上傳並套用到測試案例

1. 回到錄製頁的「③ 上傳 codegen 結果」。
2. 拖曳 `recorded_xxxx.py` 與 `trace_xxxx.zip` 到上傳區。
3. 點「上傳」。
4. 若有 trace，畫面會出現「下載 trace.zip」按鈕。
5. 點「套用至當前案例」，系統會把解析出的步驟合併到目前選中的案例。

> 若使用 D) 一鍵 PowerShell，codegen 完成後會自動 curl 上傳，不必手動拖檔。

### 5.6 API：貼上 cURL 轉成 `Http.*` 步驟

1. 切到錄製頁的 `API` 頁籤。
2. 在瀏覽器 DevTools 的 Network 面板對請求按右鍵，選 `Copy as cURL`。
3. 將 cURL 貼到文字框中；可一次貼多個，以空行分隔。
4. 點「解析為 Http.* 步驟」，平台會轉成對應的 API 測試步驟。

### 5.7 APP：貼上 Appium Python 轉成 `Mobile.*` 步驟

1. 切到錄製頁的 `APP` 頁籤。
2. 貼上由 Appium Inspector 匯出的 Python 腳本，或貼上手寫的 `driver.find_element(...)` 片段。
3. 點「解析為 Mobile.* 步驟」，平台會轉成行動測試步驟。

## 6. 排程執行

### 6.1 建立排程

1. 點 TopNav 的「排程」。
2. 按右上角「新增排程」。
3. 填入排程名稱。
4. 在清單中搜尋並勾選要執行的 `.md` 測試案例，可一次多選。
5. 設定重複規則：
   - `指定時間`：單次執行
   - `每天`
   - `每週`：至少勾選一個星期
   - `每月`：指定第幾天
6. 選擇執行環境：`Docker` 或 `本機`。
7. 視需要勾選「啟用此排程」，再按「儲存」。

補充：

- 後端會每 30 秒掃描一次到期排程。
- 目前單頁 UI 的排程視窗只列出 `.md` 測試案例，並提供搜尋與多選。
- 若要對更高層節點排程，需改由 API 建立。

### 6.2 立即執行、停用與刪除

- `立即`：立刻觸發一次，不影響下一次自動執行時間。
- 開關：可隨時啟用 / 停用排程。
- `編輯`：修改名稱、時間、重複規則、執行環境。
- `刪除`：移除排程設定。

注意：排程按下「立即」時，會沿用目前頁面頂部的環境切換狀態（Docker / 本機）。

## 7. 測試回合

### 7.1 建立測試回合

1. 點 TopNav 的「測試回合」。
2. 按右上角「新增回合」。
3. 輸入回合名稱，必要時補上說明。
4. 在清單中搜尋並勾選要納入的 `.md` 測試案例，可一次多選。
5. 選擇回合的預設執行環境：`Docker` 或 `本機`。
6. 按「儲存」。

補充：

- 回合列表會顯示案例數、預設執行環境、說明與建立時間。
- 回合的「案例數」欄位滑鼠移上去可看到完整案例名稱清單。
- 建立或編輯回合前，需先在左側載入專案，系統才知道可選哪些案例。

### 7.2 執行、編輯與刪除

- `執行`：把回合內全部案例彙總成一份報告，並跳到「測試報告」查看進度。
- `編輯`：修改回合名稱、說明、案例清單與預設執行環境。
- `刪除`：移除此回合設定。

注意：

- 從單頁 UI 按「執行」時，會以目前頁面頂部的環境切換狀態為準，而不是單純沿用回合的預設值。
- 若回合包含多筆案例，現階段建議優先使用 Docker 模式；本機模式較適合單案例或單節點回合。

### 7.3 環境變數（TopNav：🔑 環境變數）

整張表格式維護，**整個專案共用、所有測試案例都會吃**，執行時自動注入成 Robot suite variable。

- 點 ➕「新增變數」在表尾插入一列，填入 `NAME` / `VALUE` / 說明
- **NAME 限制**：英數+底線、不可數字開頭（驗證不過會跳警告）
- 含 `TOKEN` / `SECRET` / `PASSWORD` / `KEY` 字樣的值預設遮蔽（👁 可切明文）
- 「匯入 .env」可一次貼多行 `NAME=value`，同名覆蓋、不同名追加
- 「用法」欄的按鈕點一下就把 `${NAME}` 複製到剪貼簿，方便貼進步驟欄位
- 改完記得按「💾 儲存」，整批 PUT 到後端

#### 🎲 Faker 隨機產生（新）

每列右邊都有 `🎲 Faker` 按鈕，點開可選 28 種類型一鍵產生值並填進 value：

| 類別 | 範例 key |
|---|---|
| 個資 | `name` / `first_name` / `last_name` / `username` |
| 聯絡 | `email` / `phone_tw` / `phone_us` / `address` / `city` / `country` / `zipcode` |
| 身分 | `uuid` / `token_hex` / `jwt_like` / `password` / `credit_card` |
| 數值 | `int_0_100` / `int_1000_9999` / `price` / `bool` / `hex_color` |
| 時間 | `date_today` / `datetime` / `timestamp` |
| 文字 | `paragraph` / `sentence` / `company` / `url` / `ipv4` |

頁首「🎲 批次 Faker」按鈕可一次勾選多個類型，批次以 `FAKE_<KEY>` 為 NAME 加到清單（同名覆蓋）。

### 7.4 設備資訊（TopNav：📱 設備資訊）

主從式：左側清單（Android / iOS 分色），右側是單筆設備的編輯器。每個設備的 **label** 會被注入成 `&{DEVICE_<label>}` Robot dict 變數，capabilities 會完整合併。

- 必填：label（限英數+底線）；其餘 OS 版本 / Device Name / AVD（Android）/ UDID（iOS）視 platform 填
- 在 `Mobile.Open` 步驟可引用 `${DEVICE_pixel5_emu.platformName}` / `${DEVICE_pixel5_emu.deviceName}`
- `automationName` 沒填時自動帶 `UiAutomator2`（Android）/ `XCUITest`（iOS）
- 「額外 Capabilities（JSON）」會合併到最終 capabilities，JSON 錯誤不會送出儲存
- 下方「Capabilities Preview」即時顯示 Robot 注入格式與 Python dict

#### 🟢 啟動虛擬機（新）

編輯區標題列的兩顆按鈕：

- **啟動虛擬機**：後端若已實作 `POST /api/devices/launch` 就直接打 API；否則前端會把對應的本機指令（`emulator -avd ...` / `xcrun simctl boot ...`）複製到剪貼簿，並以 Toast 提示
- **關閉**：同上，對應 `POST /api/devices/stop` 或本機 `adb emu kill` / `xcrun simctl shutdown ...`
- 右邊的狀態 badge 會在「未啟動 / 啟動中… / 執行中 / 啟動失敗」之間切換

### 7.5 DB 資訊（TopNav：🗄 DB 資訊）

主從式清單 + 右側編輯器；資料依專案存於瀏覽器 `localStorage`（目前尚未同步到後端 DB）。

- 左側頂端按鈕可分別新增 MySQL / Postgres / MSSQL / Mongo / Redis / SQLite（切換 type 會自動帶預設 port，如 3306 / 5432 / 1433 / 27017 / 6379）
- 欄位：name（限英數+底線；會成為 `&{DB_<name>}`）、type、host、port、username、password（👁 可切明文）、database、charset / SSL、自訂 DSN（填了會覆蓋上方欄位）、說明
- **🔌 測試連線**：呼叫 `POST /api/db/test`（後端未實作時前端模擬為成功，並印出擬定的 DSN）
- **SQL / Query 測試區**：貼任意 SQL → 「執行」打 `POST /api/db/query`，結果 JSON 顯示於下方黑底區域；後端未實作時回傳擬定 rows 方便開發
- **注入格式 Preview**：同時顯示 Robot 注入格式、連線 DSN、Python dict（密碼會遮蔽）
- 支援「複製為新連線」、「從清單移除」、搜尋 filter

### 7.6 Mock 端點管理（TopNav：🔌 Mock）

主從式清單 + 右側編輯器；資料依專案存於瀏覽器 `localStorage`。

- 左側頂端四顆按鈕可分別新增 GET / POST / PUT / DELETE 端點；列表每列顯示 method 色塊 + path + 啟用狀態
- 編輯器欄位：method、path、狀態碼、延遲 ms、Content-Type、啟用 checkbox、說明
- **發出（Request）**：發出 Headers（JSON，選填）、發出 Body（支援 JSON / Form 範本，格式化 JSON 按鈕）
- **回應（Response）**：回應 Headers（JSON，選填）、回應 Body（JSON / List / Error 範本，格式化 JSON 按鈕）
- **Faker 佔位符**：在 Headers / Body 內可直接寫 `{{name}}` / `{{email}}` / `{{uuid}}` / `{{token_hex}}` / `{{int:1,100}}`，試打時會就地渲染
- **🛰 試打**：把渲染後的 Request 與 Response 完整內容分段列印到下方終端區
- **Mock Server 啟動 / 停止**（列表底部）：呼叫 `POST /api/mock/toggle`；後端未實作時前端顯示模擬狀態「執行中 :4523 / 已停止」

> DB 資訊與 Mock 目前僅用 localStorage 作為儲存，清快取會遺失。若要跨機器共用，後續需串接後端。

## 8. 執行測試

### 8.1 Docker / 本機模式

在頁面頂部可以切換兩種執行環境：

- `Docker`：Celery worker 為每個 testcase **動態 spawn 一個 `autotest-robot-runner` 容器**跑 Robot Framework，預設 headless（容器內走 xvfb）。容器跑完自毀；所有產物即時上傳到 MinIO。
- `本機`：由使用者電腦上的 `local_agent.py` 認領任務，會開啟有頭 Chromium。

切換後，手動執行與排程「立即」都會沿用目前的環境模式。

> 想觀察 spawn 容器：另開 PowerShell 跑 `docker ps -a --filter name=robot-`，按執行時會看到 `robot-<task>-<case>` 一閃即逝（執行完 `--rm`）。

### 8.2 啟動本機 Agent

1. 先把頂部環境切到「本機」。
2. 點旁邊的 `?` 按鈕開啟說明視窗。
3. 下載 `local_agent.py`。
4. 在本機終端機安裝：

```powershell
pip install playwright requests
playwright install chromium
```

5. 依視窗中的指令啟動 agent，例如：

```powershell
python local_agent.py --server http://localhost
```

限制：

- 本機 Agent 目前只支援 Web UI 類步驟。
- `Http.*` 與 `Mobile.*` 測試請改用 Docker 模式。
- 多分頁、檔案上傳下載等流程若依賴本機路徑，請確認 agent 執行目錄正確。
- 本機模式不會走 spawn 容器（只有 Docker 模式才 spawn），截圖 / 錄影 / Trace 由 agent 端直接上傳到 backend。

### 8.3 手動執行與即時日誌

1. 在編輯器右上方按「▶ 執行」。
2. 若案例尚未儲存，系統會先提示「請先儲存後再執行」。
3. 系統會建立一筆 Execution：
   - Docker 模式：Celery worker 收到任務 → 上傳 .robot 到 MinIO → spawn `autotest-robot-runner` 容器 → 等容器結束 → 抓回結果
   - 本機模式：等待 local agent 認領
4. 底部會自動展開 **即時日誌面板**（透過 WebSocket 串流）。終端機會印出 `🐳 啟動容器`、`🏁 容器結束` 等訊息，方便觀察 spawn 流程。
5. 過程中每個步驟會自動截圖（PRE / POST），listener 即時上傳到 MinIO `results` bucket。
6. 目前前端手動執行可由「執行」按鈕旁的⚙ 齒輪開關控制：
   - **啟用 Trace + Video**（預設開）：產生 `trace.zip` 與 `.webm`（每案例一份完整錄影；不再切片每步驟）
   - **展開 DDT 每一列都執行**（預設關）：開啟後依資料表每列重跑整個 testcase

即時日誌抽屜目前提供：

- 清除日誌
- 收合 / 展開
- 關閉抽屜

## 9. 查看報告

切到 TopNav 的「📊 執行報告」頁。

### 9.1 儀表板 (Dashboard)

- 通過率圓環圖
- 最近 N 次執行的趨勢折線圖
- 各 Feature 通過率長條圖

### 9.2 最近執行紀錄

依時間倒序列出該專案所有執行紀錄。每列欄位：**執行時間 / 專案名稱 / 測試案例 / 目標 / 觸發者 / 成功‧失敗 / 測試結果 / 執行耗時**。

- 「測試案例 / 目標」欄顯示觸發該次執行的節點 title 與 level badge（TESTCASE / PAGE / FEATURE / 測試回合）；游標停在那格可看 node_id tooltip
- **狀態顯示改進**：
  - `RUNNING`：amber 旋轉 spinner；清單裡只要有 RUNNING 紀錄就每 3 秒自動刷新一次
  - `PASSED`：綠 `PASS`
  - `FAILED`：紅 `FAIL`
  - 狀態空白 / 未知：不再強制當作 FAIL，會依 passed/failed 步驟數判斷；真的都沒資料時顯示 `—`
- 執行完成（WebSocket 收到 `done`）時，若你正停在測試報告頁，列表會自動刷新一次

### 9.3 報告詳情 (ReportDetail) — 依步驟類型自動切換面板

點進任一筆報告後，左側是步驟時間軸，右側是所選步驟的詳細面板。每個步驟依 action 前綴自動判斷類型並呈現不同面板，**E2E 案例同一份報告內可以混合多種步驟類型**：

| 步驟類型 | 觸發 action 前綴 | 右側面板呈現 |
|---|---|---|
| **UI** | 無前綴（Browser Library） | 瀏覽器風格 pre/post 截圖（含目標元素紅框高亮） |
| **API** | `Http.*` | Request / Response JSON 兩欄（附 method+url、HTTP status、耗時、size） |
| **APP** | `Mobile.*` | 直式手機框架內顯示 Appium pre/post 截圖 |
| **DB** | `Db.*` | 上方 SQL 程式碼區、下方結果列表格（欄位自動推）或「影響 N 列」提示、可選 Assert 訊息 |

步驟標題右側會有 `UI / API / APP / DB` 類型 badge；本機 Agent 回報的步驟也會顯示耗時與錯誤訊息。

PDF 匯出會以同樣分類呈現：APP 截圖放大到 320px、DB 結果表最多列出 20 列。

### 9.4 軌跡（Trace）與錄影（Video）

執行時若啟用了「Trace + Video」（預設開），點任一步驟後右側會多出一塊「軌跡 / 錄影」面板：

- ▶ **完整錄影**：頁內 Modal 播放整個案例（DDT 列）的完整錄影（`.webm`）
- ⬇ **下載錄影**：直接下載 `.webm`
- ⬇ **下載 Trace**：直接下載 `trace.zip`，於本機可用 `playwright show-trace trace.zip` 開啟
- ↗ **Trace Viewer**：新分頁開啟 `https://trace.playwright.dev/?trace=<absolute URL>`
- 🔍 **嵌入檢視**：頁內 iframe 嵌入 Trace Viewer（直接在報告頁內看完整 Playwright 軌跡，含 DOM 快照、Network、Console）
  - 已在 nginx 端 `/results/` 開啟 CORS 並排除 MinIO 自帶 ACAO；若 `BASE_URL` 改成自己的網域，記得確認該網域對 trace.playwright.dev 可達
  - 純內網部署若無公開網址，請改用「下載 Trace + playwright show-trace」

> 案例級的 trace 與完整錄影是「每個案例 / DDT 列各一份」，所以同一案例內任何步驟點開來，
> 看到的「完整錄影」與「Trace」都會是該案例這一輪的整段紀錄。
>
> 之前實驗過「每步驟切片錄影」功能已移除（使用者反饋只要完整錄影即可）。

### 9.5 Screenshot Diff（UI 前後差異比對）

對 `AssertScreenshotMatch` step，點該步驟後右側會多出紫色「Screenshot Diff」面板：

- **三聯比對**：Baseline | Actual | Diff（紅色覆蓋差異區）
- 標題顯示實際 diff %（例：差異：0.85%）
- 點任一張圖可放大檢視
- 「📷 把當下 actual 設為新 baseline」按鈕：呼叫 `POST /api/steps/{uuid}/baseline/copy-from`，把這次跑的 actual 截圖直接設為新基準（適合「現在的畫面才對」情境）

> Diff > 容忍門檻才會 FAIL；PASS 的 step 也會顯示這個面板（但 Diff 那格通常是空白，因為差異不夠多到產出 diff 圖）。
>
> baseline 的設定流程詳見 4.6 節「Screenshot Diff（UI 前後差異比對）」。

## 10. 常見問題

### Q1. 改了「前置動作」之後沒看到？
請確認是否已按「儲存」。未儲存時畫面上方會顯示「● 未儲存」。

### Q2. 為什麼錄製後沒有 `trace.zip`？

- 請使用頁面提供的 Playwright codegen 指令，這些指令已帶 `--save-trace`
- 最穩定的是使用「D) 一鍵執行＋自動上傳（PowerShell）」
- 若你自行改用其他 codegen 指令，沒有帶 `--save-trace` 就不會產生 trace

### Q3. 為什麼錄製結果沒有「比對條件 / 預期結果」？

- 只有 Click / Fill / Goto 這類純操作，不會自動產生斷言欄位
- 請在 Playwright Inspector 點 `Assert visibility`、`Assert text` 或 `Assert value`
- 讓腳本產生 `expect(...)` 後，平台才會解析成 Condition / Expected

### Q4. 點「套用至當前案例」沒有成功？

- 先確認你有先到「案例編輯」選取一筆 TESTCASE
- 再確認 `recorded.py` 已成功上傳
- 若腳本裡沒有可解析的 `goto/click/fill/expect`，系統會提示找不到可轉換動作

### Q4-1. 可以重新命名專案或節點嗎？

- 可以。專案下拉右側有「重新命名目前專案」按鈕。
- 每個節點列右側也有鉛筆圖示可修改名稱。

### Q4-2. 排程時間多久檢查一次？

- 後端目前每 30 秒掃描一次到期排程。
- 若你剛建立排程，實際觸發時間可能會有最多約半分鐘的輪詢誤差。

### Q4-3. 本機模式按了執行卻沒反應？

- 先確認頂部環境已切到「本機」。
- 再確認 `local_agent.py` 已啟動，且是用正確的 `--server` 位址連到平台。
- 若是 API 或 Mobile 類步驟，請改回 Docker 模式。

### Q5. 執行時看不到截圖？
- 確認 backend / celery 都在執行：`docker compose ps`
- 確認 `.env` 的 `BASE_URL` 與實際對外網址一致（預設 `http://localhost`）
- 若使用 `STORAGE_BACKEND=minio`，也請確認 MinIO 容器健康

### Q5-1. 為什麼 DDT 沒有每列都重跑？

- 目前單頁 UI 的「執行測試」固定送出 `ddt_expand=false`。
- 若你需要每列展開執行，必須改由 API 直接送出執行請求並設定 `ddt_expand=true`。

### Q6. WebSocket 日誌一直空白？
- 檢查瀏覽器 DevTools → Network → WS 連線是否建立
- 平台目前使用的路徑是 `/ws/executions/{task_id}/logs`，不是 `/ws/v1/...`
- 重啟 backend：`docker compose restart backend`

### Q7. 改了 Schema/Model 後 API 報錯？
- 後端容器需要重建：`docker compose build backend; docker compose up -d backend`
- 若是新增 DB 欄位，需先對 MySQL 執行 ALTER TABLE 或 `docker compose down -v` 重建

### Q7-1. 更新到新版後，「測試案例 / 目標」那欄還是一直顯示「—」？
- 這個欄位是靠後端 `/api/reports` 新增的 `source_node_id` / `source_title` 欄位回傳的。
- 請確認後端容器已經用新版 code 重啟：`docker compose build backend; docker compose up -d backend`。
- 舊的執行紀錄若建立時沒寫入 `source_node_id`（例如早期資料），那一列仍會顯示 `—`；新跑的紀錄才會正確帶出 title。

### Q7-2. 測試還在跑，結果儀表板列卻顯示 FAIL？
- 若後端完全沒在跑（例如 celery worker 掛了），report 會停留在 `RUNNING` 狀態，前端現在不會再把未知狀態當成 FAIL，只會顯示 `—`。
- 真的 FAIL 才會紅；真的 RUNNING 會顯示 amber 的旋轉 spinner。
- 只要清單裡有 RUNNING，儀表板會每 3 秒自動刷新一次；執行完成（WS `done` 事件）時也會主動刷新一次。
- 若看到很久沒變的 RUNNING：可能 worker 當掉或容器卡住，可以手動在編輯頁按停止，或 `docker logs autotest-celery` 看看。

### Q8. 想清空所有資料重新開始？
```powershell
docker compose down -v
docker compose up -d --build
```
> ⚠️ `-v` 會刪除 mysql / minio volume，所有專案、報告、截圖都會消失。
